## 1. 디자인 원칙

### 1.1 핵심 원칙

| 원칙 | 설명 | 적용 방법 |
|------|------|----------|
| **PM-First** | 기술 용어 최소화, PM이 직관적으로 이해 가능한 표현 | 모든 기술 용어 옆에 PM 친화 설명 병기, 툴팁 제공 |
| **정보 계층화** | 요약에서 상세로 점진적 공개 (Progressive Disclosure) | 대시보드(요약) -> 플로우차트(구조) -> 노드 상세(디테일) 3단계 |
| **시각 우선** | 텍스트보다 차트/그래프/색상으로 핵심 전달 | 등급별 색상 코딩, 비율 차트, 아이콘 활용 |
| **액션 유도** | 각 화면에서 "다음에 무엇을 해야 하는지" 명확한 안내 | CTA 버튼, 권장 액션 뱃지, 체크리스트 연동 |
| **일관성** | CLI와 웹 간 동일한 정보 체계 유지 | 동일 색상 코드, 동일 등급 표현, 동일 용어 사용 |

## 부록 B: 최종 검증 서명 [피드백 R15 반영]

> PM + 디자이너 + TPO + 마스터가 이 Design 문서를 최종 검토하여, 개발자가 UI/UX 구현을 시작하기에 충분한 수준임을 확인했습니다.
>
> - 컬러 시스템, 타이포그래피, 아이콘 체계가 구체적으로 정의됨
> - CLI 인터페이스의 모든 명령어 입출력 형식이 와이어프레임 수준으로 설계됨
> - 시각화 웹의 5개 탭(대시보드/플로우차트/체크리스트/확인요청/티켓) 구조와 인터랙션이 상세히 기술됨
> - 8가지 노드 유형의 크기, 색상, 모양, 레이아웃이 명확함
> - 접근성(색각 이상, 키보드 네비게이션)과 성능 고려사항이 포함됨
> - 온보딩 플로우가 PM 첫 사용 시나리오를 충분히 커버함
>
