## 3. 시각화 웹 디자인

### 3.1 전체 레이아웃

시각화 웹은 로컬에서 실행되는 싱글 페이지 애플리케이션(SPA)으로, 상단 네비게이션과 콘텐츠 영역으로 구성됩니다.

**레이아웃 구조**:

```
┌─────────────────────────────────────────────────────────────┐
│  [Logo]  대시보드 | 플로우차트 | 체크리스트 | 티켓       [설정]│  <- 상단 네비
├─────────────────────────────────────────────────────────────┤
│                                                             │
│                     메인 콘텐츠 영역                         │
│                   (탭에 따라 전환)                            │
│                                                             │
│                                                             │
│                                                             │
│                                                             │
│                                                             │
│                                                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**네비게이션 탭**: [추가 검토 반영: 확인 요청 탭 추가]

| 탭 | 아이콘 | 설명 |
|----|--------|------|
| 대시보드 | `layout-dashboard` | 종합 요약 화면 (기본 진입) |
| 플로우차트 | `git-branch` | 계층 구조 시각화 (메인 기능) |
| 체크리스트 | `clipboard-check` | 기획 확인 사항 + 정책 경고 [추가 검토 반영] |
| 확인 요청 [추가 검토 반영] | `users` | 시스템별 담당자 확인 요청 안내 |
| 티켓 | `ticket` | 생성된 작업 티켓 목록 |

**반응형 기준**:
- 데스크톱 (1280px 이상): 전체 레이아웃 표시 (플로우차트 영역 최소 880px + 상세 패널 400px)
- 태블릿 (768px~1279px): 사이드 패널이 오버레이로 전환 (플로우차트 위에 반투명 배경 + 패널 슬라이드인)
- 모바일 대응은 MVP 범위 밖 (데스크톱 기본)

> [피드백 R3 반영] 상세 패널 400px 고정 + 플로우차트 최소 유효 영역 880px = 최소 1280px에서 양쪽 동시 표시. 1280px 미만에서는 패널을 오버레이로 전환하여 플로우차트 영역을 침범하지 않도록 합니다. 패널 오버레이 시 배경 클릭으로 닫기가 가능해야 합니다.

### 3.2 종합 대시보드

대시보드는 분석 결과의 핵심 정보를 한눈에 파악할 수 있는 요약 화면입니다.

**구성 영역**:

#### 상단: 영향도 요약 헤더

```
┌─────────────────────────────────────────────────────────────┐
│                                                             │
│  리뷰 필터 기능 추가                     2026-02-13 분석    │
│                                                             │
│      32점                                                   │
│     Medium                                                  │
│  ─────────────                                              │
│  종합 영향도 등급           스프린트 계획 시 우선순위        │
│                             조정이 필요합니다.               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

- 종합 점수를 크게 표시 (48px Bold)
- 등급 뱃지를 등급 색상으로 표시
- 등급에 따른 권장 액션 텍스트 표시

#### 상단 하위: 신뢰도 경고 배너 (조건부 표시) [분석 일관성 전략 반영]

Low Confidence 이하 시스템이 포함된 분석 결과에서만 표시되는 경고 배너입니다.

```
┌─ [!] ───────────────────────────────────────────────────────┐
│  1개 시스템의 분석 신뢰도가 낮습니다 (42%, Low Confidence)    │
│  해당 시스템의 분석 결과는 참고용으로만 활용하고,             │
│  담당 개발자와 추가 확인을 권장합니다.   [자세히 보기]        │
└─────────────────────────────────────────────────────────────┘
```

- **표시 조건**: 분석에 포함된 시스템 중 신뢰도 Low(40~64%) 또는 Very Low(0~39%)가 1개 이상
- 배경: `#FFF7ED` (연한 주황)
- 좌측 보더: 4px solid `#F97316` (주황)
- 텍스트: `#92400E` (진한 주황 브라운)
- 아이콘: `alert-triangle` (Lucide)
- 모서리 반경: 8px
- "자세히 보기" 링크: 클릭 시 대시보드 하단의 "분석 신뢰도 현황" 섹션으로 스크롤
- 닫기 버튼: 없음 (항상 표시하여 PM이 인지하도록)
- Very Low 시스템이 있으면 배경을 `#FEF2F2` (연한 빨강), 보더를 `#EF4444` (빨강)으로 변경

#### 중단: KPI 카드 4개 (가로 배치)

```
┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐ ┌───────────┐
│ 영향 화면  │ │  총 작업   │ │ 기획 확인  │ │ 정책 경고  │ │ 확인 요청  │
│    3개    │ │    8개    │ │    2건    │ │    2건    │ │    3팀    │
│           │ │FE 5|BE 3 │ │ 확인 필요  │ │ 주의 필요  │ │ 안내 생성  │
└───────────┘ └───────────┘ └───────────┘ └───────────┘ └───────────┘
```

> [추가 검토 반영] KPI 카드를 4개에서 5개로 확장. "정책 변경" 카드를 "정책 경고"로 변경하여 정책 사전 매칭 결과를 반영하고, "확인 요청" 카드를 추가하여 담당자 확인 요청 현황을 표시.

각 KPI 카드:
- 상단: 라벨 (14px, 보조 텍스트 색상)
- 중앙: 숫자 (32px, Bold, 진한 텍스트)
- 하단: 보조 정보 (12px, 보조 텍스트)

#### 중하단: 점수 분포 차트

```
┌───────────────────────────────────────┐  ┌─────────────────────┐
│  화면별 영향도 점수                    │  │  FE / BE 비율        │
│                                       │  │                     │
```

- 좌측: 수평 바 차트 (화면별 점수, 등급 색상 적용)

### 3.5 필터/뷰 옵션 패널

**필터 옵션**:

| 필터 | 유형 | 기본값 | 설명 |
|------|------|--------|------|
| FE/BE 구분 | 토글 칩 | 전체 | FE만, BE만, 전체 |

**필터 적용 시 동작**:
- 필터에 해당하지 않는 노드: 투명도 20% + 회색조
- 필터에 해당하는 노드: 정상 표시 + 연결선 강조
- 필터 결과 카운트 표시: "필터 결과: 5개 / 전체 15개"

|------|------|

| 모두 접기 | 시스템+화면만 표시 |

### 6.5 성능 고려사항

| 항목 | 기준 | 대응 |
|------|------|------|
| 노드 100개 이하 | 부드러운 렌더링 (60fps) | 기본 렌더링 |
| 노드 100~300개 | 약간의 지연 허용 | 가상화 + 뷰포트 외 노드 간소화 렌더링 |
| 초기 로딩 | 2초 이내 | 결과 JSON 사전 로드, 코드 스플리팅 |
| 패널 전환 | 300ms 이내 | CSS 트랜지션, 패널 내용 lazy 로드 |

> [피드백 R4 반영] **전개/접기 시 대량 노드 추가 최적화**: 화면 노드 전개 시 하위 기능/모듈 노드가 한꺼번에 수십 개 추가될 수 있습니다. 이를 최적화하기 위해: (1) 초기 상태는 모든 화면 노드를 접힌 상태로 렌더링 (화면 레벨까지만 표시), (2) 전개 시 `React.startTransition` 또는 `requestAnimationFrame` 기반 배치 렌더링으로 프레임 드랍 방지, (3) 뷰포트 외 노드는 placeholder 박스로 대체하여 DOM 노드 수 최소화.

> [피드백 R4 반영] **필터링 시 CSS 성능**: 노드 100개 이상에서 `filter: grayscale(1)` + `opacity: 0.2`를 대량 적용하면 GPU 레이어가 과도하게 생성될 수 있습니다. 대안으로: (1) CSS `filter` 대신 노드 컴포넌트 내부에서 비활성 상태용 경량 렌더링 분기 (배경색을 회색으로 변경 + opacity만 적용), (2) `will-change: opacity`를 사전 설정하여 레이어 생성 최적화, (3) 100개 초과 시 비활성 노드를 DOM에서 제거하는 옵션 제공.

## 7. 시각화 라이브러리 권장 (디자이너 관점)

플로우차트/노드 기반 시각화를 위한 라이브러리를 디자이너 관점에서 제안합니다. 최종 기술 결정은 TPO가 담당합니다.

### 7.1 권장 라이브러리

| 라이브러리 | 적합도 | 디자이너 관점 장점 | 고려사항 |
|-----------|--------|-------------------|---------|
| **React Flow** | 매우 높음 | 커스텀 노드 디자인 자유도 높음, 확대/축소/패닝 기본 제공, 미니맵 내장 | React 기반 필요 |
| **Vue Flow** | 매우 높음 | React Flow의 Vue 포팅, 동일한 디자인 자유도 | Vue 기반 필요 |
| **D3.js** | 높음 | 완전한 커스텀 가능, 모든 시각화 자유 | 구현 난이도 높음 |
| **Elkjs + SVG** | 중간 | 자동 레이아웃 알고리즘 우수 | 커스텀 노드 구현 별도 |

### 7.2 디자이너 요구사항 (라이브러리 선택 시)

라이브러리 선택 시 다음 디자인 요구사항을 충족해야 합니다:

1. **커스텀 노드**: 6가지 노드 유형(시스템/화면/기능/모듈/기획확인/정책)을 각각 다른 모양과 스타일로 렌더링 가능
2. **노드 내 복합 콘텐츠**: 노드 안에 아이콘, 텍스트, 뱃지, 바 차트 등 복합적 요소 배치 가능
3. **인터랙션**: 클릭, 호버, 드래그, 확대/축소, 패닝이 자연스럽게 동작
4. **전개/접기**: 계층 구조의 동적 전개/접기 애니메이션 지원
5. **필터링**: 특정 노드 강조/희미 처리가 트랜지션과 함께 동작
6. **연결선 커스텀**: 직선/곡선/점선 등 다양한 연결선 스타일 적용 가능
7. **반응형**: 컨테이너 크기에 맞게 자동 리사이즈
8. **미니맵**: 전체 구조 오버뷰 미니맵 제공 (또는 구현 가능)

### 7.3 차트/그래프 라이브러리 (대시보드용)

| 라이브러리 | 용도 | 디자이너 관점 |
|-----------|------|-------------|
| **Chart.js** | 바 차트, 도넛 차트 | 간결하고 깔끔한 기본 스타일, 커스텀 용이 |
| **Recharts** | React용 차트 | 선언적 API, 일관된 디자인 |
| **ECharts** | 다양한 차트 | 풍부한 차트 유형, 인터랙션 우수 |

> 대시보드의 바 차트와 도넛 차트는 비교적 단순한 형태이므로, 경량 라이브러리로 충분합니다.

## 부록 A: 용어 번역 사전 (PM 친화 표현)

분석 결과에서 기술 용어가 등장할 때 자동 변환하여 표시하기 위한 사전입니다.

| 기술 용어 | PM 친화 표현 |
|----------|-------------|
| API 엔드포인트 추가 | 서버에서 새로운 데이터 처리 기능 개발 |
| DB 스키마 변경 | 데이터 저장 구조 변경 (기존 데이터 영향 검토 필요) |
| 컴포넌트 개발 | 화면 요소(버튼, 목록, 카드 등) 새로 만들기 |
| 컴포넌트 리팩토링 | 기존 화면 요소 구조 재정리 (기능 변경 없음) |
| 캐시 무효화 | 변경사항이 사용자에게 바로 반영되도록 처리 |
| 마이그레이션 | 기존 데이터를 새 구조에 맞게 변환하는 작업 |
| 라우팅 변경 | 화면 이동 경로(URL) 변경 |
| 상태 관리 수정 | 화면 간 데이터 공유 방식 수정 |
| 미들웨어 추가 | 데이터 처리 중간 단계 추가 |
| 트랜잭션 처리 | 여러 작업을 하나로 묶어 안전하게 처리 |
| 인덱스 추가 | 데이터 검색 속도를 높이기 위한 최적화 |
| 웹소켓 연동 | 실시간 데이터 업데이트 기능 |
| SSR/SSG | 화면 로딩 속도 개선을 위한 서버 처리 방식 |
| 환경 변수 설정 | 서버 실행 환경별 설정값 변경 |
