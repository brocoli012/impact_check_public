```markdown
# Kurly Impact Checker

기획서를 입력하면 프로젝트 코드베이스를 분석하여 영향 범위,
난이도 점수, 작업 티켓을 자동 생성합니다.

## 명령어

### /impact init <project_path>
프로젝트를 등록하고 코드 인덱싱을 수행합니다.
실행: node {skill_dir}/dist/index.js init <project_path>

### /impact analyze [--file <path>]
기획서를 입력받아 영향도를 분석합니다.
실행: node {skill_dir}/dist/index.js analyze [--file <path>]

### /impact view [--stop]
분석 결과 시각화 웹을 실행합니다.
실행: node {skill_dir}/dist/index.js view [--stop]

### /impact tickets [--create] [--detail <id>]
작업 티켓을 조회하거나 생성합니다.
실행: node {skill_dir}/dist/index.js tickets [--create] [--detail <id>]

### /impact config [--provider <name>] [--key <api_key>]
LLM 프로바이더 및 API 키를 설정합니다.
실행: node {skill_dir}/dist/index.js config [--provider <name>] [--key <api_key>]

### /impact reindex [--full]
코드 인덱스를 수동으로 갱신합니다.
실행: node {skill_dir}/dist/index.js reindex [--full]

### /impact policies [--search <keyword>] [add <content>]
정책 사전을 조회하거나 새 정책을 등록합니다.
실행: node {skill_dir}/dist/index.js policies [--search <keyword>] [add <content>]

### /impact owners [--add] [--edit <system>] [--remove <system>]
시스템별 담당자 및 팀 정보를 관리합니다.
실행: node {skill_dir}/dist/index.js owners [--add] [--edit <system>] [--remove <system>]

### /impact annotations [generate [path]] [view [path]]
보강 주석을 생성하거나 기존 보강 주석을 조회합니다.
실행: node {skill_dir}/dist/index.js annotations [generate [path]] [view [path]]

### /impact projects [--switch <name>] [--remove <name>] [--archive <name>]
멀티 프로젝트를 관리합니다 (전환, 제거, 아카이브).
실행: node {skill_dir}/dist/index.js projects [--switch <name>] [--remove <name>] [--archive <name>]

### /impact demo
샘플 데이터 기반으로 도구를 체험합니다.
실행: node {skill_dir}/dist/index.js demo

### /impact help [command] / /impact faq
도움말을 표시하거나 자주 묻는 질문(FAQ)을 조회합니다.
실행: node {skill_dir}/dist/index.js help [command]
```

#### 호출 방식 및 실행 구조

1. 사용자가 `/impact <command>` 입력
2. Claude Code Desktop/Antigravity가 SKILL.md의 명령어 매칭
3. 매칭된 실행 명령어(`node {skill_dir}/dist/index.js ...`)를 로컬에서 실행
4. Node.js 프로세스가 명령어를 처리하고 stdout으로 결과를 반환
5. Claude Code가 stdout을 사용자에게 표시

### 2.2 백엔드/코어 엔진

#### 언어: TypeScript (Node.js)

**선택 근거**:
- **통합 언어**: 프론트엔드(React)와 동일 언어로 타입 공유 가능, 개발 효율 극대화
- **풍부한 AST 파싱 생태계**: `@typescript-eslint/parser`, `@babel/parser`, `tree-sitter` 등 JavaScript/TypeScript 코드 분석 라이브러리가 Node.js에 가장 많음
- **Claude Code 스킬 호환**: 스킬 실행 환경이 Node.js 기반 (`node` 커맨드 실행)
- **비동기 I/O**: 파일 시스템 접근, LLM API 호출 등 I/O 바운드 작업에 적합
- **설치 용이성**: Node.js는 Claude Code Desktop 사용자 환경에 이미 설치되어 있을 가능성 높음

**대안 검토**:
| 대안 | 장점 | 탈락 이유 |
|------|------|----------|
| Python | ML/NLP 생태계 풍부 | 프론트엔드와 언어 분리, 배포 복잡 (가상환경), 타입 안전성 부족 |
| Go | 바이너리 배포 용이, 빠른 실행 | AST 파싱 생태계 부족, 프론트엔드 코드 공유 불가 |
| Rust | 성능 최고 | 개발 속도 느림, MVP에 과도한 선택, 학습 곡선 높음 |

#### 코드 인덱싱 엔진

| 구성 요소 | 라이브러리 | 용도 |
|----------|-----------|------|
| AST 파싱 (JS/TS) | `@swc/core` | SWC 기반 초고속 AST 파싱 (Rust로 작성, Node.js 바인딩) |
| AST 파싱 (다국어) | `tree-sitter` + 언어별 바인딩 | Java, Kotlin, Python 등 다국어 지원 |
| 의존성 분석 | `madge` + 커스텀 분석기 | import/require 그래프 추출 |
| 라우트 분석 | 커스텀 파서 (프레임워크별) | React Router, Vue Router, Next.js 등의 라우트 추출 |
| Git 연동 | `simple-git` | Git diff, log, status 조회 |
| 파일 탐색 | `fast-glob` | 고속 파일 시스템 탐색 |

**SWC 선택 근거**: Babel 대비 20~70배 빠른 파싱 속도. 10만 LOC 프로젝트 인덱싱을 5분 이내에 완료하기 위해 파싱 성능이 핵심. SWC는 Rust로 작성되어 있으나 Node.js 바인딩을 통해 JavaScript에서 직접 호출 가능.

#### 영향도 분석 엔진

- 파이프라인 패턴 기반 순차 처리
- 각 단계가 독립적으로 테스트 가능한 모듈 구성
- 단계 간 데이터는 표준화된 JSON 인터페이스로 전달

#### LLM 통합 레이어

- Provider Pattern (전략 패턴) 기반 추상화
- 각 프로바이더는 공통 인터페이스 구현
- 용도별 라우팅 테이블로 최적 프로바이더 자동 선택

### 2.3 프론트엔드 (시각화 웹)

#### 프레임워크: React 18 + TypeScript + Vite

**선택 근거**:
- **React Flow 호환**: 플로우차트 시각화의 핵심 라이브러리인 React Flow가 React 전용
- **생태계 크기**: 차트, UI 컴포넌트 라이브러리 선택지가 가장 넓음
- **Vite 번들러**: HMR 속도와 빌드 성능이 뛰어남. 로컬 개발과 배포 모두 적합
- **TypeScript 공유**: 코어 엔진과 타입 정의를 공유하여 데이터 계약 보장

**대안 검토**:
| 대안 | 장점 | 탈락 이유 |
|------|------|----------|
| Vue 3 | Vue Flow 사용 가능, 학습 곡선 낮음 | React Flow 대비 생태계 작음, 차트 라이브러리 선택폭 좁음 |
| Svelte | 번들 크기 작음, 성능 우수 | 플로우차트 라이브러리 부족, 생태계 작음 |
| Vanilla JS + D3 | 완전한 자유도 | 개발 속도 느림, UI 컴포넌트 직접 구현 필요 |

#### 플로우차트 라이브러리: React Flow v12

**선택 근거**:
- **커스텀 노드**: 8가지 노드 유형(요구사항/시스템/화면/기능/모듈/기획확인/정책/정책경고)을 각각 React 컴포넌트로 구현 가능 [추가 검토 반영: 6종 → 8종]
- **기본 기능**: 확대/축소, 패닝, 미니맵, 노드 선택 등 DESIGN-001에서 요구하는 인터랙션이 모두 내장
- **계층 레이아웃**: `dagre` 또는 `elkjs` 레이아웃 알고리즘과 조합하여 자동 배치 가능
- **성능**: 가상화 렌더링으로 100~300개 노드에서도 60fps 유지
- **활발한 유지보수**: 2026년 기준 v12로 활발히 개발 중

| 요구사항 (DESIGN-001) | React Flow 지원 |
|----------------------|----------------|
| 커스텀 노드 8가지 [추가 검토 반영] | Custom Node Component로 완벽 지원 |
| 노드 내 복합 콘텐츠 | JSX로 자유롭게 구성 |
| 확대/축소/패닝 | `<ReactFlow>` 기본 제공 |
| 미니맵 | `<MiniMap>` 컴포넌트 내장 |
| 전개/접기 | Custom logic + 노드 show/hide |
| 필터링 | 노드 style opacity/className 변경 |
| 연결선 커스텀 | Custom Edge Component |

> [피드백 R3 반영] **비표준 도형 노드 구현 방식**: DESIGN-001의 기획 확인 노드(다이아몬드)와 정책 변경 노드(육각형)는 React Flow Custom Node 내에서 SVG `<polygon>` 요소를 사용하여 렌더링합니다. 각 노드 컴포넌트(`CheckNode.tsx`, `PolicyNode.tsx`)에서 CSS `clip-path` 또는 인라인 SVG로 비표준 도형을 구현하고, React Flow의 nodeTypes에 등록합니다. 노드의 connection handle 위치도 도형에 맞게 커스터마이즈해야 합니다.

#### 차트 라이브러리: Recharts

**선택 근거**:

### 2.5 LLM 통합

#### 멀티 프로바이더 추상화 레이어

```typescript
// 공통 인터페이스
interface LLMProvider {
  name: string;
  chat(messages: Message[], options?: LLMOptions): Promise<LLMResponse>;
  estimateTokens(text: string): number;
  estimateCost(inputTokens: number, outputTokens: number): number;
  validateApiKey(key: string): Promise<boolean>;
}

// 용도별 라우팅
interface LLMRouter {
  route(task: LLMTask): LLMProvider;
}
```

#### 프로바이더별 모델 추천

| 프로바이더 | 추천 모델 | 주요 용도 | 장점 | 예상 비용 (10페이지 분석 1회) |
|-----------|----------|----------|------|----------------------------|
| **Anthropic** | Claude Sonnet 4 | 코드 분석, 영향도 추론, 점수 산출 | 코드 이해력 최상, 구조적 출력 우수 | ~$0.15~0.30 |
| **OpenAI** | GPT-4o | 기획서 파싱, 텍스트 요약 | 범용 텍스트 처리, 빠른 응답 | ~$0.10~0.20 |
| **Google** | Gemini 2.0 Flash | PDF/이미지 분석, 멀티모달 처리 | 이미지 기획서 처리, 대용량 컨텍스트 | ~$0.05~0.10 |

> **비용 절감 전략**: 코드 인덱싱 단계에서는 LLM을 사용하지 않고 정적 분석(AST)만 수행. LLM은 영향도 추론과 점수 산출에만 사용하여 API 호출 최소화.

#### 프롬프트 전략

1. **시스템 프롬프트**: 역할 정의 (코드 분석 전문가 / PM 어시스턴트)
2. **컨텍스트 주입**: 인덱스 데이터 + 기획서 파싱 결과를 구조화된 JSON으로 전달
3. **구조화 출력**: JSON Schema 또는 Tool Use를 활용하여 정형화된 응답 보장
4. **청킹 전략**: 대규모 인덱스는 화면 단위로 분할하여 병렬 처리
5. **프롬프트 템플릿 관리**: 별도 디렉토리에 `.prompt.md` 파일로 관리, 버전 관리 용이

#### 보강 주석 상세 분석 프롬프트 전략 [정책서 상세화 반영]

보강 주석 생성 시(Step 3.8) 기존 "함수 요약 + 정책명/설명" 수준을 넘어, 기획자가 활용할 수 있는 상세 정보를 추출하는 3단계 프롬프트 전략을 적용합니다.

**1단계: 함수 레벨 상세 분석 (MVP 포함)**

기존 보강 주석 생성 프롬프트를 확장하여, 각 함수에서 다음을 추가 추출합니다:

- 모든 IF/ELSE IF/ELSE/switch 분기를 자연어 + 원본 코드로 추출 (PolicyCondition[])
- 입출력 변수와 함수 내 상수값 추출 (PolicyVariable[], PolicyConstant[])
  - 상수값은 하드코딩/설정파일/환경변수/DB 중 출처를 판별
- 제약사항 도출: 하드코딩 값, 미지원 케이스, TODO/FIXME 주석, 예외 처리 부재 (PolicyConstraint[])
- 데이터 출처 파악: 각 입력 데이터의 궁극적 출처 추적 (PolicyDataSource[])
- 기획자 확인 사항 자동 도출: 하드코딩 값 확인, 미지원 케이스 검토 등 (PolicyReviewItem[])

프롬프트 핵심 지시:
```
분석 대상 함수의 모든 조건분기를 다음 형태로 변환하세요:
  IF {자연어 조건} THEN {자연어 결과}
상수값은 출처(하드코딩/설정파일/DB 등)를 반드시 판별하세요.
기획자(비개발자)가 코드 없이 이해할 수 있는 수준으로 작성하세요.
```

**비용 영향**: 기존 프롬프트 대비 입출력 토큰 약 1.5배 증가. 파일당 추가 비용 미미.

**2단계: 정책 간 관계 분석 (Phase 2)**

인덱스의 dependencies.json(함수 호출 그래프)과 보강 주석의 policies 필드를 조합하여 정책 간 의존/충돌 관계를 도출합니다.

- 정적 분석: 함수 호출 관계에서 정책 함수 간 caller/callee 관계 추출
- LLM 추론: 의미적 관계(같은 비즈니스 도메인 내 암묵적 의존) 보완
- 시스템 단위 1회 호출이므로 파일 수에 비례하지 않음

**3단계: 변경 영향 범위 분석 (Phase 2)**

인덱스의 컴포넌트 의존 관계 그래프와 라우트 정보를 활용하여 각 정책의 변경 영향 범위를 도출합니다.

- caller 추적으로 관련 API 엔드포인트, 화면(라우트) 식별
- 변경 시 예상 영향 1문장 요약 생성

**기획자 확인 사항 자동 생성 규칙** [정책서 상세화 반영]:

LLM이 기획자 확인 사항(reviewItems)을 도출할 때 적용하는 규칙:

| 트리거 | 생성 내용 | 우선순위 |
|--------|----------|---------|
| 상수값 source가 hardcoded | "이 값이 현재 비즈니스 요구와 일치하는가?" | high |
| 제약사항 type이 unsupported_case | "미지원 케이스가 향후 필요한가?" | high |
| confidence < 0.7 | "분석 신뢰도가 낮아 개발팀 검증 필요" | high |
| 의존관계 type이 conflict | "충돌 가능 정책의 동시 적용 규칙이 정의되어 있는가?" | medium |
| 영향 화면 3개 이상 | "변경 영향 범위가 넓어 모든 영향 화면 검토 필요" | medium |
| 조건분기 5개 이상 | "복잡한 분기 로직이 기획 의도와 일치하는가?" | medium |
| TODO/FIXME 주석 발견 | "기술적 부채가 향후 기능 확장에 영향 가능" | low |

> [피드백 R7 반영] **코드 인덱스 토큰 효율화 전략**: 10만 LOC 프로젝트의 전체 인덱스(5~15MB)를 LLM에 직접 전달하면 토큰 한도를 초과합니다. 다음과 같은 단계적 압축/선택 전략을 적용합니다:
>
> **1단계: 관련성 기반 인덱스 필터링 (LLM 호출 전)**
> - 기획서 파싱에서 추출한 키워드로 인덱스를 사전 필터링
> - 매칭되는 화면, 컴포넌트, API만 추출하여 "관련 인덱스 서브셋" 생성
> - 의존 그래프에서 매칭된 노드의 1-hop 이웃까지만 포함 (관련성 확장)
>
> **2단계: 인덱스 요약 형태 변환**
> - 전체 컴포넌트 JSON 대신 "화면명 - 컴포넌트명 - 파일경로 - import 목록" 형태의 요약 테이블 사용
> - API 엔드포인트는 "메서드 + 경로 + 핸들러명 + 파라미터" 한 줄 요약 형태
> - 의존 관계는 adjacency list 형태로 압축 (예: `ReviewSection → [ReviewFilter, ReviewSort, api-007]`)
>
> **3단계: 청킹 및 병렬 처리**
> - 화면 단위로 분할하여 각 화면별 독립적 LLM 호출 (Promise.all 병렬 처리)
> - 각 청크의 컨텍스트 예산: 입력 토큰 4,000~8,000 이내 유지
> - 화면 간 교차 영향은 최종 집계 단계에서 별도 LLM 호출로 판단
>
> **토큰 예산 분배** (10페이지 기획서 + 영향 화면 5개 기준):
> | 단계 | 예상 입력 토큰 | LLM | 호출 수 |
> |------|-------------|-----|--------|
> | 기획서 파싱 | ~3,000 | GPT-4o | 1회 |
> | 화면별 영향도 분석 | ~5,000 x 5 | Claude Sonnet | 5회 (병렬) |
> | 점수 산출 | ~2,000 x 5 | Claude Sonnet | 5회 (병렬) |
> | 교차 영향 집계 | ~3,000 | Claude Sonnet | 1회 |
> | **합계** | ~41,000 | - | 12회 |

```typescript
// src/router.ts
const COMMANDS = {
  'init':     InitCommand,
  'analyze':  AnalyzeCommand,
  'view':     ViewCommand,
  'tickets':  TicketsCommand,
  'config':   ConfigCommand,
  'reindex':  ReindexCommand,
  'demo':     DemoCommand,     // [피드백 R12 반영] 샘플 데이터 기반 체험 명령
  'help':     HelpCommand,     // [피드백 R12 반영] 인라인 도움말
  'faq':      FaqCommand,      // [피드백 R12 반영] 자주 묻는 질문
  'projects': ProjectsCommand, // [피드백 R13 반영] 멀티 프로젝트 관리
  'policies': PoliciesCommand, // [추가 검토 반영] 정책 사전 조회
  'owners':   OwnersCommand,   // [추가 검토 반영] 시스템별 담당자 관리
  'annotations': AnnotationsCommand, // [보강 주석 시스템 반영] 보강 주석 생성/조회
} as const;

export function route(args: string[]): Command {
  const commandName = args[0];
  const CommandClass = COMMANDS[commandName];
  if (!CommandClass) {
    throw new UnknownCommandError(commandName, Object.keys(COMMANDS));
  }
  return new CommandClass(args.slice(1));
}
```

```typescript
import { parseSync } from '@swc/core';

function parseFile(filePath: string, code: string): ParsedFile {
  const ast = parseSync(code, {
    syntax: filePath.endsWith('.tsx') ? 'typescript' : 'ecmascript',
    tsx: filePath.endsWith('.tsx'),
    jsx: filePath.endsWith('.jsx'),
  });

  return {
    imports: extractImports(ast),
    exports: extractExports(ast),
    components: extractComponents(ast),
    functions: extractFunctions(ast),
    apiCalls: extractApiCalls(ast),
    routeDefinitions: extractRoutes(ast),
  };
}
```

#### AnnotationFile 스키마 (보강 주석) [보강 주석 시스템 반영]

```typescript
// 보강 주석 파일 인터페이스
interface AnnotationFile {
  file: string;                    // 원본 파일 경로
  system: string;                  // 소속 시스템
  lastAnalyzed: string;            // ISO 8601 타임스탬프
  sourceHash: string;              // 원본 파일 SHA-256 해시 (변경 감지용)
  analyzerVersion: string;         // 분석 엔진 버전
  llmModel: string;                // 분석에 사용된 LLM 모델

  fileSummary: {
    description: string;           // 파일 전체 요약
    confidence: number;            // 0.0~1.0
    businessDomain: string;        // 비즈니스 도메인 (배송, 결제, 프로모션 등)
    keywords: string[];
  };

  annotations: FunctionAnnotation[];
}

interface FunctionAnnotation {
  line: number;                    // 시작 라인
  endLine: number;                 // 종료 라인
  function: string;                // 함수/메서드명
  signature: string;               // 함수 시그니처
  original_comment: string | null; // 원본 주석 (없으면 null)
  enriched_comment: string;        // LLM이 생성한 보강 주석
  confidence: number;              // 0.0~1.0
  type: 'business_logic' | 'utility' | 'data_access' | 'integration' | 'config';
  userModified: boolean;           // 사용자 수동 수정 여부
  lastModifiedBy: string | null;   // 수동 수정 시 수정자
  inferred_from: string;           // 추론 근거

  policies: InferredPolicy[];      // 추론된 정책 목록
  relatedFunctions: string[];      // 관련 함수 목록
  relatedApis: string[];           // 관련 API 목록
}

interface InferredPolicy {
  // === 기존 필드 (유지) ===
  name: string;                    // 정책명
  description: string;             // 정책 설명 (1~2문장 요약)
  confidence: number;              // 0.0~1.0
  category: string;                // 카테고리 (배송, 결제, 프로모션 등)
  inferred_from: string;           // 추론 근거

  // === 신규 필드 (상세 정책서용) [정책서 상세화 반영] ===

  // 조건-로직-결과 흐름
  conditions?: PolicyCondition[];  // IF/ELSE 조건 분기 목록
  defaultResult?: string;          // ELSE (기본) 결과 설명
  exceptionHandling?: string | null; // 예외 처리 로직 설명

  // 입출력 변수
  inputVariables?: PolicyVariable[];    // 입력 파라미터
  outputVariables?: PolicyVariable[];   // 출력(반환) 값
  constants?: PolicyConstant[];         // 사용 상수값 (하드코딩 여부 포함)
  internalVariables?: PolicyVariable[]; // 내부 핵심 변수

  // 제약사항 및 한계
  constraints?: PolicyConstraint[];     // 현재 제약사항 목록

  // 데이터 출처
  dataSources?: PolicyDataSource[];     // 데이터 출처 목록

  // 정책 간 관계 (Phase 2에서 채워짐)
  dependencies?: PolicyDependency[];    // 정책 간 의존 관계

  // 변경 영향 범위 (Phase 2에서 채워짐)
  impactScope?: PolicyImpactScope;      // 변경 시 영향 범위

  // 기획자 확인 사항 (LLM이 자동 도출)
  reviewItems?: PolicyReviewItem[];     // 기획자 확인 체크리스트
}

// [정책서 상세화 반영] 조건 분기 항목
interface PolicyCondition {
  order: number;                   // 분기 순서 (1, 2, 3...)
  type: 'if' | 'else_if' | 'else'; // 분기 유형
  condition: string;               // 조건 설명 (자연어)
  conditionCode: string;           // 원본 조건 코드
  result: string;                  // 결과 설명 (자연어)
  resultValue: string;             // 결과값 (예: "shippingFee = 0")
}

// [정책서 상세화 반영] 변수 정보
interface PolicyVariable {
  name: string;                    // 변수명
  type: string;                    // 타입 (boolean, number, string 등)
  description: string;             // 역할 설명 (자연어)
}

// [정책서 상세화 반영] 상수값 정보
interface PolicyConstant {
  name: string;                    // 상수명 또는 변수명
  value: string;                   // 현재 설정된 값
  type: string;                    // 타입
  description: string;             // 역할 설명
  source: 'hardcoded' | 'config_file' | 'env_variable' | 'db_query' | 'api_call';
  codeLocation: string;            // 값이 정의된 코드 위치 (파일:라인)
}

// [정책서 상세화 반영] 제약사항
interface PolicyConstraint {
  severity: 'warning' | 'info' | 'critical';
  type: 'hardcoded_value' | 'unsupported_case' | 'no_exception_handling'
        | 'tech_debt' | 'extension_limit';
  description: string;             // 제약사항 설명 (자연어)
  recommendation: string;          // 개선 권장사항
  relatedCode: string;             // 관련 코드 위치
}

// [정책서 상세화 반영] 데이터 출처
interface PolicyDataSource {
  variableName: string;            // 사용하는 변수명
  sourceType: 'db' | 'api' | 'user_input' | 'session' | 'config' | 'constant';
  sourceDetail: string;            // 상세 출처 (테이블명, API 경로 등)
  description: string;
}

// [정책서 상세화 반영] 정책 간 의존 관계 (Phase 2)
interface PolicyDependency {
  relationType: 'prerequisite' | 'dependent' | 'conflict' | 'related';
  policyName: string;
  description: string;
  functionName: string | null;
  filePath: string | null;
}

// [정책서 상세화 반영] 변경 영향 범위 (Phase 2)
interface PolicyImpactScope {
  callers: PolicyReference[];      // 이 정책을 호출하는 상위 함수/화면
  callees: PolicyReference[];      // 이 정책이 호출하는 하위 함수
  relatedApis: string[];           // 관련 API 엔드포인트
  affectedScreens: string[];       // 영향 받는 화면 목록
  impactSummary: string;           // 변경 시 영향 요약 (1문장)
}

interface PolicyReference {
  name: string;
  filePath: string;
  description: string;
}

// [정책서 상세화 반영] 기획자 확인 항목
interface PolicyReviewItem {
  priority: 'high' | 'medium' | 'low';
  category: 'logic_check' | 'value_check' | 'scope_check' | 'conflict_check' | 'missing_case';
  question: string;                // 확인 질문 (자연어)
  context: string;                 // 왜 확인이 필요한지 설명
  relatedConstraint: string | null;
}

// 보강 주석 메타 정보
interface AnnotationMeta {
  version: string;
  createdAt: string;
  lastUpdatedAt: string;
  totalFiles: number;
  totalAnnotations: number;
  totalPolicies: number;
  systems: Record<string, {
    files: number;
    annotations: number;
    policies: number;
  }>;
  avgConfidence: number;
  lowConfidenceCount: number;      // confidence < 0.5인 항목 수
  userModifiedCount: number;
}
```

**보강 주석 파일 저장 위치**: `.impact/projects/<project-id>/annotations/{시스템명}/{파일경로}.annotations.yaml`

## 5. 영향도 분석 엔진

> [분석 일관성 전략 반영] 분석 파이프라인은 다층 분석 전략(Multi-Layer Analysis)을 적용하여, 주석이 없는 시스템에서도 Layer 1(구조), Layer 2(의존성), Layer 4(LLM 추론)를 통해 최소한의 영향도 분석이 가능합니다. Step 3(인덱스 매칭)이 Layer 1+2에 해당하고, Step 4(LLM 분석)가 Layer 4에 해당하며, Step 5.5(정책 매칭)가 Layer 3에 해당합니다.

### 5.1 분석 파이프라인

```
[Step 1: 기획서 파싱]
│
│  입력: 텍스트 / PDF
│  처리: LLM (GPT-4o)으로 구조화
│  출력: ParsedSpec {
│          requirements: Requirement[], // [추가 검토 반영] 기획서에서 복수 요구사항 도출
│          features: Feature[],
│          businessRules: Rule[],
│          targetScreens: string[],
│          keywords: string[]
│        }
│
▼
[Step 2: 키워드/의도 추출]
│
│  입력: ParsedSpec
│  처리: 기능별 키워드 추출 + 동의어 확장
│  출력: SearchQuery[] (검색 쿼리 목록)
│
▼
[Step 3: 인덱스 매칭]
│
│  입력: SearchQuery[] + CodeIndex
│  처리:
│    (a) 키워드 기반 텍스트 매칭 (파일명, 컴포넌트명, API 경로)
│    (b) 의존 그래프 탐색 (매칭된 노드의 상위/하위 추적)
│    (c) 유사도 점수 기반 후보 목록 생성
│  출력: MatchedEntities[] {
│          screens, components, apis, models
│          + 각각의 매칭 신뢰도 점수
│        }
│
▼
[Step 4: LLM 영향도 분석]
│
│  입력: ParsedSpec + MatchedEntities + 관련 코드 스니펫 + 보강 주석 [보강 주석 시스템 반영]
│  처리: LLM (Claude Sonnet)으로 영향도 심층 분석
│    - 보강 주석이 존재하는 파일은 추가 컨텍스트로 제공 [보강 주석 시스템 반영]
│    - 각 매칭된 엔티티가 실제로 영향 받는지 검증
│    - 작업 유형 분류 (신규/수정/설정)
│    - FE/BE 구분
│    - 기획 확인 사항 도출
│    - 정책 변경 필요 여부 판단
│  출력: ImpactResult {
│          affectedScreens: ScreenImpact[],
│          tasks: Task[],
│          planningChecks: Check[],
│          policyChanges: PolicyChange[]
│        }
│
▼
[Step 5: 점수 산출]
│
│  입력: ImpactResult
│  처리: LLM (Claude Sonnet) + 규칙 기반 하이브리드
│    - 4차원 점수 산출 (개발복잡도, 영향범위, 정책변경, 의존성)
│    - 각 점수의 산출 근거 텍스트 생성
│    - 종합 점수 계산 및 등급 결정
│  출력: ScoredResult {
│          screenScores: ScreenScore[],
│          totalScore: number,
│          grade: 'Low' | 'Medium' | 'High' | 'Critical',
│          recommendation: string
│        }
│
▼
[Step 5.5: 정책 사전 매칭 및 담당자 매핑] [추가 검토 반영]
│
│  입력: ImpactResult + PolicyIndex + OwnersConfig
│  처리:
│    (a) 영향 받는 코드/모듈 근처의 정책 주석 매칭
│    (b) 정책 위반 가능성 평가 (변경 내용 vs 정책 내용)
│    (c) 영향 받는 시스템별 담당자 자동 매핑
│    (d) 담당자별 확인 요청 메일 초안 생성
│  출력: EnrichedResult {
│          ...ScoredResult,
│          policyWarnings: PolicyWarning[],
│          ownerNotifications: OwnerNotification[]
│        }
│
▼
[Step 5.7: 신뢰도 점수 산출] [분석 일관성 전략 반영]
│
│  입력: EnrichedResult + ProjectProfile + AnnotationMeta [보강 주석 시스템 반영]
│  처리:
│    (a) Layer별 정보 제공 정도 평가
│        - Layer 1 (구조): 프레임워크 파서 성공률, 화면/API 추출 성공 여부
│        - Layer 2 (의존성): 의존 그래프 완성도, import 추적 성공률
│        - Layer 3 (정책): 정책 주석 밀도, 정책 사전 항목 수, 보강 주석 존재 여부 [보강 주석 시스템 반영]
│        - Layer 4 (LLM): LLM 추론 성공률, 근거 명확도, 보강 주석 참조 여부 [보강 주석 시스템 반영]
│    (b) 가중 합산 (L1: 25%, L2: 25%, L3: 20%, L4: 30%)
│    (c) 시스템별 신뢰도 등급 결정 (High/Medium/Low/Very Low)
│    (d) 신뢰도 Low 이하 시스템에 대한 보완 가이드 생성
│  출력: ConfidenceEnrichedResult {
│          ...EnrichedResult,
│          confidenceScores: SystemConfidence[],
│          lowConfidenceWarnings: ConfidenceWarning[]
│        }
│
▼
[Step 6: 결과 직렬화]
│
│  입력: ConfidenceEnrichedResult
│  처리: 결과 JSON 저장, 시각화 데이터 구조 변환
│  출력: .impact/results/<date>_<title>/result.json
```

### 5.2 프롬프트 엔지니어링 전략

#### 기획서 파싱 프롬프트

```markdown
# Role
```

### 5.3 점수 산출 로직 구현

```typescript
interface ScoreBreakdown {
  developmentComplexity: { score: number; weight: 0.35; rationale: string };
  impactScope:          { score: number; weight: 0.30; rationale: string };
  policyChange:         { score: number; weight: 0.20; rationale: string };
  dependencyRisk:       { score: number; weight: 0.15; rationale: string };
}

function calculateTaskScore(breakdown: ScoreBreakdown): number {
  return (
    breakdown.developmentComplexity.score * 0.35 +
    breakdown.impactScope.score * 0.30 +
    breakdown.policyChange.score * 0.20 +
    breakdown.dependencyRisk.score * 0.15
  );
}

function calculateScreenScore(tasks: TaskScore[]): number {
  return tasks.reduce((sum, task) => sum + task.totalScore, 0);
}

function calculateTotalScore(screens: ScreenScore[]): number {
  return screens.reduce((sum, screen) => sum + screen.screenScore, 0);
}

function determineGrade(totalScore: number): Grade {
  if (totalScore <= 15) return 'Low';
  if (totalScore <= 40) return 'Medium';
  if (totalScore <= 70) return 'High';
  return 'Critical';
}

function getRecommendation(grade: Grade): string {
  const recommendations: Record<Grade, string> = {
    'Low':      '일반 스프린트 내 처리 가능합니다.',
    'Medium':   '스프린트 계획 시 우선순위 조정이 필요합니다.',
    'High':     '별도 프로젝트 계획이 필요합니다. 리소스 배분을 검토하세요.',
    'Critical': '아키텍처 리뷰가 필수입니다. 단계별 릴리즈 계획을 수립하고 경영진 보고를 권고합니다.',
  };
  return recommendations[grade];
}
```

```typescript
}

function calculateConfidenceScore(layers: LayerConfidence): number {
  return (
    layers.layer1Structure.score * 0.25 +
    layers.layer2Dependency.score * 0.25 +
    layers.layer3Policy.score * 0.20 +
    layers.layer4LLM.score * 0.30
  );
}

function determineConfidenceGrade(score: number): ConfidenceGrade {
  if (score >= 85) return 'high';
  if (score >= 65) return 'medium';
  if (score >= 40) return 'low';
  return 'very_low';
}

interface ProjectProfile {
  projectId: string;
  analyzedAt: string;
  overallReadiness: number;              // 0~100 분석 준비도
  expectedConfidence: ConfidenceGrade;   // 예상 신뢰도 등급
  metrics: {
    frameworkRecognition: { score: number; framework: string; details: string };

    commentDensity: { score: number; ratio: number; totalComments: number; totalLines: number };
    policyAnnotations: { score: number; count: number };

      score: number;

      totalAnnotations: number;

      avgConfidence: number;
    };
    dependencyGraph: { score: number; trackedModules: number; totalModules: number };

    typeCoverage: { score: number; tsFiles: number; totalFiles: number };
  };

  description: string;
  impact: string;              // "Layer 3 점수 향상 → 정책 분석 활성화" 등
}

interface ManualPolicy {
  name: string;
  description: string;
  category: string;            // "배송", "결제", "프로모션" 등
  system: string;              // 관련 시스템명
  relatedPaths: string[];      // 관련 코드 경로
}

interface PoliciesYaml {
  policies: ManualPolicy[];
}

/** Layer 3 점수 산출 (보강 주석 반영) [보강 주석 시스템 반영] */
function calculateLayer3Score(
  policyAnnotationCount: number,
  hasReadmePolicy: boolean,
  hasManualPolicies: boolean,
  enrichedAnnotations: { exists: boolean; avgConfidence: number; policyCount: number; userModifiedCount: number }
): number {
  let score = 0;

  // 기존 점수 산출
  if (policyAnnotationCount >= 1) score += 40;
  if (policyAnnotationCount >= 5) score += 20;
  if (hasReadmePolicy) score += 20;
  if (hasManualPolicies) score += 20;

  // 보강 주석 보너스 [보강 주석 시스템 반영]
  if (enrichedAnnotations.exists) score += 15;
  if (enrichedAnnotations.avgConfidence >= 0.7) score += 10;
  if (enrichedAnnotations.policyCount >= 5) score += 10;
  if (enrichedAnnotations.userModifiedCount > 0) score += 5;

  return Math.min(score, 100);  // 최대 100점
}
```

### 7.1 추상화 레이어 인터페이스

```typescript
// src/llm/types.ts

export interface LLMMessage {
  role: 'system' | 'user' | 'assistant';
  content: string | ContentBlock[];
}

export interface ContentBlock {
  type: 'text' | 'image';
  text?: string;
  imageUrl?: string;
  mimeType?: string;
}

export interface LLMOptions {
```

#### 프로바이더 레지스트리

```typescript
// src/llm/registry.ts
export class ProviderRegistry {
  private providers = new Map<string, LLMProvider>();

  register(provider: LLMProvider): void {
    this.providers.set(provider.name, provider);
  }

  get(name: string): LLMProvider {
    const provider = this.providers.get(name);
    if (!provider) throw new ProviderNotFoundError(name);
    return provider;
  }

  list(): ProviderInfo[] {
    return Array.from(this.providers.values()).map(p => ({
      name: p.name,
      displayName: p.displayName,
      models: p.listModels(),
    }));
  }
}
```

#### 새 프로바이더 추가 방법


> [피드백 R8 반영] **새 LLM 프로바이더 추가 절차 간편성**: 위 3단계만으로 새 프로바이더 추가가 완료됩니다. (1) `LLMProvider` 인터페이스를 구현하는 클래스 1개 작성 (~100줄), (2) `registry.register()` 호출 1줄 추가, (3) 필요 시 라우팅 테이블에 1줄 추가. 코어 엔진이나 프론트엔드 변경은 불필요합니다. `/impact config`의 프로바이더 목록은 registry에서 자동 생성되므로 UI 변경도 없습니다.

### 7.3 용도별 라우팅 전략

```typescript
// src/llm/router.ts
export class LLMRouter {
  // 기본 라우팅 테이블
  private routingTable: Record<LLMTask, string> = {
    'spec-parsing':       'openai',       // GPT-4o: 빠른 텍스트 처리
    'impact-analysis':    'anthropic',    // Claude: 코드 이해력 최상
    'score-calculation':  'anthropic',    // Claude: 구조적 출력 우수
    'multimodal-parsing': 'google',       // Gemini: 이미지 분석 강점
    'general':            'openai',       // GPT-4o: 범용
  };

  route(task: LLMTask): LLMProvider {
    const providerName = this.routingTable[task];
    const provider = this.registry.get(providerName);

    // 해당 프로바이더가 설정되지 않은 경우 폴백
    if (!provider) {
      return this.getFallbackProvider();
    }
    return provider;
  }

  // 사용자가 라우팅 테이블 커스터마이즈 가능
  setRoute(task: LLMTask, providerName: string): void {
    this.routingTable[task] = providerName;
  }

  // 폴백: 설정된 프로바이더 중 첫 번째
  private getFallbackProvider(): LLMProvider {
    const available = this.registry.list();
    if (available.length === 0) throw new NoProviderConfiguredError();
    return this.registry.get(available[0].name);
  }
}
```

**라우팅 폴백 체인**:
- 1순위: 용도별 지정 프로바이더
- 2순위: 기본 프로바이더 (config.json에서 설정)
- 3순위: 설정된 프로바이더 중 첫 번째

### 7.4 비용 관리

```typescript
// src/llm/cost-tracker.ts
export class CostTracker {
  private sessionCosts: CostRecord[] = [];

  record(response: LLMResponse): void {
    this.sessionCosts.push({
      timestamp: new Date(),
      provider: response.provider,
      model: response.model,
      inputTokens: response.usage.inputTokens,
      outputTokens: response.usage.outputTokens,
      cost: response.usage.estimatedCost,
    });
  }

  getSessionSummary(): CostSummary {
    return {
      totalCost: this.sessionCosts.reduce((sum, r) => sum + r.cost, 0),
      totalTokens: this.sessionCosts.reduce((sum, r) =>
        sum + r.inputTokens + r.outputTokens, 0),
      byProvider: this.groupByProvider(),
      callCount: this.sessionCosts.length,
    };
  }

  // 분석 전 예상 비용 안내
  // [피드백 R2 반영] PM에게는 토큰 수가 아닌 예상 금액(USD/KRW)을 중심으로 안내.
  // CLI 출력 시 "이번 분석에 약 $0.25 (약 330원)이 소요될 예정입니다." 형태로 표시.
  estimateAnalysisCost(indexSize: number, specLength: number): CostEstimate {
    // 인덱스 크기와 기획서 길이 기반 토큰 수 추정
    const estimatedInputTokens = Math.ceil(indexSize / 4) + Math.ceil(specLength / 4);
    const estimatedOutputTokens = Math.ceil(estimatedInputTokens * 0.3);
    const estimatedCostUsd = this.calculateCost(estimatedInputTokens, estimatedOutputTokens);

    return {
      estimatedCost: estimatedCostUsd,
      estimatedCostKrw: Math.ceil(estimatedCostUsd * 1350), // 원화 환산 (대략)
      estimatedTokens: estimatedInputTokens + estimatedOutputTokens,
      displayMessage: `이번 분석에 약 $${estimatedCostUsd.toFixed(2)} (약 ${Math.ceil(estimatedCostUsd * 1350)}원)이 소요될 예정입니다.`,
      warning: estimatedInputTokens > 100000
        ? '대규모 분석입니다. 비용이 높을 수 있습니다. 계속하시겠습니까?'
        : undefined,
    };
  }
}
```

**비용 절감 전략**:
1. **코드 인덱싱에 LLM 미사용**: AST 파싱은 100% 로컬 처리
2. **청킹 + 병렬 처리**: 화면 단위로 분할하여 컨텍스트 크기 최적화
3. **캐시**: 동일 기획서 + 동일 인덱스 조합은 결과 재사용
4. **분석 전 비용 예측**: 사용자에게 예상 비용 안내 후 진행 여부 확인

> [피드백 R9 반영] **LLM 전송 시 민감 정보 기본 보호 (MVP)**:
> 코드 마스킹 고급 옵션은 Phase 2로 미뤄졌지만, MVP에서도 기본적인 민감 정보 보호가 필요합니다:
>
> **1. 인덱스 단계에서의 필터링** (LLM 호출 전, 로컬 처리):
> - `.env`, `.env.local`, `credentials.json`, `*.pem`, `*.key` 등 민감 파일은 인덱싱에서 자동 제외
> - `.gitignore` 패턴을 준수하여 비추적 파일 제외
>
> **2. 코드 스니펫 전송 시 기본 마스킹**:
> - 코드 내 환경 변수 값(`process.env.XXX = "actual_value"`)에서 실제 값을 `***` 로 치환
> - API 키 패턴 자동 감지 (`sk-`, `AIza`, `ghp_` 등 알려진 패턴) 및 마스킹
> - 하드코딩된 URL의 호스트명은 유지하되 credential 파라미터는 마스킹
>
> **3. 사용자 동의 프로세스**:
> - 최초 분석 시 "코드 일부가 LLM API를 통해 외부로 전송됩니다. 진행하시겠습니까?" 확인
> - 동의 여부를 config.json에 저장하여 매번 묻지 않도록 처리
> - `/impact config --show-sent-data` 명령으로 마지막 분석에서 LLM에 전송된 데이터를 확인 가능

#### LLM 병렬 호출 전략

```typescript
// 화면별 청킹 시 동시 호출 수 제한
const LLM_CONCURRENCY = {
  'impact-analysis': 3,    // 영향도 분석: 동시 3개 화면 병렬 처리
  'score-calculation': 5,  // 점수 산출: 동시 5개 (입력 토큰이 적어 가벼움)
  'spec-parsing': 1,       // 기획서 파싱: 순차 1건 (1회만 호출)
};

// Promise.allSettled + p-limit 라이브러리로 동시성 제어
import pLimit from 'p-limit';

const analysisLimit = pLimit(LLM_CONCURRENCY['impact-analysis']);
const screenAnalyses = screens.map(screen =>
  analysisLimit(() => analyzeScreen(screen, context))
);
const results = await Promise.allSettled(screenAnalyses);
```

- **동시 호출 수 제한 이유**: 대부분의 LLM API에서 동시 요청 제한(RPM)이 있으므로 무제한 병렬은 rate limit을 유발
- **화면 5개 기준**: 3개 병렬 + 2개 대기 -> 약 2라운드에 완료 (라운드당 ~30초 가정 시 ~60초)
- **실패 처리**: `Promise.allSettled`로 개별 실패를 격리, 성공한 화면 결과만 수집

#### 시각화 웹 초기 로딩 최적화

```
[목표: 2초 이내 첫 화면 표시]

1. 코드 스플리팅 전략:
   - Route-based splitting: Dashboard / FlowChart / Checklist / Owners / Tickets 각각 lazy import
   - React.lazy + Suspense로 탭 전환 시 동적 로드
   - React Flow는 FlowChart 탭 진입 시에만 로드 (~150KB gzipped)

2. 초기 로딩 우선순위:
   [0ms]    index.html + 기본 CSS (인라인)
   [200ms]  Dashboard 번들 로드 (메인 진입점)
   [500ms]  분석 결과 JSON fetch (GET /api/results/latest)
   [800ms]  Dashboard 렌더링 완료 (KPI 카드, 차트)
   [사용자 탭 전환 시] FlowChart/Checklist/Owners/Tickets 번들 로드

3. 빌드 최적화:
   - Vite의 rollupOptions.output.manualChunks로 vendor 분리
   - react-flow, recharts, dagre를 별도 청크로 분리
   - gzip 압축 (Vite 기본 지원)
   - 전체 빌드 크기 목표: 초기 번들 < 100KB (gzipped), 전체 < 500KB (gzipped)
```

#### 100+ 노드 렌더링 성능

```
[React Flow 가상화 + LOD 전략]

노드 수에 따른 렌더링 모드:
- 1~50 노드: Full Rendering (모든 노드 완전 렌더링)
- 51~100 노드: 뷰포트 내 Full + 뷰포트 외 Simplified (텍스트만, 아이콘 생략)
- 101~300 노드: 뷰포트 내 Full + 뷰포트 외 Placeholder (단색 박스)
- 300+ 노드: 기본 접힌 상태로 시작 + 사용자 전개 시 점진적 렌더링

LOD (Level of Detail) 구현:
- React Flow의 onNodeInViewport 콜백으로 뷰포트 내 노드 감지
- 줌 레벨에 따라 노드 상세도 조절:
  - 줌 100%+: 전체 내용 표시
  - 줌 50~99%: 이름 + 점수만 표시
  - 줌 50% 미만: 색상 박스만 표시 (라벨 생략)
```

#### 인덱스 파일 크기 관리 및 압축

| 프로젝트 규모 | 인덱스 예상 크기 | 관리 전략 |
|-------------|---------------|----------|
| ~10만 LOC | 5~15MB | 비압축 JSON (디버깅 용이, 디스크 영향 미미) |
| ~30만 LOC | 15~45MB | JSON 비압축 유지, 이전 버전 1개만 보관 |

- `/impact status` 명령으로 인덱스/결과/전체 디스크 사용량 확인 가능

### 엣지 케이스 기술 구현 [피드백 R11 반영]

PLAN-001 9.5절에서 정의한 엣지 케이스에 대한 기술 구현 세부사항:

```typescript
// src/core/indexing/scanner.ts - 대규모 프로젝트 감지
interface ScanOptions {
  scope?: string[];           // --scope 옵션으로 전달된 디렉토리
  maxLOCWarning: number;      // 30만 LOC 이상 시 경고 (기본값)
  timeoutMinutes: number;     // 인덱싱 타임아웃 (기본값: 10분)
}

// src/core/analysis/matcher.ts - 매칭률 검증
interface MatchValidation {
  minimumMatchThreshold: number;  // 최소 매칭률 (기본: 0.1 = 10%)
  noMatchAction: 'abort' | 'warn'; // 매칭 0% 시 동작
}

// src/llm/retry.ts - Rate Limit 핸들링
interface RetryConfig {
  maxRetries: number;         // 최대 재시도 횟수 (기본: 3)
  backoffBase: number;        // 백오프 기본 시간 (ms, 기본: 1000)
  backoffMultiplier: number;  // 지수 백오프 배수 (기본: 2)
  fallbackProvider?: string;  // 폴백 프로바이더
}
```

파일 접근 오류 시 Graceful Degradation 흐름:
```
파일 접근 시도 → 실패 → 3회 재시도(파일 잠금) 또는 건너뛰기(권한/인코딩)
→ 실패 파일 목록을 indexing-errors.log에 기록
→ 나머지 파일 정상 인덱싱 계속
→ 완료 시 "N개 파일 제외" 경고 표시
```
