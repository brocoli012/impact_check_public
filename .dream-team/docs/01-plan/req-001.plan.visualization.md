## 6. 시각화 설계 방향

### 6.1 플로우차트 구조 (요구사항 > 시스템 > 화면 > 하위기능 + 모듈) [추가 검토 반영]

> [추가 검토 반영] 기존 "시스템 > 화면 > 하위기능 > 모듈" 4단계에서, 최상위에 "요구사항" 레벨을 추가하여 5단계 계층으로 변경합니다. 하나의 기획서에서 여러 요구사항이 도출될 수 있고, 각 요구사항이 서로 다른 시스템에 영향을 줄 수 있으므로, "이 요구사항이 어떤 시스템들에 걸쳐 영향을 주는지"를 한눈에 파악할 수 있도록 합니다. 하나의 시스템이 여러 요구사항에서 공유될 수 있으며, 이 경우 시스템 노드가 여러 요구사항에 연결됩니다.

```
[기획서: 장바구니 개선]
    │
    ├── [요구사항A: 쿠폰 자동 적용]
    │     ├── [시스템: 컬리 커머스]
    │     │     └── [화면: 장바구니]
    │     │           └── [기능: 쿠폰 적용 UI]
    │     │                 ├── [모듈: CouponApply.vue] (FE, 5점)
    │     │                 └── [모듈: /api/cart/coupon] (BE, 6점)
    │     └── [시스템: 프로모션]
    │           └── [화면: 쿠폰 관리]
    │                 └── [기능: 자동 적용 로직]
    │                       └── [모듈: /api/promotion/auto-apply] (BE, 7점)
    │
    ├── [요구사항B: 배송일 표시 개선]
    │     ├── [시스템: 컬리 커머스]
    │     │     └── [화면: 장바구니]
    │     │           └── [기능: 배송일 표시]
    │     │                 └── [모듈: DeliveryDate.vue] (FE, 4점)
    │     └── [시스템: 물류]
    │           └── [화면: 배송 스케줄]
    │                 └── [기능: 배송일 계산]
    │                       └── [모듈: /api/logistics/delivery-date] (BE, 5점)
    │
    ├── [정책 경고] [추가 검토 반영]
    │     ├── [정책: 컬리 멤버스 무료배송 - 프로모션 시스템]
    │     └── [정책: 할인 중복 적용 규칙 - 커머스 시스템]
    │
    └── [확인 요청 안내] [추가 검토 반영]
          ├── [프로모션 팀 - 김OO: 쿠폰 자동 적용 시 기존 프로모션 중복 적용 정책 확인]
          └── [물류개발팀 - 이OO: 배송일 계산 로직 변경 시 기존 SLA 영향 검토]
```

### 6.2 노드 유형 및 표현 방식 [추가 검토 반영: 기존 6종 → 8종]

| 노드 유형 | 모양 | 색상 체계 | 표시 정보 |
|----------|------|----------|----------|
| **요구사항** [추가 검토 반영] | 큰 둥근 사각형 (이중 테두리) | 진한 인디고 (`#4338CA`) | 요구사항명, 영향 시스템 수, 총 점수 |
| 시스템 | 큰 둥근 사각형 | 진한 남색 | 시스템명, 총 점수, 등급 |
| 화면 | 중간 사각형 | 등급별 색상 (Low=초록, Medium=노랑, High=주황, Critical=빨강) | 화면명, 화면 점수 |
| 기능 | 작은 둥근 사각형 | 등급별 색상 | 기능명, 작업 유형(신규/수정), 점수 |
| 모듈 | 작은 원형/태그 | FE=파랑, BE=녹색 | 파일/모듈명, 점수 |
| 기획 확인 | 다이아몬드 | 보라색 | 확인 사항, 긴급도 |
| 정책 변경 | 육각형 | 빨간 테두리 | 정책 내용, 영향도 |
| **정책 경고** [추가 검토 반영] | 방패 형태 | 주황 배경 + 빨간 테두리 | 정책명, 위반 가능성, 관련 시스템 |

### 6.3 참고 이미지 기반 설계 방향

**PP-SOL-REQ 스타일 (플로우차트형)** 참고:
- 기획서의 요구사항(REQ)이 어떤 문제(PP)에서 출발하여 어떤 해결책(SOL)을 거쳐 도출되었는지 추적 가능하게 표현
- 각 노드에 중요도 뱃지, 예상 작업량, 상태 표시
- 노드 간 화살표로 인과 관계 및 의존 관계 표현
- 이 패턴을 영향도 분석에 적용: **기획 요구사항 -> 영향 화면 -> 영향 기능 -> 작업 모듈** 흐름

**Obsidian 캔버스 스타일 (카드형)** 참고:
- 카드 형태의 노드에 상세 정보를 담아 한눈에 파악
- 카테고리별 그룹핑 (기본 작업 / 고급 작업, 또는 FE / BE)
- 각 카드에 설명, 관련 파일 경로, 점수 등 상세 정보 포함
- 중앙 허브 노드에서 관련 노드로 방사형 연결
- 이 패턴을 영향도 분석에 적용: **화면별 카드 그룹** 내에 **기능 카드**들을 배치, 카드 내에 작업 상세 표시

**시각화 뷰 전환**:
- 사용자가 플로우차트 뷰와 캔버스(카드) 뷰를 전환하여 볼 수 있도록 설계
- 플로우차트: 전체 흐름과 계층 파악에 적합
- 캔버스: 상세 정보 확인과 작업 계획에 적합
