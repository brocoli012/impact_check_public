## 사용자 스토리 (US-001 ~ US-013)

### US-001: 스킬 설치 및 초기 설정
> **As a** 컬리 PM,
> **I want to** Claude Code Desktop 또는 Antigravity에 Kurly Impact Checker 스킬을 설치하고, 내 LLM API 키를 등록할 수 있다.
> **So that** 별도 도구 설치 없이 기존 AI 도구 환경에서 영향도 분석을 바로 시작할 수 있다.

**수용 기준 (AC)**:
- [ ] 스킬 설치 후 `/impact` 또는 유사 명령어로 스킬 호출 가능
- [ ] 최초 실행 시 LLM 프로바이더 선택 및 API 키 입력 가이드 제공
- [ ] API 키가 로컬에 안전하게 저장됨
- [ ] 잘못된 API 키 입력 시 명확한 오류 메시지 표시

### US-002: 프로젝트 등록 및 코드 인덱싱
> **As a** 컬리 PM,
> **I want to** 분석 대상 프로젝트의 Git 레포 경로를 등록하면 자동으로 코드 인덱스가 생성된다.
> **So that** 이후 영향도 분석 시 전체 코드를 매번 읽지 않고 빠르게 분석할 수 있다.

**수용 기준 (AC)**:
- [ ] 로컬 Git 레포 경로 지정 시 전체 프로젝트 인덱싱 실행
- [ ] 인덱스에 화면(Screen), 기능(Feature), 모듈(Module), API 엔드포인트 정보 포함
- [ ] 인덱싱 진행 상황(%) 표시
- [ ] 인덱싱 완료 후 프로젝트 구조 요약 제공
- [ ] 인덱스 파일이 로컬에 저장되어 재사용 가능

### US-003: 코드 인덱스 자동 업데이트
> **As a** 컬리 PM,
> **I want to** Git에서 새 소스를 pull 받으면 변경된 부분만 인덱스가 자동 업데이트된다.
> **So that** 항상 최신 코드 기반으로 정확한 영향도를 분석할 수 있다.

**수용 기준 (AC)**:
- [ ] `/impact analyze` 실행 시 마지막 인덱싱 이후 Git 커밋 변경을 자동 감지하여 부분 재인덱싱 수행 [피드백 R1 반영: "자동"의 의미를 명확화 - 상시 감시가 아닌 분석 명령 실행 시점에 트리거]
- [ ] 변경 파일만 부분 인덱싱 (전체 재인덱싱 불필요)
- [ ] 업데이트된 항목 수 및 변경 요약 표시
- [ ] 인덱스 버전 관리 (이전 인덱스 복원 가능)

### US-004: 기획서 입력
> **As a** 컬리 PM,
> **I want to** PDF, Confluence 링크, 텍스트, 이미지 등 다양한 형태로 기획서를 입력할 수 있다.
> **So that** 기존 작성한 기획서를 형식 변환 없이 그대로 활용할 수 있다.

**수용 기준 (AC)**:
- [ ] 텍스트 직접 입력 지원 (채팅 형태로 기획 의도 전달)
- [ ] PDF 파일 첨부 및 내용 파싱
- [ ] ~~Confluence 페이지 URL 입력 시 내용 크롤링/파싱~~ → Phase 2로 이동 (MVP 범위 외) [피드백 R10 반영]
- [ ] ~~이미지 첨부 시 OCR 또는 멀티모달 분석으로 내용 추출~~ → Phase 2로 이동 (MVP 범위 외) [피드백 R10 반영]
- [ ] 10페이지 이내 기획서 처리 가능
- [ ] 파싱 결과를 사용자에게 확인 요청 (오파싱 방지)

### US-005: 영향도 분석 실행
> **As a** 컬리 PM,
> **I want to** 기획서를 입력하면 해당 기능이 기존 시스템의 어떤 화면/기능/모듈에 영향을 미치는지 자동 분석된다.
> **So that** 개발팀에 직접 문의하지 않고도 영향 범위를 빠르게 파악할 수 있다.

**수용 기준 (AC)**:
- [ ] 기획서 내용과 코드 인덱스를 매칭하여 영향 받는 화면 목록 도출
- [ ] 각 화면 내 영향 받는 기능(하위기능) 단위 분석
- [ ] 백엔드/프론트엔드 작업 사항 분류
- [ ] 필요한 기획 사항 및 확인해야 할 정책 식별
- [ ] 신규 개발 vs. 기존 기능 수정 구분
- [ ] 분석 소요 시간 3분 이내 (10페이지 기획서 + 10만 LOC 이하 프로젝트 + 영향 화면 5개 이하 기준, LLM API 응답 지연 제외) [피드백 R1 반영: 기술적 전제 조건 구체화 - 프로젝트 규모, 영향 범위, 네트워크 변수를 명시]

### US-006: 난이도 점수화 및 등급 산출
> **As a** 컬리 PM,
> **I want to** 영향 받는 각 화면/기능에 대해 개발 난이도를 점수로 확인하고, 종합 등급을 받을 수 있다.
> **So that** 일정 산정과 우선순위 결정에 객관적 근거를 활용할 수 있다.

**수용 기준 (AC)**:
- [ ] 화면/기능 단위로 개발 난이도 점수 부여 (1~10점)
- [ ] 점수 산출 근거(기준별 세부 점수) 투명하게 표시
- [ ] 종합 점수 자동 산출
- [ ] 종합 점수 기반 영향도 등급 제안 (Low / Medium / High / Critical)
- [ ] 등급별 권장 액션 안내 (예: Critical이면 "아키텍처 리뷰 필요")

### US-007: 작업 티켓 생성
> **As a** 컬리 PM,
> **I want to** 분석 결과를 바탕으로 개발 작업 티켓이 자동 생성된다.
> **So that** 분석 결과를 바로 개발팀에 전달할 수 있는 형태로 활용할 수 있다.

**수용 기준 (AC)**:
- [ ] 화면/기능 단위로 작업 티켓(MD 파일) 자동 생성
- [ ] 티켓에 제목, 설명, 영향 범위, 난이도 점수, 작업 분류(FE/BE) 포함
- [ ] 티켓 간 의존 관계(dependency) 표시
- [ ] 생성된 티켓 목록 요약 제공
- [ ] 티켓 내용 수정/보완 가능

### US-008: 영향도 시각화 웹 확인
> **As a** 컬리 PM,
> **I want to** 분석 결과를 로컬 웹 브라우저에서 시각적 플로우차트로 확인할 수 있다.
> **So that** 분석 결과를 직관적으로 이해하고 팀에 공유할 수 있다.

**수용 기준 (AC)**:
- [ ] 로컬 웹 서버 자동 실행 (localhost)
- [ ] 요구사항 > 시스템 > 화면 > 하위기능 (+ 모듈) 계층 구조 플로우차트 표시 [추가 검토 반영: 요구사항 레벨을 최상위로 추가하여 하나의 기획서에서 도출된 여러 요구사항이 각각 어떤 시스템에 영향을 주는지 한눈에 파악 가능]
- [ ] 각 노드에 난이도 점수 및 영향도 등급 컬러 표시
- [ ] 노드 클릭 시 상세 정보(작업 내용, 관련 코드, 기획 확인 사항) 표시
- [ ] 백엔드/프론트엔드 구분 표시
- [ ] 종합 결과 대시보드 (총 점수, 등급, 작업 수, FE/BE 비율)
- [ ] 결과 스크린샷 또는 리포트 내보내기

### US-009: LLM 프로바이더 관리
> **As a** 컬리 PM,
> **I want to** 여러 LLM 프로바이더 중 선택하여 사용하고, 필요 시 변경할 수 있다.
> **So that** 비용과 성능에 맞게 유연하게 AI 엔진을 조절할 수 있다.

**수용 기준 (AC)**:
- [ ] 3~4개 LLM 프로바이더 지원 (예: OpenAI, Anthropic, Google, 등)
- [ ] 사용자별 API 키 등록/변경 가능
- [ ] 프로바이더 변경 시 기존 분석 결과에 영향 없음
- [ ] 각 프로바이더별 대략적인 비용/성능 안내
- [ ] API 키 유효성 검증 기능

### US-010: 정책 사전 연동 및 정책 위반 경고 [추가 검토 반영]
> **As a** 컬리 PM,
> **I want to** 코드 내 주석이나 문서에서 추출된 기획 정책이 인덱싱되어, 영향도 분석 시 관련 정책 위반 가능성을 경고받을 수 있다.
> **So that** 기존 정책을 모르고 수정하는 실수를 방지하고, 정책 확인이 필요한 사항을 사전에 파악할 수 있다.

**수용 기준 (AC)**:
- [ ] 코드 인덱싱 시 `// 정책:`, `// Policy:`, `@policy` 등 정책 관련 주석을 자동 추출하여 정책 사전에 저장
- [ ] 정책 사전에 정책명, 설명, 관련 파일 경로, 관련 시스템/모듈 정보 포함
- [ ] 영향도 분석 시 변경 대상 코드와 정책 사전을 매칭하여 관련 정책 경고 표시
- [ ] 시각화 웹에서 정책 경고를 별도 섹션으로 확인 가능
- [ ] 정책 사전 목록 조회 명령 제공 (`/impact policies`)

> **MVP 범위**: 코드 내 주석 기반 정책 추출만 포함. Confluence 등 외부 문서 연동은 Phase 2.

### US-011: 시스템별 담당자/팀 매핑 및 확인 요청 안내 [추가 검토 반영]
> **As a** 컬리 PM,
> **I want to** 각 시스템(또는 서비스)별 담당자/팀 정보를 등록해 두고, 영향도 분석 결과에 따라 "어떤 팀/담당자에게 무엇을 확인 요청해야 하는지" 안내를 받을 수 있다.
> **So that** 영향도 분석 후 어떤 팀에 연락해야 하는지 빠르게 파악하고, 확인 요청 누락을 방지할 수 있다.

**수용 기준 (AC)**:
- [ ] 시스템/서비스별 담당자(이름, 팀, 연락처) 정보를 설정 파일에 등록 가능
- [ ] 담당자 정보 등록/수정/삭제 명령 제공 (`/impact owners`)
- [ ] 영향도 분석 결과에 영향 받는 시스템별 담당자 정보 자동 매핑
- [ ] "확인 요청 안내" 섹션에 담당자별 확인 필요 사항 정리
- [ ] 시각화 웹에서 "확인 요청 안내" 탭 또는 섹션으로 표시

### US-012: 보강 주석(Enriched Annotation) 시스템 [보강 주석 시스템 반영]
> **As a** 컬리 PM,
> **I want to** 코드 인덱싱 시 LLM이 분석한 내용이 별도의 보강 주석 파일에 자동으로 기록되어, 원본 소스에 주석이 없어도 분석된 비즈니스 로직과 정책 정보를 참조할 수 있다.
> **So that** 주석이 없는 레거시 시스템도 일관된 수준의 영향도 분석이 가능하고, 재분석 시 이전 분석 결과를 참조하여 더 빠르고 정확한 분석을 받을 수 있다.

**수용 기준 (AC)**:
- [ ] 코드 인덱싱 시 LLM이 각 함수/모듈의 비즈니스 로직을 분석하여 보강 주석을 자동 생성
- [ ] 기존 주석이 있으면 인덱스에 포함하고, 없으면 LLM이 "추론된 주석"을 생성하여 보강 주석 파일에 추가
- [ ] 보강 주석은 `.impact/annotations/` 폴더에 별도 저장 (원본 소스 절대 수정 불가)
- [ ] 원본 파일의 해시값(sourceHash)으로 변경 감지, 변경된 파일만 재분석
- [ ] 사용자가 수동 수정한 보강 주석은 `userModified: true` 플래그로 보호되어 재분석 시 덮어쓰이지 않음
- [ ] 각 보강 주석 항목에 confidence 점수(0.0~1.0)와 추론 근거(inferred_from)가 포함
- [ ] 영향도 분석 시 인덱스 + 보강 주석을 함께 참조하여 분석 품질 향상
- [ ] 보강 주석이 있는 시스템은 Layer 3 신뢰도 점수가 자동 상승
- [ ] `/impact annotations generate` 명령으로 수동 생성 가능
- [ ] `/impact annotations view [파일경로]` 명령으로 보강 주석 조회 가능

> **MVP 범위**: 보강 주석 자동 생성, 변경 감지 기반 증분 업데이트, 영향도 분석 시 참조 기능 포함. 웹 UI에서의 편집 기능과 정책서 역생성은 Phase 2.

### US-013: 코드 기반 정책서 역생성 [보강 주석 시스템 반영] [정책서 상세화 반영]
> **As a** 컬리 PM,
> **I want to** 보강 주석에 축적된 비즈니스 로직/정책 정보를 바탕으로 "이 시스템에는 이런 정책들이 있다"는 **상세 정책서**가 자동으로 생성된다. 정책서에는 조건/로직/결과 흐름, 입출력 변수, 상수값, 제약사항, 정책 간 의존 관계, 기획자 확인 체크리스트가 포함된다.
> **So that** 개발팀 없이도 시스템의 비즈니스 규칙을 **코드 수준의 상세도**로 파악하고, 기획 시 기존 정책과의 충돌을 사전에 확인하며, 정책 변경 시 영향 범위를 예측할 수 있다.

**수용 기준 (AC)**:
- [ ] 보강 주석의 policies 필드에서 정책 항목을 수집하여 시스템별 정책서 자동 생성
- [ ] 정책서는 Markdown 형식으로 `.impact/annotations/_policies/` 폴더에 저장
- [ ] 시스템별 정책서와 전체 통합 정책서 생성
- [ ] 각 정책 항목에 신뢰도 점수, 관련 코드 경로, 추론 근거 표시
- [ ] **[정책서 상세화 반영]** 각 정책에 **조건 -> 로직 -> 결과** 흐름을 자연어/의사코드로 표현 (IF/ELSE IF/ELSE 분기, 예외 처리 포함)
- [ ] **[정책서 상세화 반영]** 각 정책의 **입출력 변수 및 사용 상수값** 목록 포함 (변수명, 타입, 설명, 상수값의 출처(하드코딩/설정파일/DB 등))
- [ ] **[정책서 상세화 반영]** 각 정책의 **현재 제약사항 및 한계** 표시 (하드코딩된 값, 미지원 케이스, 예외 처리 부재, 기술적 부채 등)
- [ ] **[정책서 상세화 반영]** 각 정책의 **데이터 출처** 표시 (DB 테이블/컬럼, API 호출, 사용자 입력, 세션/상태, 설정값)
- [ ] **[정책서 상세화 반영]** **정책 간 의존 관계** 분석 및 표시 (선행/후행/충돌 가능/관련 정책)
- [ ] **[정책서 상세화 반영]** 각 정책의 **변경 영향 범위** 표시 (호출하는 함수/화면 목록, 관련 API, 영향 받는 화면, 영향 요약)
- [ ] **[정책서 상세화 반영]** **기획자 확인 체크리스트** 자동 생성 (하드코딩 값 확인, 미지원 케이스 검토, 충돌 정책 확인 등)
- [ ] `/impact policies generate` 명령으로 정책서 생성
- [ ] `/impact policies view --generated` 명령으로 생성된 정책서 조회
- [ ] 생성된 정책서를 PM이 검토/수정 가능

> **MVP 범위**: 정책서 문서 생성(MD 파일)은 Phase 2로 유지. 단, **InferredPolicy 확장 스키마(조건/변수/제약사항 등)와 상세 분석 프롬프트**는 **MVP의 보강 주석 생성에 포함**하여 상세 데이터를 조기 축적. [정책서 상세화 반영]
> - MVP 포함: InferredPolicy 확장 스키마, 1단계 상세 분석 프롬프트, CLI 보강 주석 조회 시 상세 정보 표시
> - Phase 2: 정책 간 관계 분석, 변경 영향 범위 분석, 정책서 MD 파일 생성, 통합 정책서, 웹 UI 정책서 뷰
