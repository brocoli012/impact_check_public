## 5. 영향도 점수 체계 (Scoring System)

### 5.1 점수 기준 정의

영향도 점수는 **4가지 차원**을 종합하여 산출합니다:

| 차원 | 설명 | 가중치 | 점수 범위 |
|------|------|--------|----------|
| **개발 복잡도** (Development Complexity) | 코드 변경의 기술적 난이도 | 35% | 1~10 |
| **영향 범위** (Impact Scope) | 변경이 미치는 화면/기능의 수와 범위 | 30% | 1~10 |
| **정책/기획 변경** (Policy Change) | 비즈니스 규칙이나 정책 변경 수반 여부 | 20% | 1~10 |
| **의존성 위험도** (Dependency Risk) | 다른 기능/모듈과의 의존 관계 복잡도 | 15% | 1~10 |

### 5.2 차원별 세부 기준

#### 개발 복잡도 (35%)
| 점수 | 기준 |
|------|------|
| 1~2 | 텍스트/라벨 수정, CSS 변경, 설정값 변경 |
| 3~4 | 기존 컴포넌트 속성 추가, 간단한 로직 수정 |
| 5~6 | 새로운 컴포넌트 개발, API 엔드포인트 추가, DB 스키마 변경 |
| 7~8 | 복잡한 비즈니스 로직 구현, 여러 서비스 간 연동, 트랜잭션 처리 |
| 9~10 | 아키텍처 변경, 대규모 리팩토링, 실시간 처리/동시성 이슈 |

#### 영향 범위 (30%)
| 점수 | 기준 |
|------|------|
| 1~2 | 단일 화면의 단일 기능에만 영향 |
| 3~4 | 단일 화면의 여러 기능에 영향 |
| 5~6 | 2~3개 화면에 걸쳐 영향 |
| 7~8 | 4개 이상 화면에 영향, 공통 모듈 변경 수반 |
| 9~10 | 시스템 전반에 영향, 공통 인프라/라이브러리 변경 필요 |

#### 정책/기획 변경 (20%)
| 점수 | 기준 |
|------|------|
| 1~2 | 정책 변경 없음, 기존 정책 내 구현 가능 |
| 3~4 | 경미한 정책 확인 필요 (예: 노출 순서 변경) |
| 5~6 | 새로운 비즈니스 규칙 추가, 기획 확인 필요 사항 존재 |
| 7~8 | 기존 정책 수정 필요, 이해관계자 합의 필요 |
| 9~10 | 핵심 비즈니스 정책 변경, 법/규정 검토 필요 |

#### 의존성 위험도 (15%)
| 점수 | 기준 |
|------|------|
| 1~2 | 독립적 작업, 다른 기능에 영향 없음 |
| 3~4 | 1~2개 모듈과 약한 의존 관계 |
| 5~6 | 여러 모듈과 의존 관계, 순서 조율 필요 |
| 7~8 | 외부 서비스/API와 연동 필요, 배포 순서 중요 |
| 9~10 | 다수 팀 간 동시 작업 필요, 데이터 마이그레이션 수반 |

### 5.3 화면/기능 단위 점수화 규칙

```
화면 점수 = 해당 화면 내 모든 기능 점수의 단순 합산 (작업 총량 반영)
기능 점수 = (개발복잡도 x 0.35) + (영향범위 x 0.30) + (정책변경 x 0.20) + (의존성위험도 x 0.15)
```

> [피드백 R1 반영] "가중 합산"을 "단순 합산"으로 명확화. 5.4절의 종합 점수 계산(합산)과 일관성 유지. 화면 내 기능 간 별도 가중치를 부여하지 않으며, 기능 점수 자체에 이미 4차원 가중치가 반영되어 있으므로 화면 레벨에서는 단순 합산이 적절.

**각 기능에 대해 FE/BE를 분리하여 개별 점수 산출**:
- FE 점수: 프론트엔드 작업 관점의 4차원 점수
- BE 점수: 백엔드 작업 관점의 4차원 점수
- 기능 점수 = max(FE 점수, BE 점수) 또는 (FE 점수 + BE 점수) / 2 (설정 가능)

### 5.4 종합 점수 및 등급 분류

**종합 점수** = 모든 영향 기능 점수의 합산 (가중 평균 아닌 합산 - 작업 총량 반영)

| 등급 | 종합 점수 범위 | 의미 | 권장 액션 |
|------|--------------|------|----------|
| **Low** | 1~15점 | 소규모 작업 | 일반 스프린트 내 처리 가능 |
| **Medium** | 16~40점 | 중간 규모 작업 | 스프린트 계획 시 우선순위 조정 필요 |
| **High** | 41~70점 | 대규모 작업 | 별도 프로젝트 계획 필요, 리소스 배분 검토 |
| **Critical** | 71점 이상 | 초대규모 작업 | 아키텍처 리뷰 필수, 단계별 릴리즈 계획 필요, 경영진 보고 권고 |

> 등급 경계값은 팀 특성에 맞게 조정 가능하도록 설정 기능 제공

### 5.5 신뢰도 점수와 영향도 점수의 통합 [분석 일관성 전략 반영]

영향도 점수(Scoring)와 분석 신뢰도(Confidence Score)는 별도 체계로 산출되지만, PM에게는 통합된 형태로 제시합니다.

**통합 표시 원칙**:
- 영향도 점수(1~10, 4차원)는 "작업의 규모와 난이도"를 나타냄
- 신뢰도 점수(0~100%)는 "이 영향도 점수를 얼마나 신뢰할 수 있는지"를 나타냄
- 두 점수를 함께 표시하여 PM이 정보에 입각한 판단 가능

**표시 예시**:
```
[상품 상세 화면] 영향도: 18점 (High) | 신뢰도: 92% (HC)
  → 이 분석은 높은 신뢰도로 수행되었습니다.

[레거시 정산] 영향도: 15점 (Medium) | 신뢰도: 42% (LC)
  → 이 분석은 참고용입니다. 정책 정보 부족으로 실제 영향은 다를 수 있습니다.
  → 보완: /impact policies add 명령으로 정책을 수동 등록하세요.
```

**신뢰도에 따른 점수 해석 가이드**:
| 신뢰도 등급 | 영향도 점수 해석 | PM 안내 |
|------------|-----------------|---------|
| High (85%+) | 높은 정확도, 의사결정에 바로 활용 가능 | "이 점수를 기반으로 일정을 계획하세요" |
| Medium (65~84%) | 대체로 정확하나 일부 항목 추가 확인 권장 | "대부분 신뢰할 수 있으나 정책 관련 항목은 확인하세요" |
| Low (40~64%) | 참고용, 담당 개발자 확인 필수 | "이 점수는 참고용입니다. 담당자와 확인 후 활용하세요" |
| Very Low (0~39%) | 매우 제한적, 직접 확인 필요 | "분석 정보가 부족합니다. 담당 개발자와 직접 논의하세요" |

### 5.6 점수 산출 투명성

각 점수에 대해 반드시 **산출 근거**를 함께 표시합니다:
```
예시)
[상품 상세 화면 - 리뷰 섹션 개선]
- 개발 복잡도: 6/10 (신규 컴포넌트 개발 + API 추가)
  → 근거: ReviewFilterComponent 신규 개발, /api/reviews/filter 엔드포인트 추가
- 영향 범위: 4/10 (상품 상세 + 리뷰 목록 2개 화면)
  → 근거: ProductDetail.vue, ReviewList.vue 변경 필요
- 정책 변경: 3/10 (리뷰 필터 기준 기획 확인 필요)
  → 근거: 리뷰 정렬 기준, 필터 옵션 목록 기획 확정 필요
- 의존성 위험도: 2/10 (독립적 작업)
  → 근거: 기존 리뷰 API와 하위호환, 별도 배포 가능
- 기능 점수: (6x0.35)+(4x0.30)+(3x0.20)+(2x0.15) = 4.2점
```
