## 3. 기능 요구사항 (Functional Requirements)

### 3.1 스킬 시스템 (Claude Code Desktop / Antigravity 스킬)

이 프로덕트는 독립 애플리케이션이 아닌 **Claude Code Desktop 또는 Antigravity에 설치되는 "스킬" 형태**로 배포됩니다.

| 항목 | 설명 |
|------|------|
| **배포 형태** | Claude Code Desktop / Antigravity 스킬 |
| **호출 방식** | 슬래시 커맨드 (예: `/impact`, `/impact-check`) 또는 자연어 호출 |
| **설치** | 스킬 레포지토리에서 설치 또는 로컬 폴더 지정 |
| **설정 파일** | 스킬 설정 파일(SKILL.md 등)에 기능 정의 |
| **실행 환경** | 사용자 로컬 PC (스킬 실행 시 로컬 리소스 활용) |

**주요 스킬 명령어 (예시)**:
- `/impact init` - 프로젝트 등록 및 최초 인덱싱
- `/impact analyze` - 기획서 입력 및 영향도 분석 실행
- `/impact view` - 시각화 웹 실행 및 브라우저 열기
- `/impact tickets` - 생성된 티켓 확인/관리
- `/impact config` - LLM 프로바이더 및 API 키 설정
- `/impact reindex` - 인덱스 수동 갱신
- `/impact policies` - 정책 사전 조회 [추가 검토 반영]
- `/impact owners` - 시스템별 담당자/팀 관리 [추가 검토 반영]
- `/impact annotations generate` - 보강 주석 수동 생성 [보강 주석 시스템 반영]
- `/impact annotations view` - 보강 주석 조회 [보강 주석 시스템 반영]
- `/impact policies generate` - 코드 기반 정책서 역생성 (Phase 2) [보강 주석 시스템 반영]

### 3.2 프로젝트 등록 & 코드 인덱싱

**등록 프로세스**:
1. 사용자가 분석할 Git 레포의 로컬 경로를 지정
2. 프로젝트 메타데이터(이름, 기술 스택 등) 자동 감지
3. 전체 코드 스캔 및 인덱스 생성

**인덱스 포함 정보**:
- 디렉토리/파일 구조
- 화면(Screen/Page) 목록 및 라우팅 정보
- 컴포넌트/모듈 의존 관계
- API 엔드포인트 목록 및 호출 관계
- 데이터 모델/스키마 정보
- 설정 파일 및 환경 변수
- 정책 사전 (코드 주석 기반 정책 정보) [추가 검토 반영]
- 보강 주석 (LLM이 코드를 분석하여 추론한 비즈니스 로직/정책 정보, `.impact/annotations/`에 별도 저장) [보강 주석 시스템 반영]

**보강 주석(Enriched Annotation) 생성** [보강 주석 시스템 반영]:
- 인덱싱 시 LLM이 각 함수/모듈의 비즈니스 로직을 분석하여 보강 주석을 자동 생성
- 기존 주석이 있으면 → 보강 주석에 원본 주석을 포함하고 LLM이 의미를 확장
- 기존 주석이 없으면 → LLM이 코드를 분석하여 "추론된 주석"을 생성
- 예: `calculateShippingFee()` 함수에 주석이 없어도 → "배송비 계산 함수. 멤버스 여부에 따라 무료배송 판단 로직 포함" 같은 보강 주석 생성
- 보강 주석은 `.impact/annotations/` 폴더에 별도 저장 (원본 소스 절대 수정 불가)
- 각 항목에 confidence 점수(0.0~1.0)와 추론 근거를 함께 기록
- sourceHash 기반 변경 감지로 변경된 파일만 재분석
- 재분석 시 기존 보강 주석을 참조하여 일관성 유지

> [피드백 R1 반영] **MVP 지원 기술 스택**: React (CRA/Vite), Next.js, Vue 3, Spring Boot, Express/NestJS. 프레임워크별로 화면/라우트 추출 방식이 상이하므로 (예: React는 react-router Route 분석, Vue는 vue-router routes 배열, Spring은 @Controller 매핑) 각각 전용 파서가 필요합니다. 지원하지 않는 프레임워크의 경우 파일 구조 기반 폴백 분석을 제공합니다.

**변경 감지 및 자동 업데이트**:
- Git diff 기반 변경 파일 감지
- 변경 파일만 부분 재인덱싱
- 인덱스 버전 히스토리 관리

### 3.3 기획서 입력 & 파싱

**지원 입력 형태**:

| 입력 형태 | 처리 방식 |
|----------|----------|
| 텍스트 직접 입력 | 채팅 인터페이스에서 자연어로 기획 의도 전달 |
| PDF 파일 | 파일 첨부 -> 텍스트/이미지 추출 및 파싱 |
| Confluence URL | 페이지 접근 -> 내용 크롤링/파싱 |
| 이미지 | 첨부 -> 멀티모달 LLM으로 내용 분석 |

**파싱 결과 구조화**:
- 기능 요구사항 항목 추출
- 화면/기능 단위 분류
- 비즈니스 규칙/정책 사항 식별
- 변경 범위 키워드 추출

**파싱 확인 프로세스**:
- 파싱 결과를 사용자에게 요약 표시
- 사용자가 검토/수정 후 분석 진행 승인

### 3.4 영향도 분석 엔진

**분석 프로세스 - 다층 분석 전략 (Multi-Layer Analysis)** [분석 일관성 전략 반영]:

시스템별 코드 품질(주석, 구조, 프레임워크) 편차로 인한 분석 결과의 불일관성을 해소하기 위해, 주석에만 의존하지 않는 4개 Layer 기반 분석 파이프라인을 적용합니다.

1. **Layer 1: 코드 구조 분석** (주석 불필요 - 항상 수행)
   - AST 파싱, 파일 구조 스캔, 프레임워크별 라우팅 추출
   - 화면(Screen), 컴포넌트, API 엔드포인트 자동 식별
   - 비표준 프레임워크의 경우 파일 구조 기반 폴백 분석 적용

2. **Layer 2: 의존성 그래프 분석** (주석 불필요 - 항상 수행)
   - import/export 관계 추적으로 영향 전파 범위 식별
   - API 호출 관계 (FE→BE) 자동 매핑
   - 공유 모듈 식별 및 위험도 평가

3. **Layer 3: 주석/문서 기반 정책 분석** (주석 필요 - 있으면 보강)
   - 정책 주석, README, 설정 파일에서 비즈니스 정책 추출
   - 수동 입력 정책 (policies.yaml, `/impact policies add`) 포함
   - 주석 없는 시스템에서는 "정책 정보 없음" 안내 및 보완 가이드 제공

4. **Layer 4: LLM 추론 분석** (주석 없어도 부분적 가능)
   - LLM의 코드 이해력으로 비즈니스 로직 추론
   - 기획서와 코드 컨텍스트 매칭으로 영향 판단
   - 추론 근거를 함께 제시하여 투명성 확보

5. **기획서 파싱 결과** + **프로젝트 인덱스**를 LLM에 입력
6. 기획서의 각 기능 요구사항에 대해:
   - 영향 받는 화면(Screen) 식별
   - 화면 내 영향 기능(Feature) 매핑
   - 기능 내 하위기능(Sub-feature) 분석
   - 관련 모듈(Module)/파일 추적
7. 각 영향 항목에 대해:
   - 작업 유형 분류 (신규 개발 / 기존 수정 / 설정 변경)
   - FE / BE 작업 구분
   - 난이도 점수 산출
   - 필요 기획/정책 확인 사항 도출
   - **신뢰도 점수 산출** (4개 Layer의 정보 제공 정도 기반) [분석 일관성 전략 반영]

**분석 결과 포함 항목**:
- 영향 화면 목록 및 각 화면의 영향도
- 화면별 필요 작업 항목 (FE/BE 구분)
- 기획 확인 필요 사항 목록
- 정책 변경 필요 여부 및 내용
- 의존 관계 (이 작업이 완료되어야 다음 작업 가능)
- 정책 사전 매칭 결과 및 정책 위반 경고 [추가 검토 반영]
- 시스템별 담당자/팀 매핑 및 확인 요청 안내 [추가 검토 반영]
- **시스템별 분석 신뢰도 점수 (Confidence Score)** [분석 일관성 전략 반영]
- **정보 부족 시스템에 대한 보완 가이드** [분석 일관성 전략 반영]

### 3.5 티켓 생성

**티켓 형식**: Markdown 파일 (MVP)

**티켓 포함 정보**:
```
- 티켓 ID (자동 생성)
- 제목
- 작업 유형 (FE / BE / 기획 / 정책 확인)
- 관련 화면 및 기능
- 작업 설명
- 영향 받는 파일/모듈 목록
- 난이도 점수
- 선행 작업 (의존 관계)
- 기획 확인 사항
- 예상 작업 규모 (SP 또는 시간)
```

**티켓 관리**:
- 생성된 티켓 목록 조회
- 티켓 내용 수정/삭제
- 티켓 간 의존 관계 시각화
- 향후 Jira 연동을 위한 구조 설계

### 3.6 영향도 시각화 웹 (로컬)

**구성 요소**:

1. **종합 대시보드**
   - 총 영향도 점수 및 등급 (상단 크게 표시)
   - 영향 화면 수, 총 작업 수, FE/BE 비율
   - 점수 분포 차트

2. **플로우차트 뷰** (메인 시각화) [추가 검토 반영: 요구사항 레벨 추가]
   - **요구사항 레벨**: 기획서에서 도출된 각 요구사항 노드 (최상위) [추가 검토 반영]
   - **시스템 레벨**: 요구사항별 영향 받는 시스템 노드
   - **화면 레벨**: 시스템 내 영향 받는 화면 노드들
   - **하위기능 레벨**: 각 화면 내 영향 기능 노드들
   - **모듈 레벨**: 실제 변경 필요한 코드 모듈 노드들
   - 계층 간 화살표로 연결 관계 표시
   - 노드 색상으로 난이도/영향도 등급 시각 구분
   - 하나의 시스템이 여러 요구사항에서 공유될 수 있으며 시각적으로 표현 [추가 검토 반영]

3. **노드 상세 패널**
   - 노드 클릭 시 우측 또는 하단에 상세 정보 표시
   - 작업 내용, 관련 코드 경로, 기획 확인 사항, 난이도 산출 근거

4. **필터 및 뷰 옵션**
   - FE/BE 필터링
   - 난이도 등급별 필터링
   - 작업 유형별 필터링
   - 확대/축소, 전체 보기

5. **리포트 내보내기**
   - 결과 요약 텍스트 복사
   - 스크린샷 저장 (PNG)
   - 상세 리포트 MD 파일 내보내기

### 3.7 LLM 멀티 프로바이더

**지원 프로바이더** (MVP 기준 3개):

| 프로바이더 | 용도 | 특징 |
|-----------|------|------|
| Anthropic (Claude) | 코드 분석, 영향도 추론 | 높은 코드 이해력 |
| OpenAI (GPT) | 기획서 파싱, 요약 | 범용적 텍스트 처리 |
| Google (Gemini) | 멀티모달(이미지 분석) | 이미지 기획서 처리에 강점 |

> [피드백 R5 반영] 로컬 LLM(Ollama 등)은 MVP에서 제외하고 Phase 2로 이동. Tech-Design과 일관성 유지. MVP에서는 클라우드 LLM 3개로 충분히 검증 후 확장.

**관리 기능**:
- 사용자별 API 키 등록/수정/삭제
- 프로바이더별 모델 선택
- 기본 프로바이더 설정
- 용도별 프로바이더 자동 라우팅 (예: 이미지는 Gemini, 코드는 Claude)
- 변경 시 기존 결과에 영향 없음 보장

### 3.8 사용자 관리

MVP에서는 경량 사용자 관리:
- 사용자 프로필 (이름, 역할) 로컬 저장
- 사용자별 LLM API 키 분리 저장
- 사용자별 프로젝트 등록 목록 관리
- 분석 히스토리 로컬 저장

> 참고: MVP에서는 인증/로그인 시스템 불필요 (로컬 실행이므로)

### 3.9 멀티 프로젝트 관리 [피드백 R13 반영]

PM이 여러 프로젝트를 등록하고 전환하며 사용하는 시나리오를 지원합니다.

**프로젝트 관리 명령어**:
- `/impact projects` : 등록된 프로젝트 목록 표시 (최근 사용 순 정렬)
- `/impact projects --switch <이름>` : 활성 프로젝트 전환
- `/impact projects --remove <이름>` : 프로젝트 등록 해제 (인덱스 삭제 여부 선택)
- `/impact projects --archive <이름>` : 프로젝트 아카이브 (인덱스 유지, 목록에서 숨김)

**프로젝트 전환 UX**:
```
/impact projects

  등록된 프로젝트 (3개):

  #  이름                    경로                          최근 사용      상태
  ── ──────────────────────── ──────────────────────────── ────────────── ──────
  1  kurly-commerce-front *   /Users/pm/kurly-commerce     2026-02-13    활성
  2  kurly-admin              /Users/pm/kurly-admin        2026-02-10    등록됨
  3  kurly-api-server         /Users/pm/kurly-api          2026-02-05    등록됨

  * = 현재 활성 프로젝트

  전환: /impact projects --switch [이름 또는 번호]
```

**프로젝트별 데이터 격리**:
- 각 프로젝트의 인덱스/분석 결과/티켓은 프로젝트별 독립 디렉토리에 저장
- 프로젝트 간 데이터가 섞이지 않도록 엄격히 격리
- 분석 결과에 항상 프로젝트명이 표시되어 혼동 방지

**프로젝트 삭제/아카이브 처리**:
- 삭제 시: "프로젝트 등록을 해제하면 인덱스와 분석 기록이 삭제됩니다. 분석 결과 파일은 유지할까요? [Y/n]" 확인
- 아카이브 시: 목록에서 숨기되 데이터는 보존, `/impact projects --archived` 로 확인 가능
- 프로젝트 간 영향도 비교는 MVP에서 미지원 (Phase 2)

### 3.10 정책 사전 (Policy Dictionary) [추가 검토 반영]

코드 내 주석에서 기획 정책을 추출하여 인덱싱하고, 영향도 분석 시 정책 위반 가능성을 경고합니다.

**정책 추출 대상 (MVP)**:
- 코드 주석: `// 정책:`, `// Policy:`, `/* 정책: */`, `@policy` 어노테이션
- README.md 내 "정책" 또는 "Policy" 섹션
- 설정 파일 내 주석
- **수동 입력 정책 (policies.yaml)**: PM이 직접 작성하는 정책 사전 파일 [분석 일관성 전략 반영]
- **CLI 명령어 입력**: `/impact policies add` 명령으로 개별 정책 등록 [분석 일관성 전략 반영]

**정책 사전 구조**:
```
- 정책 ID (자동 생성)
- 정책명
- 정책 설명 (주석 내용 또는 수동 입력 내용)
- 관련 파일 경로
- 관련 시스템/모듈
- 정책 카테고리 (배송, 결제, 프로모션, 회원 등)
- 등록 방식 (comment: 코드 주석 / manual: 수동 입력) [분석 일관성 전략 반영]
```

**정책 매칭 및 경고**:
- 영향도 분석 시 변경 대상 코드 근처의 정책 주석을 자동 감지
- 변경 코드와 관련된 정책이 있으면 경고 메시지 생성
- 경고 예시: "이 코드 변경은 '컬리 멤버스 무료배송' 정책에 영향을 줄 수 있습니다"

**정보 부족 시스템 보완 전략** [분석 일관성 전략 반영]:

주석이 없거나 부족한 시스템에서도 정책 분석이 가능하도록 보완 경로를 제공합니다.

1. **수동 정책 입력 (CLI)**: PM이 알고 있는 정책을 채팅 형태로 등록
   ```
   /impact policies add "배송비 정책: 3만원 이상 무료배송"
   /impact policies add --system "물류" --category "배송" "새벽배송은 서울/경기만 가능"
   ```

2. **정책 사전 파일 (policies.yaml)**: 대량 정책을 YAML 파일로 일괄 등록
   - 파일 위치: `.impact/projects/<project-id>/policies.yaml`
   - 인덱싱 시 자동 로드하여 정책 사전에 병합
   - 코드 주석 정책과 수동 입력 정책을 구분하여 관리

3. **보완 가이드 자동 생성**: 인덱싱 완료 후 정책 정보가 부족한 시스템에 대해 자동 안내
   - 주석 추가 권장 파일 목록 제시
   - 수동 정책 등록 명령어 안내
   - 보완 후 `/impact reindex`로 프로파일 업데이트

**명령어**:
- `/impact policies` : 현재 프로젝트의 정책 사전 전체 조회
- `/impact policies --search <키워드>` : 키워드로 정책 검색
- `/impact policies add <정책 내용>` : 수동 정책 등록 [분석 일관성 전략 반영]
- `/impact policies add --system <시스템명> --category <카테고리> <정책 내용>` : 상세 정책 등록 [분석 일관성 전략 반영]
- `/impact policies generate` : 보강 주석 기반 코드 기반 정책서 역생성 (Phase 2) [보강 주석 시스템 반영]
- `/impact policies view --generated` : 생성된 정책서 조회 (Phase 2) [보강 주석 시스템 반영]

**코드 기반 정책서 역생성** [보강 주석 시스템 반영] [정책서 상세화 반영]:

보강 주석(`.impact/annotations/`)에 축적된 비즈니스 로직/정책 정보를 바탕으로, 시스템별 **상세 정책서**를 자동 생성하는 기능입니다.

- 보강 주석의 `policies` 필드에서 정책 항목을 수집
- 시스템별로 그룹핑하여 Markdown 정책서 자동 생성
- 정책서에 각 정책의 신뢰도, 관련 코드 경로, 추론 근거 포함
- PM이 개발팀 없이도 시스템의 비즈니스 규칙을 파악 가능
- 생성된 정책서를 PM이 검토/수정 가능

**[정책서 상세화 반영] 상세 정책서 필수 포함 항목**:

기획자가 정책을 실질적으로 활용하기 위해, 정책서에는 단순한 이름/설명을 넘어 다음 항목이 반드시 포함되어야 합니다:

1. **조건 -> 로직 -> 결과 흐름**: 모든 IF/ELSE 분기를 자연어/의사코드로 표현
   - 예: `IF 사용자.멤버스등급 == "멤버스" THEN 배송비 = 0원`
   - 예외 처리(try/catch) 및 기본값(default) 포함

2. **입출력 변수 및 사용 상수값**:
   - 입력 파라미터 (이름, 타입, 역할)
   - 출력 값 (이름, 타입, 역할)
   - 상수값 (이름, 현재값, 출처 - 하드코딩/설정파일/환경변수/DB/API)

3. **현재 제약사항 및 한계**:
   - 하드코딩된 비즈니스 값 (변경 시 코드 수정 필요)
   - 미지원 케이스 (코드에 분기가 없는 경우)
   - 예외 처리 부재, 기술적 부채 (TODO/FIXME 주석)

4. **데이터 출처**:
   - 각 입력 데이터의 궁극적 출처 (DB 테이블/컬럼, API, 사용자 입력 등)

5. **정책 간 의존 관계**:
   - 선행 정책 (이 정책 실행 전에 필요한 것)
   - 후행 정책 (이 정책 결과를 참조하는 것)
   - 충돌 가능 정책 (동시 적용 시 문제될 수 있는 것)

6. **변경 영향 범위**:
   - 이 정책을 사용하는 함수/화면 목록
   - 관련 API 엔드포인트
   - 변경 시 영향 요약

7. **기획자 확인 체크리스트**:
   - 하드코딩 값 현행화 확인
   - 미지원 케이스 향후 필요성 검토
   - 충돌 정책 규칙 정의 여부 확인
   - 변경 영향 범위 수용 가능성 확인

> **MVP 범위**: 정책서 MD 파일 생성은 Phase 2로 유지. 단, **InferredPolicy 확장 스키마(조건/변수/상수/제약사항 필드)와 상세 분석 프롬프트**는 MVP 보강 주석 생성에 포함하여 데이터를 조기 축적. [정책서 상세화 반영]

> Phase 2에서 Confluence 정책 문서 연동을 추가하여 코드 외부 정책까지 포괄

### 3.11 시스템별 담당자/팀 매핑 [추가 검토 반영]

각 시스템(또는 서비스)별 담당자/팀 정보를 등록하고, 영향도 분석 결과에 따라 확인 요청 안내를 제공합니다.

**담당자 정보 등록 방식 (MVP)**:
- 설정 파일 기반: `.impact/owners.json`에 시스템별 담당자 정보 등록
- CLI 명령어로 등록/수정/삭제 가능

**담당자 정보 구조**:
```
- 시스템/서비스명
- 담당 팀명
- 담당자명
- 연락처 (이메일, Slack 채널 등)
- 담당 범위 설명 (선택)
```

**확인 요청 안내 생성**:
- 영향도 분석 결과에서 영향 받는 시스템 식별
- 해당 시스템의 등록된 담당자 정보를 자동 매핑
- 담당자별 확인 필요 사항을 정리하여 "확인 요청 안내" 섹션 생성
- 예시:
  ```
  [확인 요청 안내]
  - 프로모션 시스템 (프로모션 팀 - 김OO)
    → "쿠폰 자동 적용 시 기존 프로모션 중복 적용 정책 확인 필요"
  - 물류 시스템 (물류개발팀 - 이OO)
    → "배송일 계산 로직 변경 시 기존 SLA 영향 검토 필요"
  ```

**명령어**:
- `/impact owners` : 등록된 담당자 목록 표시
- `/impact owners --add` : 새 담당자 등록
- `/impact owners --edit <시스템명>` : 담당자 정보 수정
- `/impact owners --remove <시스템명>` : 담당자 정보 삭제
