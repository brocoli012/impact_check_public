## 4. 비기능 요구사항

### 성능
| 항목 | 목표 |
|------|------|
| 코드 인덱싱 (최초) | 중규모 프로젝트(10만 LOC) 기준 5분 이내 |
| 인덱스 부분 업데이트 | 변경 파일 수 기준 30초 이내 |
| 영향도 분석 | 10페이지 기획서 기준 3분 이내 |
| 시각화 웹 로딩 | 2초 이내 (초기 번들 100KB 이하, gzipped) [피드백 R14 반영] |
| 시각화 인터랙션 | 노드 100개 이하에서 60fps 부드러운 렌더링, 300개 이하에서 가상화로 대응 [피드백 R14 반영] |
| 대규모 프로젝트 (50만 LOC) | 분할 인덱싱으로 15분 이내 [피드백 R14 반영] |

> [피드백 R14 반영] 벤치마크 기준 환경: MacBook Pro M1, 16GB RAM. LLM 병렬 호출은 동시 3~5개로 rate limit을 고려하여 제한. 상세 전략은 TECH-DESIGN-001 11.1절 참조.

### 보안
- API 키 로컬 암호화 저장 (평문 저장 금지)
- 분석 대상 코드는 로컬에서만 처리 (외부 전송 시 사용자 동의 필요)
- LLM API 호출 시 민감 코드 마스킹 옵션 제공
- 분석 결과 로컬에만 저장

### 확장성
- LLM 프로바이더 추가가 설정만으로 가능한 구조
- 입력 형식 추가 용이 (Notion, Google Docs 등)
- 시각화 뷰 유형 추가 가능 (리스트 뷰, 매트릭스 뷰 등)
- 티켓 시스템 연동 플러그인 구조 (Jira, Linear 등)

### 분석 신뢰도 (Analysis Confidence) [분석 일관성 전략 반영]

시스템별 코드 품질 편차(주석, 구조, 프레임워크)에도 불구하고 일관된 분석 결과를 제공하기 위한 요구사항입니다.

| 항목 | 목표 |
|------|------|
| 다층 분석 | 4개 Layer(구조, 의존성, 정책, LLM 추론)를 통해 주석 없는 시스템에서도 최소한의 영향도 분석 가능 |
| 신뢰도 점수 | 각 시스템별 분석 결과에 0~100% 신뢰도 점수를 함께 표시 |
| 신뢰도 등급 | High(85~100%), Medium(65~84%), Low(40~64%), Very Low(0~39%) 4단계 등급 |
| 프로젝트 프로파일 | 인덱싱 완료 시 프레임워크 인식률, 주석 밀도, 의존 그래프 완성도 등 분석 준비도 프로파일 자동 생성 |
| 정보 부족 경고 | 신뢰도 Low 이하 시스템에 대해 원인과 보완 방법을 명시적으로 안내 |
| 보완 경로 | PM이 수동으로 정책을 등록할 수 있는 CLI 명령어 및 YAML 파일 경로 제공 |
| 투명성 | 분석 결과에 "이 분석은 참고용입니다" vs "높은 신뢰도로 수행되었습니다" 맥락 안내 |

### 사용성
- PM(비개발자)이 5분 이내 첫 분석 완료 가능
- 전문 용어에 대한 툴팁/설명 제공
- 분석 과정 진행률 및 현재 단계 표시
- 오류 발생 시 명확한 안내 메시지와 해결 방안 제시
- 설치 가이드에서 기술 용어를 최소화하고 복사-붙여넣기만으로 설치 가능하도록 설계 [피드백 R12 반영]
- `/impact demo` 명령으로 샘플 데이터 기반 체험 기능 제공 (점수 체계 이해 지원) [피드백 R12 반영]
- `/impact help`, `/impact faq` 명령으로 인라인 도움말 제공, 웹 시각화에서도 `?` 버튼으로 상시 접근 [피드백 R12 반영]

## 10. 리스크 및 고려사항

| # | 리스크 | 영향도 | 발생 가능성 | 대응 방안 |
|---|--------|--------|-----------|----------|
| T-1 | 코드 인덱싱 정확도 부족 | HIGH | MEDIUM | 프로젝트 기술 스택별 파서 최적화, 인덱싱 결과 사용자 검증 단계 추가 |
| T-2 | LLM 영향도 분석 할루시네이션 | HIGH | HIGH | 분석 결과에 반드시 근거(관련 코드 경로) 첨부, 사용자 피드백 루프로 개선 |
| T-3 | 대규모 프로젝트 인덱싱 시간 과다 | MEDIUM | MEDIUM | 증분 인덱싱, 우선순위 기반 부분 인덱싱, 캐시 활용 |
| T-4 | LLM API 비용 예측 어려움 | MEDIUM | HIGH | 분석 전 예상 토큰 수/비용 안내, 분석 범위 제한 옵션 제공 |
| T-5 | 다양한 프로젝트 구조 대응 한계 | MEDIUM | MEDIUM | 주요 프레임워크(React, Vue, Spring 등) 우선 지원, 커스텀 파서 플러그인 구조 |

### 비즈니스 리스크

| # | 리스크 | 영향도 | 발생 가능성 | 대응 방안 |
|---|--------|--------|-----------|----------|

| B-2 | 분석 결과 신뢰도 부족 | HIGH | MEDIUM | 점수 산출 근거 투명 공개, 파일럿 기간 중 정확도 개선 |

## 11. 추가 제안 (PM 관점)

### 제안 1: "영향도 요약 카드" - 한눈에 파악하는 1페이지 리포트

기획자가 가장 먼저 보게 되는 화면을 **신용카드 크기의 요약 카드** 형태로 제공합니다:

```
┌─────────────────────────────────────────────┐
│  영향도 요약         [Medium - 32점]         │
│                                             │
│  영향 화면: 3개    총 작업: 8개              │
│  FE: 5개 (62%)    BE: 3개 (38%)             │
│  기획 확인: 2건    정책 변경: 0건             │
│                                             │
│  Top 3 영향 화면:                            │
│  1. 상품 상세 (18점, High)                   │
│  2. 장바구니 (9점, Low)                      │
│  3. 마이페이지 (5점, Low)                    │
│                                             │
│  권장: 스프린트 계획 시 우선순위 조정 필요     │
└─────────────────────────────────────────────┘
```

이 카드는 텍스트 기반으로 터미널에서도 바로 확인 가능하고, 시각화 웹의 상단에도 동일하게 표시됩니다. PM이 경영진/팀 리더에게 빠르게 공유할 수 있는 형태입니다.

### 제안 2: "기획 확인 사항 체크리스트" 자동 생성

영향도 분석 결과에서 도출된 기획 확인/정책 확인 사항을 **체크리스트 형태**로 자동 생성합니다:

```markdown
## 기획 확인 체크리스트 (리뷰 필터 기능 추가)

### 반드시 확인 (기능 개발 전)
- [ ] 리뷰 필터 옵션 목록 확정 (별점별, 최신순, 사진 리뷰 외 추가?)
- [ ] 필터 적용 시 기본 정렬 기준 확정

### 정책 확인 필요
- [ ] 비구매 리뷰 필터 포함 여부 (기존 정책: 구매자만 리뷰 가능)
- [ ] 리뷰 신고/삭제된 리뷰 필터 시 노출 여부

### 디자인 검토 필요
- [ ] 필터 UI 위치 및 형태 (드롭다운 vs 탭 vs 칩)
- [ ] 모바일 반응형 처리 방식
```

이를 통해 PM은 "개발 요청 전에 미리 확인해야 할 것"을 놓치지 않고 챙길 수 있습니다.

### 제안 3: "이전 유사 작업" 참조 기능

분석 시 이전에 유사한 작업이 있었다면 자동으로 찾아서 참조 정보를 제공합니다:

```
유사 작업 참조:
- [2025-12] "상품 목록 필터 기능 추가" → 당시 영향도 28점(Medium), 실제 소요 2주
- 이번 작업과의 유사도: 72%
- 참고: 당시 예상보다 BE API 작업이 1.5배 소요됨 (캐시 전략 추가 필요했음)
```

실제 과거 데이터를 바탕으로 예측 정확도를 높이고, PM이 일정 산정 시 현실적인 참고치를 활용할 수 있습니다. (MVP 이후 분석 데이터가 쌓이면 자동화 가능)

### 제안 4: "What-if 분석" - 기획 변경 시뮬레이션

PM이 기획 범위를 조정할 때 영향도가 어떻게 변하는지 빠르게 시뮬레이션할 수 있습니다:

```
현재 기획: 리뷰 필터 (별점 + 정렬 + 사진) → 32점 (Medium)

What-if 시뮬레이션:
- "사진 리뷰 필터만 제외하면?" → 예상 24점 (Medium, -8점)
- "모바일 대응 제외하면?" → 예상 22점 (Medium, -10점)
- "별점 필터만 하면?" → 예상 15점 (Low, -17점)
```

이를 통해 PM이 "MVP에서는 별점 필터만 먼저 하고, 사진 필터는 다음 스프린트에" 같은 스코핑 결정을 데이터 기반으로 내릴 수 있습니다.

### 제안 5: 분석 결과의 "용어 번역"

개발 용어를 PM이 이해하기 쉬운 비즈니스 용어로 자동 변환하여 표시합니다:

| 개발 용어 | PM 친화 표현 |
|----------|-------------|
| API 엔드포인트 추가 | 서버에서 새로운 데이터 처리 기능 개발 |
| DB 스키마 변경 | 데이터 저장 구조 변경 (기존 데이터 영향 검토 필요) |
| 컴포넌트 리팩토링 | 기존 화면 요소 구조 재정리 (기능 변경 없음) |
| 캐시 무효화 | 변경사항이 사용자에게 바로 반영되도록 처리 |
| 마이그레이션 필요 | 기존 데이터를 새 구조에 맞게 변환하는 작업 필요 |

모든 분석 결과에서 기술 용어 옆에 PM 친화 설명을 병기하여, 개발 배경 없는 PM도 결과를 정확히 이해할 수 있도록 합니다.

### 제안 6: "영향도 트렌드" 대시보드 (MVP 이후 데이터 축적 시)

분석 히스토리가 쌓이면 다음과 같은 트렌드를 시각화합니다:
- 월별 영향도 점수 추이 (시스템 복잡도 증가 모니터링)
- 자주 영향 받는 화면 Top 5 (리팩토링 우선순위 참고)
- 예상 vs 실제 소요 시간 비교 (점수 체계 보정)
- 팀별/프로젝트별 영향도 분포
