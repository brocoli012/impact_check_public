## 9. 확장 계획

### Phase 2: 외부 연동 및 고급 입력

| 기능 | 설명 | 기대 효과 |
|------|------|----------|
| Jira 티켓 연동 | MD 티켓을 Jira에 자동 생성/동기화 | 작업 관리 도구와 직접 연결 |
| GitHub 자동 동기화 | Git pull 없이 원격 레포 변경 감지 및 인덱스 업데이트 | 항상 최신 상태 유지 |
| Confluence 파싱 | Confluence URL 입력 시 자동 내용 추출 | 기존 기획 문서 활용 편의 |
| 이미지 기획서 분석 | 와이어프레임/스크린샷 분석 | 다양한 기획 자료 지원 |
| 캔버스 뷰 시각화 | Obsidian 캔버스 스타일 카드형 뷰 추가 | 상세 정보 확인 편의 |

> [피드백 R8 반영] **Phase 2 확장을 위한 MVP 아키텍처 확장점**: 각 Phase 2 기능이 현재 아키텍처에서 어떤 모듈을 추가/변경해야 하는지 명확히 합니다:
>
> | Phase 2 기능 | 변경 범위 | 아키텍처 확장점 |
> |-------------|----------|---------------|
> | Jira 연동 | TicketGenerator에 JiraExporter 추가 | TicketGenerator 인터페이스에 `export(target)` 메서드 추가, Jira REST API 클라이언트 모듈 신규 |
> | GitHub 자동 동기화 | IndexingEngine에 RemoteWatcher 추가 | Git 폴링 또는 webhook 수신 모듈 신규, 인덱싱 트리거 확장 |
> | Confluence 파싱 | SpecParser에 ConfluenceAdapter 추가 | InputAdapter 인터페이스 구현체 1개 추가 (MVP의 Input Adapter 패턴 활용) |
> | 이미지 기획서 | SpecParser에 ImageAdapter 추가 | InputAdapter 구현 + 멀티모달 LLM 호출 (Gemini 활용, 기존 LLM Layer 재사용) |
> | 캔버스 뷰 | 프론트엔드에 CanvasView 페이지 추가 | 기존 분석 결과 JSON은 동일, 렌더링 컴포넌트만 추가 (FlowChart.tsx와 병렬 뷰) |
>
> 위 확장점들이 MVP 아키텍처 설계에 이미 반영되어 있어, Phase 2 기능 추가 시 코어 엔진 변경 없이 모듈 추가만으로 대응 가능합니다.

### Phase 3: 팀 협업 및 고급 분석

## 9.5 엣지 케이스 및 예외 처리 정책 [피드백 R11 반영]

운영 환경에서 발생 가능한 엣지 케이스에 대한 처리 정책을 정의합니다.

### 9.5.1 빈 프로젝트(코드 없음) 인덱싱

| 상황 | 처리 방식 |
|------|----------|
| Git 레포이나 소스 파일 0개 | 인덱싱 완료 처리 (빈 인덱스 생성) + "프로젝트에 분석 가능한 소스 파일이 없습니다. 코드를 추가한 후 `/impact reindex`를 실행하세요." 안내 |
| 지원 프레임워크 미감지 | 파일 구조 기반 폴백 인덱싱 수행 + "프레임워크를 자동 감지하지 못했습니다. 기본 파일 구조 기반으로 인덱싱했습니다." 경고 |
| 인덱싱 대상 파일이 극소수 (10개 미만) | 정상 인덱싱 후 "파일 수가 적어 분석 정확도가 낮을 수 있습니다." 안내 |

### 9.5.2 기획서와 코드의 연관성 부족

| 상황 | 처리 방식 |
|------|----------|
| 기획서 키워드와 코드 인덱스 매칭률 0% | "입력하신 기획서와 현재 프로젝트 코드 간 관련성을 찾지 못했습니다. 기획서 내용을 더 구체적으로 입력하거나, 올바른 프로젝트가 등록되었는지 확인하세요." 안내 후 분석 중단 |
| 매칭률 극히 낮음 (화면 1개 미만 매칭) | 매칭 결과를 PM에게 보여주고 "이 내용이 맞는지 확인해 주세요. 기획서가 이 프로젝트와 무관한 내용일 수 있습니다." 확인 요청 |

### 9.5.3 동일 기획서 재분석 일관성

- 동일 기획서 + 동일 인덱스 조합은 결과 캐시를 재사용하여 일관성 보장
- 인덱스가 변경된 경우에만 재분석 수행
- LLM temperature=0 설정으로 응답 변동 최소화
- 재분석 시 이전 결과와의 차이가 20% 이상이면 PM에게 "이전 분석과 결과 차이가 큽니다. 코드 변경이 영향을 준 것으로 보입니다." 안내

### 9.5.4 인덱싱 중 파일 접근 오류

| 상황 | 처리 방식 |
|------|----------|
| 파일 잠금 (다른 프로세스 사용 중) | 3회 재시도 (1초 간격) 후 건너뛰기, 완료 후 경고 |
| 파일 권한 부족 (읽기 불가) | 건너뛰고 경고 로그 기록, "N개 파일에 접근할 수 없어 분석에서 제외했습니다." 안내 |
| 파일 인코딩 오류 (비UTF-8) | 인코딩 자동 감지 시도, 실패 시 건너뛰기 |
| 심볼릭 링크 순환 | 순환 감지 후 해당 경로 건너뛰기 |

### 9.5.5 대규모 프로젝트 (50만+ LOC) 인덱싱 전략

- **자동 감지**: 파일 스캔 단계에서 총 LOC를 추정하여 30만 LOC 이상이면 PM에게 안내
- **모듈 단위 분할 인덱싱**: `packages/`, `modules/`, `apps/` 등 최상위 디렉토리 기준으로 분할
- **우선순위 인덱싱**: PM이 분석 대상 모듈을 지정할 수 있는 `--scope` 옵션 제공 (예: `/impact init /path --scope src/views,src/api`)
- **인덱싱 시간 초과 대응**: 10분 초과 시 현재까지 결과로 부분 인덱스 생성 + "전체 인덱싱이 오래 걸립니다. 현재까지 N개 파일이 인덱싱되었습니다. [계속 대기] [현재 결과 사용]" 안내
- **성능 목표**: 50만 LOC 프로젝트에서도 최대 15분 이내 완료 (SWC 병렬 파싱 활용)

### 9.5.6 LLM API Rate Limit 도달 시 처리

- **자동 재시도**: 429 응답 시 `Retry-After` 헤더 기반 대기 후 자동 재시도 (최대 3회)
- **백오프 전략**: 지수 백오프 (1초 -> 2초 -> 4초) 적용
- **프로바이더 폴백**: 주 프로바이더 rate limit 시 대체 프로바이더로 자동 전환 (예: Claude 한도 시 GPT-4o로 전환)
- **PM 안내**: "분석 서버가 일시적으로 바쁩니다. 잠시 후 자동으로 재시도합니다. (재시도 1/3)" 메시지 표시
- **중간 결과 보존**: rate limit으로 분석이 장시간 지연될 경우 `partial-result.json`에 진행 상황 저장, `--resume`으로 재개 가능
