## 2. Gap Health Widget (프로젝트 보드)

### 2.1 위치 및 레이아웃

Gap Health Widget은 프로젝트 보드(`/`)에서 **프로젝트 상태 배너 바로 아래, 분석 이력 섹션 바로 위**에 배치한다. 이유는 다음과 같다:

1. 프로젝트 상태 배너에서 "이 프로젝트가 무엇인지"를 파악한 직후
2. 분석 이력을 살펴보기 전에 "해결해야 할 문제가 있는지"를 먼저 인지
3. 문제가 없으면 (누락 0건) 위젯이 축소 상태로 표시되어 공간을 차지하지 않음

### 2.7 KPI 카드 상세 디자인

| 요소 | 스타일 |
|------|--------|
| HIGH 카드 배경 | `bg-red-50 border border-red-200 rounded-lg` |
| HIGH 숫자 | `text-2xl font-bold text-red-700` |
| HIGH 라벨 | `text-xs font-semibold text-red-500 uppercase` |
| MEDIUM 카드 배경 | `bg-amber-50 border border-amber-200 rounded-lg` |
| MEDIUM 숫자 | `text-2xl font-bold text-amber-700` |
| MEDIUM 라벨 | `text-xs font-semibold text-amber-500 uppercase` |
| LOW 카드 배경 | `bg-gray-50 border border-gray-200 rounded-lg` |
| LOW 숫자 | `text-2xl font-bold text-gray-600` |
| LOW 라벨 | `text-xs font-semibold text-gray-400 uppercase` |

HIGH가 0이면 숫자를 `text-gray-300`으로 처리하여 "위험 없음"을 시각적으로 강조한다. MEDIUM, LOW도 동일 규칙 적용.

### 2.8 상세 목록 항목 디자인

각 누락 항목은 다음 구조를 가진다:

```
+------------------------------------------------------------------+
| [심각도 배지]  누락 유형 + 건수                                     |
|   설명 텍스트 (영향 상세)                                           |
|                                              [해결 CTA 버튼]       |
+------------------------------------------------------------------+
```

- 심각도 배지: `[HIGH]` = `bg-red-100 text-red-700`, `[MEDIUM]` = `bg-amber-100 text-amber-700`, `[LOW]` = `bg-gray-100 text-gray-600`
- 설명 텍스트: `text-sm text-gray-600`
- CTA 버튼: `text-xs text-purple-600 hover:text-purple-800 border border-purple-200 rounded px-2 py-1`

### 2.9 CTA 버튼별 동작

| 누락 유형 | CTA 버튼 텍스트 | 동작 |
|----------|----------------|------|
| Stale 링크 | `인덱스 갱신` / `재감지` | 클립보드에 CLI 명령어 복사 + 토스트 안내 |
| 미분석 프로젝트 | `분석 실행 안내` | 클립보드에 CLI 명령어 복사 + 토스트 안내 |
| 저신뢰도 분석 | `재분석 안내` | 클립보드에 CLI 명령어 복사 + 토스트 안내 |
| 인덱스 미갱신 | `전체 재인덱싱` | 클립보드에 CLI 명령어 복사 + 토스트 안내 |
| 전체 자동 해결 | `자동 해결` | 클립보드에 `gap-check --fix` 명령어 복사 + 토스트 안내 |

**클립보드 복사 토스트 예시:**

```
+----------------------------------------------+
|  [v] 명령어가 클립보드에 복사되었습니다.         |
|      node dist/index.js gap-check --fix       |
+----------------------------------------------+
```

토스트: `bg-gray-900 text-white rounded-lg shadow-lg p-3`, 3초 후 자동 사라짐.

## 3. 플로우차트 "전체" 모드 개선

### 3.1 현재 상태 (As-Is)

현재 플로우차트 "전체" 모드 (`projectMode === 'all'`)는 다음 구성이다:

```
As-Is "전체" 모드:
+------+  +-------------------------------------------------------------+
| LNB  |  | 프로젝트  [v 전체 ^]                                         |
|      |  +-------------------------------------------------------------+
|      |  |                                                             |
|      |  | +-------------------------------------------------------+  |
|      |  | | 전체 프로젝트 영향도 - {specTitle}                       |  |
|      |  | |                                                       |  |
|      |  | | (CrossProjectDiagram - 원형 배치, 프로젝트 노드만)       |  |
|      |  | |                                                       |  |
|      |  | | (CrossProjectSummary - KPI 3개 + 타입 통계)             |  |
|      |  | +-------------------------------------------------------+  |
|      |  |                                                             |
+------+  +-------------------------------------------------------------+
```

문제점:
- 데이터가 `cross-project.json`에 없어 빈 화면 표시
- 프로젝트 간 관계만 표시하고, 공유 엔티티/이벤트는 미표시
- 상호작용이 제한적 (노드 클릭 시 동작 없음)

### 3.2 개선 방향 (To-Be)

REQ-015에서 `cross-project.json`에 자동 저장되므로, 데이터가 존재하는 전제로 풍부한 시각화를 설계한다.

## 4. 보완 분석 제안 UI

### 4.1 보완 분석 발생 시점

보완 분석은 CLI에서 `init` 명령어 실행 시 트리거되므로, 웹 대시보드에서 직접 발생하지 않는다. 다만, 보완 분석 결과가 저장된 후 웹에서 다음 두 곳에 표시된다:

1. **LNB 결과 목록** - 보완 분석 결과에 `(보완)` 라벨 표시
2. **프로젝트 보드 분석 이력 테이블** - 보완 분석 행에 `보완` 배지 표시

### 4.2 와이어프레임 - LNB 보완 분석 결과 카드

```
기존 ResultCard:                     보완 ResultCard:
+------------------------------+    +------------------------------+
| C   2026-02-20              |    | B   2026-02-22              |
|                              |    |                              |
| 상품 물리 스펙 수정 주체 변경 |    | [보완] 상품 물리 스펙 수정    |
|                              |    | -> e-scm-batch              |
| 총점 62/100                  |    | 총점 45/100                  |
| 영향 화면 3개 . 작업 12건    |    | 영향 화면 0개 . 작업 5건     |
+------------------------------+    +------------------------------+
 ^-- border-l-4 transparent         ^-- border-l-4 border-violet-400
```

**보완 라벨 디자인**:
```
[보완] = inline-flex items-center px-1.5 py-0.5 mr-1
         text-xs font-medium
         bg-violet-100 text-violet-700 rounded
```

- 기존 ResultCard의 `[예시]` 라벨(`bg-purple-100 text-purple-700`)과 유사한 패턴
- 색상을 `violet`으로 차별화하여 보완 분석임을 구분
- 좌측 보더: `border-violet-400` (기존 예시 결과의 `border-purple-400`과 구분)

### 4.3 와이어프레임 - 분석 이력 테이블 보완 표시

```
+-----------------------------------------------------------+
| 최근 분석 결과                                 [전체 보기 ->] |
|-----------------------------------------------------------|
| #  기획서명                        등급  점수  분석일       |
|----|-------------------------------|-----|------|----------|
| 1  상품 물리 스펙 수정 주체 변경    [C]    62    02/20     |
| 2  [보완] 상품 물리 스펙 -> batch  [B]    45    02/22     |
| 3  배송 정책 변경                  [B]    45    02/18     |
| 4  회원 등급 개편                  [A]    28    02/15     |
+-----------------------------------------------------------+
```

- `[보완]` 배지: 기획서명 앞에 `bg-violet-100 text-violet-700` 배지 추가
- 보완 분석 행의 기획서명은 `{원본 기획서명} -> {트리거 프로젝트명}` 형식

### 4.4 보완 분석 진행 중 표시

보완 분석은 CLI에서 실행되므로 웹에서 진행 상태를 실시간 표시하지 않는다. 단, Gap Health Widget에서 "보완 분석 필요" 항목으로 표시할 수 있다:

```
+------------------------------------------------------------------+
| [MEDIUM] 보완 분석 미실행 1건                                       |
|   신규 프로젝트 'e-scm-batch' 등록 후 기존 분석 보완 미수행           |
|   매칭 항목: "상품 물리 스펙 수정" (55%)                             |
|                                      [보완 분석 안내]               |
+------------------------------------------------------------------+
```

이 항목은 GapDetector의 "미분석 프로젝트" 유형에 포함되어 자동 표시된다.

### 4.5 보완 결과와 기존 결과의 관계 표시

보완 분석 결과를 LNB에서 선택하면, 기획 분석 대시보드(`/analysis`) 상단에 원본 분석 참조 배너를 표시한다:

```
+-----------------------------------------------------------------------+
| [i] 이 분석은 "상품 물리 스펙 수정 주체 변경"의 보완 분석입니다.          |
|     원본 분석 보기 ->                                                   |
+-----------------------------------------------------------------------+
```

- 배경: `bg-violet-50 border border-violet-200 rounded-lg p-3`
- 아이콘: info 서클 (`text-violet-500`)
- "원본 분석 보기" 링크: 클릭 시 LNB에서 원본 결과를 선택

### 8.1 `/api/gap-check` 응답 스키마 (예상)

```typescript
interface GapCheckResponse {
  gaps: GapItem[];
  summary: {
    high: number;
    medium: number;
    low: number;
    total: number;
  };
  lastCheckedAt: string;
}

interface GapItem {
  severity: 'high' | 'medium' | 'low';
  type: 'stale_link' | 'unanalyzed_project' | 'low_confidence' | 'outdated_index';
  title: string;
  description: string;
  fixCommand: string;
  count: number;
  details: GapDetail[];
}

interface GapDetail {
  subject: string;       // 대상 (프로젝트명, 링크명, 분석명)
  info: string;          // 추가 정보
  lastUpdated?: string;  // 마지막 갱신일
}
```

### 8.2 보완 분석 결과 식별

기존 `/api/results` 응답의 `ResultSummary`에 다음 필드를 추가하여 보완 분석을 식별한다:

```typescript
interface ResultSummary {
  // ... 기존 필드
  isSupplement?: boolean;        // 보완 분석 여부
  supplementOf?: string;         // 원본 분석 ID
  triggerProject?: string;       // 보완 트리거 프로젝트
}
```
