## 검토 항목 1: 플로우차트 뷰 계층 구조 변경

### 요청 내용
기존 4단계 계층(시스템 > 화면 > 하위기능 > 모듈)을 5단계 계층(요구사항 > 시스템 > 화면 > 하위기능 > 모듈)으로 변경한다.
최상위에 "요구사항(Requirement)" 레벨을 추가하여, 하나의 요구사항이 여러 시스템에 걸쳐 영향을 줄 때 이를 한눈에 파악할 수 있게 한다.

### PM 의견
- **찬성**. 컬리 PM들이 가장 많이 하는 질문이 "이 요구사항이 어디에 영향을 주는가?"이다. 시스템 단위로만 보면 요구사항 간의 관계를 파악하기 어렵다.
- 요구사항 레벨이 최상위에 있으면, PRD 기반 분석 결과를 바로 요구사항별로 그룹핑해서 볼 수 있어 실무 활용도가 높다.
- 하나의 요구사항이 여러 시스템(예: 주문, 결제, 알림)에 걸쳐 영향을 줄 때, 현재 구조에서는 시스템별로 분리되어 전체 그림을 보기 어렵다. 요구사항 레벨이 이 문제를 해결한다.
- US-008의 AC를 업데이트하고, 새로운 사용자 스토리 추가는 불필요하다 (기존 US-008의 범위 확장으로 충분).

### Designer 의견
- **찬성**. RequirementNode를 최상위 노드로 설계하면 시각적 계층이 명확해진다.
- RequirementNode 디자인: 260x130px, Indigo(#4338CA) 색상, 더블 보더로 다른 노드와 시각적으로 구분. 내부에 영향 시스템 수, 총 점수, 위험도 뱃지를 표시하여 정보 밀도를 높인다.
- expand/collapse 인터랙션이 중요하다. 요구사항 노드 클릭 시 하위 시스템들이 펼쳐지고, "모두 접기" 시 요구사항+시스템 레벨까지만 표시되어야 한다.
- 하나의 시스템이 여러 요구사항에 속할 수 있으므로, 공유 시스템은 각 요구사항 하위에 중복 표시하되 시각적으로 "공유" 표시(점선 테두리 + 링크 아이콘)를 추가한다.
- 5단계 계층이 화면에 깊이가 깊어질 수 있으므로, dagre 레이아웃의 ranksep/nodesep 조정이 필요하다.

### TPO 의견
- **찬성 (조건부)**. 기술적으로 구현 가능하며 dagre 레이아웃은 5단계까지 충분히 지원한다.
- ParsedSpec 인터페이스에 `requirements: ParsedRequirement[]` 필드를 추가하고, LLM 파싱 단계에서 요구사항 추출을 수행한다.
- React Flow 커스텀 노드가 6종에서 8종으로 증가하여 프론트엔드 볼륨이 다소 증가하지만, MVP 범위 내에서 관리 가능하다.
- **조건**: RequirementNode의 layout 크기(260x130)가 기존 SystemNode(240x120)보다 크므로, dagre 레이아웃 설정에서 노드별 크기 분기 처리가 필요하다. 이는 단순 설정 변경이므로 리스크는 낮다.
- 복잡도 영향: 시각화 웹 +0.5 (7→7.5/10), 전체 복잡도 7→7.5/10으로 소폭 상승.

### 최종 결정
- **승인**: MVP에 포함
- 기존 4단계 계층을 5단계로 확장 (요구사항 > 시스템 > 화면 > 하위기능 > 모듈)
- RequirementNode를 React Flow 커스텀 노드로 추가 구현
- ParsedSpec에 requirements 필드 추가, LLM 파싱 시 요구사항 추출
- 공유 시스템 처리: 점선 테두리 + 링크 아이콘으로 시각적 표현

### 사용자 관점 검토
- **긍정적 영향**: PM이 "요구사항 A는 어떤 시스템에 영향을 주는가?"를 한눈에 파악 가능. 기존에는 시스템별로 개별 분석 결과를 확인해야 했으나, 이제 요구사항 중심으로 통합 조회 가능.
- **편의성**: 큰 PRD에 여러 요구사항이 있을 때, 요구사항별로 접기/펼치기가 가능하여 정보 과부하를 줄일 수 있다.
- **학습 비용**: 기존 4단계에서 5단계로 늘어나지만, 최상위 레벨 추가이므로 기존 사용 경험에 영향이 없다. 자연스러운 확장.

## 검토 항목 2: 정책 사전(Policy Dictionary) 연동

### 요청 내용
코드 내 주석에서 기획 정책을 자동 추출하여 인덱싱하고, 영향도 분석 시 관련 정책 위반 가능성을 경고하는 기능을 추가한다.
주석 패턴: `// 정책:`, `// Policy:`, `@policy`, JSDoc `@policy` 태그 등.

### PM 의견
- **강력 찬성**. 컬리에서 가장 빈번한 장애 원인 중 하나가 "기존 정책을 모르고 코드를 수정"하는 것이다.
- 예시: "컬리 멤버스 가입자만 무료배송" 정책이 코드에 하드코딩되어 있는데, 배송 관련 기능을 수정하면서 이 정책을 인지하지 못해 멤버스 비가입자에게도 무료배송이 적용되는 사고가 발생할 수 있다.
- 정책 경고가 영향도 분석 결과에 자동으로 포함되면, PM이 "이 변경 사항에 관련된 정책이 있나요?"를 묻기 전에 선제적으로 경고를 받을 수 있다.
- MVP에서는 코드 주석 기반 정책 추출로 시작하고, Phase 2에서 Confluence 문서 연동을 추가하는 점진적 접근이 적절하다.
- 새로운 사용자 스토리 US-010으로 추가.

### Designer 의견
- **찬성**. 정책 경고 노드(PolicyWarningNode)를 플로우차트에 추가하여 시각적으로 즉시 인지 가능하게 한다.
- PolicyWarningNode 디자인: 200x100px, Orange(#F97316) 배경 + Red(#EF4444) 보더로 경고 성격을 강조. 방패(shield) + 경고(alert-triangle) 아이콘 조합.
- 체크리스트 뷰에도 정책 경고 섹션을 추가하여, 개발자가 체크리스트를 확인할 때 정책 관련 항목도 함께 볼 수 있게 한다.
- 정책 경고 카드 레이아웃: 정책명, 원문 위치(파일+라인), 위반 위험도(Low/Medium/High 뱃지), 경고 메시지를 포함.
- KPI 카드에 "정책 경고" 항목 추가 (기존 4개 → 5개).

### TPO 의견
- **찬성**. 기술적으로 코드 주석 파싱은 정규식 기반으로 구현 가능하며, AST 수준의 복잡한 분석은 불필요하다.
- 인덱싱 파이프라인에 Step 3.5(정책 추출)를 추가. 파일 스캔 시 주석 패턴 매칭으로 정책을 추출하여 `policies.json`에 저장.
- 분석 파이프라인에 Step 5.5(정책 매칭)를 추가. 영향받는 컴포넌트와 정책의 `relatedComponents`를 매칭하여 위반 가능성을 판단.
- 새로운 파일 추가: `policy-extractor.ts`(추출), `policy-matcher.ts`(매칭), `policies.ts`(커맨드)
- CodeIndex 인터페이스에 `policies: Policy[]` 필드 추가, ImpactResult에 `policyWarnings: PolicyWarning[]` 추가.
- **리스크**: 주석 기반 추출의 한계 - 주석 형식이 통일되지 않은 레포에서는 추출률이 낮을 수 있다. MVP에서는 `// 정책:` 패턴만 지원하고, 사용자 피드백을 받아 패턴을 확장하는 것이 현실적이다.
- 복잡도 영향: 데이터 레이어 +1 (4→5/10), 전체 복잡도에 0.1 기여.

### 최종 결정
- **승인**: MVP에 포함
- 코드 주석 기반 정책 추출 (`// 정책:`, `// Policy:`, `@policy` 패턴)
- policies.json으로 인덱싱, 분석 시 자동 매칭
- PolicyWarningNode 플로우차트 노드 + 체크리스트 경고 섹션
- Phase 2: Confluence 문서 기반 정책 추출 확장
- `/impact policies` 커맨드 추가

### 사용자 관점 검토
- **긍정적 영향**: PM이 영향도 분석을 요청하면, 관련 정책 위반 가능성을 사전에 경고받아 사고를 예방할 수 있다. "이 변경 때문에 기존 정책이 깨질 수 있어요"라는 정보를 자동으로 받는 것은 높은 실무 가치가 있다.
- **편의성**: 별도로 정책 문서를 찾아보지 않아도, 코드에 기록된 정책이 자동으로 연결되어 표시된다.
- **한계 인지**: 코드 주석에 정책이 기록되지 않은 경우 추출 불가. MVP 단계에서는 이 한계를 사용자에게 명시하고, 주석 추가를 권장하는 가이드를 제공해야 한다.

## 검토 항목 3: 시스템별 담당자/팀 매핑 및 확인 요청 안내

### 요청 내용
시스템별 담당 팀/담당자를 등록하고, 영향도 분석 결과에서 어떤 팀에 무엇을 확인해야 하는지를 안내하며, 이메일 초안(확인 요청 메일)을 자동 생성하는 기능을 추가한다.

### PM 의견
- **강력 찬성**. 영향도 분석 결과를 받은 후 "그래서 누구에게 확인해야 하지?"가 항상 다음 질문이다. 현재는 영향받는 시스템 목록만 나오고, 담당자 정보는 PM이 별도로 찾아야 한다.
- 확인 요청 안내가 영향도 분석 결과에 자동으로 포함되면, 분석 → 확인 요청까지의 워크플로우가 끊기지 않는다.
- 이메일 초안 자동 생성은 PM의 커뮤니케이션 업무 부담을 크게 줄여준다. "주문 시스템 변경에 대해 결제팀에 확인 요청"같은 메일을 매번 작성할 필요가 없다.
- 새로운 사용자 스토리 US-011로 추가.
- `/impact owners` 커맨드로 담당자 정보 관리 가능.

### Designer 의견
- **찬성**. 확인 요청 안내를 별도 탭("확인 요청")으로 분리하여, 플로우차트 뷰와 체크리스트 뷰의 복잡도를 높이지 않으면서 기능을 추가한다.
- 네비게이션 탭 추가: [플로우차트] [체크리스트] [확인 요청] [티켓] [보고서]
- OwnerCard 디자인: 320x160px, Sky Blue(#E0F2FE) 배경, 카드 형식. 팀명, 담당자명, 이메일, 확인 항목 리스트, 이메일 초안 복사 버튼을 포함.
- 확인 요청 안내 뷰 와이어프레임: 팀별로 카드를 그룹핑하여 표시. 각 카드에 해당 팀이 확인해야 할 항목과 이메일 초안 버튼을 배치.
- 이메일 초안은 모달 또는 사이드 패널로 표시하며, 클립보드 복사 기능을 제공.

### TPO 의견
- **찬성**. 기술적으로 가장 구현이 간단한 항목이다.
- `owners.json` 설정 파일 기반으로 담당자 매핑. 별도의 외부 시스템 연동 없이, 프로젝트 루트에 JSON 파일로 관리.
- 분석 파이프라인 Step 5.5에서 영향받는 시스템과 owners.json을 매칭하여 OwnerNotification 목록을 생성.
- ImpactResult에 `ownerNotifications: OwnerNotification[]` 추가.
- 이메일 초안 생성은 LLM을 활용하여 자연스러운 문장으로 생성. 템플릿 기반이 아닌, 분석 결과를 반영한 맞춤 초안.
- 새로운 파일 추가: `owner-mapper.ts`(매핑), `owners.ts`(커맨드), `Owners.tsx`(페이지), `OwnerCard.tsx`, `EmailDraft.tsx`
- REST API 6개 추가: owners CRUD(4개) + policies GET(1개) + notifications GET(1개)
- **리스크**: owners.json의 최신성 관리. 조직 변경 시 수동 업데이트 필요. Phase 2에서 HR 시스템/Slack 연동으로 자동화 가능.
- 복잡도 영향: 최소 (JSON 파일 기반 단순 매핑).

### 최종 결정
- **승인**: MVP에 포함
- owners.json 기반 시스템-팀-담당자 매핑
- 분석 결과에 OwnerNotification 자동 포함
- 확인 요청 안내 뷰 (별도 탭) 신규 구현
- 이메일 초안 LLM 자동 생성 + 클립보드 복사
- Phase 2: HR 시스템/Slack 연동으로 담당자 정보 자동 동기화
- `/impact owners` 커맨드 추가 (init/list/add/remove)

### 사용자 관점 검토
- **긍정적 영향**: 영향도 분석 → 확인 요청까지의 전체 워크플로우가 하나의 도구에서 완결된다. PM이 분석 결과를 받고 즉시 관련 팀에 확인 요청을 보낼 수 있다.
- **편의성**: 이메일 초안 자동 생성으로 커뮤니케이션 준비 시간을 크게 절약. "결제팀에 확인 요청 메일을 어떻게 쓸까"를 고민하지 않아도 된다.
- **한계 인지**: owners.json을 수동으로 관리해야 하므로, 초기 설정 비용이 있다. `/impact owners init` 커맨드로 대화형 설정을 지원하여 진입 장벽을 낮춘다.

## 결론

3건의 추가 요청 모두 **MVP 범위에 포함하여 승인**한다.

- **요구사항 레벨 계층**: PM의 핵심 질문("이 요구사항이 어디에 영향을 주는가?")을 직접 해결하며, 기술적 리스크가 낮다.
- **정책 사전**: 컬리의 실제 장애 예방에 직접 기여하는 고가치 기능이며, 점진적 확장이 가능하다.
- **담당자 매핑**: 영향도 분석의 실무 활용도를 크게 높이며, 구현 복잡도가 가장 낮다.

종합 복잡도가 7.0에서 7.5로 소폭 상승하지만, 기존 아키텍처(React Flow, dagre, Express)를 확장하는 방식이므로 추가 기술 스택 도입 없이 구현 가능하다.
