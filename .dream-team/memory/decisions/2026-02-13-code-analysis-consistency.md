## 문제 정의

### 핵심 문제
"시스템별로 주석이 있는 것도 있고 없는 것도 있어서, 일관된 소스 분석을 통한 영향도 체크가 어렵다."

### 문제의 본질 (심층 분석)

이 문제는 단순히 "정책 주석"의 유무에만 국한되지 않는다. 컬리의 다양한 시스템이 가진 구조적 편차가 영향도 분석의 일관성과 신뢰성을 위협하는 근본 문제이다.

| 편차 유형 | 설명 | 영향도 분석에 미치는 영향 |
|-----------|------|-------------------------|
| **코드 품질 편차** | 어떤 시스템은 주석/문서가 풍부하고, 어떤 시스템은 거의 없음 | 정책 사전 추출 결과의 불균형 |
| **코드 구조 편차** | 표준 프레임워크 구조 vs 레거시/비정형 구조 | 화면/라우트 자동 추출 성공률 차이 |
| **프레임워크 편차** | React, Vue, Spring, Express 등 다양한 스택 혼재 | 파서별 분석 깊이와 정확도 차이 |
| **모노레포 vs 멀티레포** | 같은 레포에 있는 것도, 분리된 것도 있음 | 프로젝트 간 의존 관계 추적 난이도 |
| **정보 부족 시스템** | 주석도 없고, 라우팅도 비표준이고, 구조도 비정형 | 분석 결과 자체의 신뢰도 저하 |

### 위험 시나리오

PM이 영향도 분석 결과를 받았을 때:
- **시스템 A** (주석 풍부, 표준 구조): 정확하고 상세한 분석 결과
- **시스템 B** (주석 없음, 레거시 구조): 피상적이고 불완전한 분석 결과

PM은 두 결과를 동일한 신뢰도로 받아들이게 되며, 이는 잘못된 의사결정으로 이어질 수 있다.

## PM 관점 분석

### PM이 가장 중요하게 생각하는 것

1. **신뢰성**: "이 분석 결과를 믿어도 되는가?"
   - 분석 결과의 신뢰도를 모르면 PM은 결과를 활용할 수 없다
   - 부정확한 결과보다 "정확도가 낮을 수 있음"을 아는 것이 낫다

2. **일관성**: "시스템마다 결과 품질이 다르면 혼란스럽다"
   - 어떤 시스템은 상세하고 어떤 시스템은 빈약하면 비교가 불가능
   - 최소한의 기본 분석은 모든 시스템에서 가능해야 함

3. **투명성**: "왜 이 분석이 정확하지 않은지 알려달라"
   - 단순히 "신뢰도 60%"보다 "주석 정보가 부족하여 정책 분석이 제한적"이 유용
   - PM이 보완할 수 있는 액션이 함께 제시되어야 함

### PM 관점 결론

- 분석 결과에 **신뢰도 점수**를 함께 표시하는 것이 핵심
- 신뢰도가 낮은 경우 **이유와 보완 방법**을 안내
- "60% 신뢰도"라도 **출발점으로서 가치**가 있음 (0%보다 나음)
- PM이 직접 정책 정보를 보강할 수 있는 **수동 입력 경로** 필요

## TPO 관점 분석

### 기술적 현실 인식

코드 분석의 정보원은 "주석"만이 아니다. 주석 없이도 파악 가능한 정보가 많으며, 다층적 분석을 통해 주석 의존도를 낮출 수 있다.

| 정보 유형 | 주석 필요 여부 | 추출 방법 |
|-----------|--------------|----------|
| 파일/디렉토리 구조 | 불필요 | 파일 시스템 스캔 |
| import/export 의존 관계 | 불필요 | AST 파싱 |
| 라우팅 구조 | 불필요 (표준 프레임워크) | 프레임워크별 파서 |
| API 엔드포인트 | 불필요 | AST 패턴 매칭 |
| 컴포넌트 props/emit | 불필요 | AST 파싱 |
| 비즈니스 로직 의도 | **필요** | 주석, 문서, LLM 추론 |
| 정책 규칙 | **필요** | 주석, 정책 문서 |
| 기획 배경/맥락 | **필요** | 주석, 외부 문서 |

### 핵심 인사이트

전체 분석 영역 중 주석이 반드시 필요한 부분은 "비즈니스 로직 의도"와 "정책 규칙"에 한정된다. 나머지 구조/의존성/API 분석은 코드 자체에서 100% 추출 가능하다. 또한 LLM의 코드 이해 능력을 활용하면 주석 없이도 비즈니스 로직을 일부 추론할 수 있다.

## 해결 방안: 다층 분석 전략 (Multi-Layer Analysis)

### 전략 개요

주석에만 의존하는 단일 분석에서 벗어나, 4개의 독립적인 분석 레이어를 구축하여 각 레이어의 결과를 종합한다. 주석이 없는 시스템에서도 Layer 1, 2, 4를 통해 최소한의 영향도 분석이 가능하다.

```
[기획서 + 코드 인덱스]
        │
        ├── Layer 1: 코드 구조 분석 ──────────── 주석 불필요 (항상 가능)
        │     └── AST, 파일 구조, 라우팅 추출
        │
        ├── Layer 2: 의존성 그래프 분석 ────────── 주석 불필요 (항상 가능)
        │     └── import/export, API 호출 관계
        │
        ├── Layer 3: 주석/문서 기반 정책 분석 ──── 주석 필요 (있으면 보강)
        │     └── 정책 주석, README, 설정 파일
        │
        └── Layer 4: LLM 추론 분석 ──────────── 주석 없어도 부분적 가능
              └── 코드 맥락에서 비즈니스 로직 추론
        │
        ▼
[분석 결과 + 신뢰도 점수]
```

### Layer 1: 코드 구조 분석 (주석 불필요)

**항상 수행 가능한 기본 분석 레이어**. 코드 구조만으로 영향 범위를 파악한다.

| 분석 항목 | 방법 | 산출물 |
|-----------|------|--------|
| 파일/디렉토리 구조 | `fast-glob` 스캔 | 모듈 경계 식별 |
| 라우팅 구조 | 프레임워크별 파서 (react-router, vue-router, @Controller 등) | 화면 목록 |
| 컴포넌트 계층 | AST 파싱 (SWC, tree-sitter) | 컴포넌트 트리 |
| API 엔드포인트 | Express/Spring 라우터 패턴 매칭 | API 목록 |
| 데이터 모델 | TypeScript interface, ORM 모델 분석 | 데이터 구조 |

**비표준 프레임워크 대응 (폴백 전략)**:
- 프레임워크 파서가 실패할 경우, 파일 구조 기반 분석으로 폴백
- `pages/`, `views/`, `screens/` 등 관례적 디렉토리명으로 화면 추론
- `api/`, `routes/`, `controllers/` 등으로 API 추론
- 폴백 사용 시 해당 프로젝트의 신뢰도 점수에 반영

**Layer 1이 제공하는 정보**:
- "이 기능은 파일 A, B, C에 영향을 줄 수 있음" (구조적 매칭)
- "화면 X와 화면 Y는 같은 컴포넌트를 공유함" (공유 리소스 식별)
- "API 엔드포인트 N개가 관련될 수 있음" (API 범위)

### Layer 2: 의존성 그래프 분석 (주석 불필요)

**모듈 간 연결 관계를 추적하여 영향 전파 범위를 식별**한다.

| 분석 항목 | 방법 | 산출물 |
|-----------|------|--------|
| import/export 그래프 | `madge` + 커스텀 분석기 | 모듈 의존 DAG |
| API 호출 관계 | axios/fetch 패턴 매칭 → 엔드포인트 매핑 | FE→BE 호출 그래프 |
| 이벤트/메시지 흐름 | EventEmitter, pub/sub 패턴 | 비동기 의존 관계 |
| 공유 모듈 식별 | 진입차수(in-degree) 분석 | 고위험 공유 모듈 |

**의존성 기반 영향 전파 분석**:
```
매칭된 엔티티 → 1-hop 의존 (직접 영향) → 2-hop 의존 (간접 영향)
                 ↑ 높은 신뢰도            ↑ 보통 신뢰도
```
- 직접 의존 (1-hop): 높은 신뢰도로 영향 판단
- 간접 의존 (2-hop): 보통 신뢰도, "가능성 있음"으로 표시
- 3-hop 이상: 신뢰도 낮음, 기본적으로 표시하지 않음 (옵션으로 활성화)

**Layer 2가 제공하는 정보**:
- "모듈 A를 변경하면 B, C, D가 영향 받음" (영향 전파 경로)
- "이 컴포넌트는 15개 다른 모듈에서 사용됨" (위험도 평가)
- "프론트엔드 컴포넌트 X가 백엔드 API Y를 호출함" (FE-BE 매핑)

### Layer 3: 주석/문서 기반 정책 분석 (주석 필요)

**기존 PLAN-001의 정책 사전(섹션 3.10) 기능에 해당**하며, 주석이 있는 시스템에서만 추가 정보를 제공한다.

| 분석 항목 | 방법 | 산출물 |
|-----------|------|--------|
| 정책 주석 | `// 정책:`, `@policy` 패턴 매칭 | 정책 사전 |
| README 정책 섹션 | 마크다운 파싱 | 문서 기반 정책 |
| 설정 파일 주석 | 설정 파일 내 주석 파싱 | 설정 관련 정책 |
| 수동 입력 정책 | policies.yaml (사용자 직접 작성) | 보강 정책 |

**주석이 없을 때의 동작**:
- Layer 3의 결과가 비어 있어도 분석은 계속 진행
- "이 시스템에서 정책 주석이 발견되지 않았습니다" 안내 표시
- 수동 정책 입력 가이드 제공 (policies.yaml 편집 또는 CLI 명령어)

**Layer 3이 제공하는 정보**:
- "이 코드 변경은 '무료배송' 정책에 영향을 줄 수 있음" (정책 매칭)
- "관련 정책 3건이 발견됨" (정책 카운트)
- 주석 없는 시스템: "정책 정보 없음 - 수동 확인 필요" (명시적 경고)

### Layer 4: LLM 추론 분석 (주석 없어도 부분적 가능)

**LLM의 코드 이해 능력을 활용하여 비즈니스 로직을 추론**한다. 주석이 없어도 코드 패턴, 함수명, 변수명 등에서 의도를 유추할 수 있다.

| 분석 항목 | 방법 | 산출물 |
|-----------|------|--------|
| 비즈니스 로직 추론 | LLM에 코드 스니펫 제공 → 의도 추론 | 추론된 비즈니스 규칙 |
| 변경 영향 심층 분석 | 기획서 + 코드 컨텍스트 → 영향 판단 | 상세 영향 분석 |
| 기획 확인 사항 도출 | 기획서 vs 코드 차이점 분석 → 확인 필요 사항 | 기획 확인 체크리스트 |
| 작업 난이도 평가 | 코드 복잡도 + 변경 범위 → 점수 산출 | 난이도 점수 |

**LLM 추론의 한계 인식**:
- 주석/문서가 있으면 LLM의 추론 정확도가 대폭 향상
- 주석 없이 코드만으로 추론할 때는 정확도가 떨어질 수 있음
- LLM 추론 결과에는 항상 "추론 근거"를 함께 제시
- 추론 결과가 불확실한 경우 "확인 필요" 마커를 부착

**Layer 4가 제공하는 정보**:
- "이 함수는 배송비 계산과 관련된 로직으로 보입니다" (비즈니스 의도 추론)
- "이 변경은 결제 프로세스의 할인 적용 순서에 영향을 줄 수 있습니다" (영향 추론)
- "주석이 없어 정확도가 제한적입니다. 담당 개발자 확인을 권장합니다" (한계 명시)

## 신뢰도 점수(Confidence Score) 시스템

### 산출 공식

신뢰도 점수는 4개 Layer의 정보 제공 정도를 종합하여 산출한다.

```
신뢰도 점수 = (Layer1 가중치 x Layer1 점수) + (Layer2 가중치 x Layer2 점수)
            + (Layer3 가중치 x Layer3 점수) + (Layer4 가중치 x Layer4 점수)
```

| Layer | 가중치 | 점수 범위 | 점수 산출 기준 |
|-------|--------|----------|---------------|
| Layer 1 (구조 분석) | 25% | 0~100 | 프레임워크 파서 성공률, 화면/API 추출 성공 여부 |
| Layer 2 (의존성 분석) | 25% | 0~100 | 의존 그래프 완성도, import 추적 성공률 |
| Layer 3 (정책 분석) | 20% | 0~100 | 정책 주석 밀도, 정책 사전 항목 수 |
| Layer 4 (LLM 추론) | 30% | 0~100 | LLM 추론 성공률, 근거 명확도 |

**Layer별 점수 산출 세부 기준**:

**Layer 1 점수**:
- 프레임워크 자동 감지 성공: +30
- 라우팅/화면 추출 성공: +30
- API 엔드포인트 추출 성공: +20
- 컴포넌트 계층 추출 성공: +20
- 폴백 분석 사용 시: 각 항목 절반 점수

**Layer 2 점수**:
- import/export 그래프 생성 성공: +40
- API 호출 관계 매핑 성공: +30
- 공유 모듈 식별 성공: +15
- 크로스 프로젝트 의존 추적: +15

**Layer 3 점수**:
- 정책 주석 1건 이상 존재: +40
- 정책 주석 5건 이상: +20 (밀도 보너스)
- README 정책 섹션 존재: +20
- 수동 정책 입력(policies.yaml) 존재: +20

**Layer 4 점수**:
- LLM 응답 구조화 성공 (JSON 파싱 가능): +30
- 영향 분석에 코드 경로 근거 포함: +30
- 비즈니스 로직 추론에 명확한 근거 제시: +20
- 점수 산출 근거가 Calibration Example과 일관성: +20

### 종합 신뢰도 등급

| 신뢰도 범위 | 등급 | 의미 | PM에게 표시 |
|------------|------|------|------------|
| 85~100 | **High Confidence** | 모든 Layer가 충분한 정보 제공 | "이 분석은 높은 신뢰도로 수행되었습니다" |
| 65~84 | **Medium Confidence** | 대부분의 Layer가 정보 제공, 일부 제한 | "이 분석은 대체로 신뢰할 수 있으나 일부 항목에서 추가 확인이 필요합니다" |
| 40~64 | **Low Confidence** | 핵심 Layer 정보 부족 | "이 분석은 참고용입니다. 정책/비즈니스 규칙에 대한 추가 확인이 필요합니다" |
| 0~39 | **Very Low Confidence** | 대부분의 Layer 정보 부족 | "이 시스템의 분석 정보가 매우 부족합니다. 담당 개발자와 직접 확인을 권장합니다" |

### 표시 방식 (PM 친화적)

**터미널 (CLI) 표시**:
```
 영향도 분석 완료

  [프로모션 시스템]  신뢰도: ████████████████████ 92% (High)
  [물류 시스템]      신뢰도: ████████████░░░░░░░░ 68% (Medium)
  [레거시 정산]      신뢰도: ██████░░░░░░░░░░░░░░ 42% (Low)
                     └ 정책 주석 없음, 비표준 라우팅 구조
                     └ 보완 방법: /impact policies add 또는 policies.yaml 편집
```

**웹 시각화 표시**:
- 시스템 노드에 신뢰도 배지 표시 (아이콘 + 색상)
- 대시보드 프로젝트 프로파일 카드에 각 프로젝트의 종합 신뢰도
- 신뢰도가 Low 이하인 항목에 경고 아이콘

## 프로젝트 프로파일 시스템

### 개요

인덱싱 완료 시 각 프로젝트의 "분석 가능성 프로파일"을 자동 생성한다. 이를 통해 PM이 프로젝트별 분석 품질을 사전에 인지하고, 보완이 필요한 부분을 파악할 수 있다.

### 프로파일 항목

| 항목 | 측정 방법 | 점수 범위 | 영향 |
|------|----------|----------|------|
| **프레임워크 인식률** | 지원 프레임워크 감지 여부 | 0~100 | Layer 1 점수에 직접 영향 |
| **라우팅 표준화 수준** | 프레임워크 표준 라우터 사용 여부 | 0~100 | 화면 추출 정확도 |
| **주석 밀도** | 코드 라인 대비 주석 비율 (%) | 0~100 | Layer 3 점수에 직접 영향 |
| **정책 주석 수** | 정책 관련 주석 패턴 발견 수 | 절대값 | 정책 분석 가능 여부 |
| **의존 그래프 완성도** | import/export 추적 성공률 | 0~100 | Layer 2 점수에 직접 영향 |
| **API 매핑률** | 감지된 API 호출 중 엔드포인트 매핑 성공률 | 0~100 | FE-BE 연관 분석 정확도 |
| **타입 커버리지** | TypeScript 타입 정의 비율 | 0~100 | 데이터 모델 분석 정확도 |
| **테스트 커버리지** | 테스트 파일 존재 비율 | 0~100 | 참고 정보 (신뢰도에 간접 영향) |

## 정보 부족 시스템 보완 전략

### 수동 정책 입력

주석이 없는 시스템의 정책 정보를 PM이 직접 입력할 수 있는 경로를 제공한다.

**CLI 명령어**:
```
/impact policies add "배송비 정책: 3만원 이상 무료배송"
/impact policies add --system "물류" --category "배송" "새벽배송은 서울/경기만 가능"
/impact policies add --file src/services/delivery/ "배송일 계산 시 공휴일 제외"
```

**명령어 옵션**:

| 옵션 | 설명 | 필수 |
|------|------|------|
| `<정책 내용>` | 정책 설명 텍스트 | 필수 |
| `--system <시스템명>` | 관련 시스템 지정 | 선택 |
| `--category <카테고리>` | 정책 카테고리 (배송, 결제, 프로모션 등) | 선택 |
| `--file <경로>` | 관련 코드 파일/디렉토리 경로 | 선택 |

**입력 확인**:
```
 정책이 등록되었습니다.

  정책 ID: policy-manual-003
  내용: 배송비 정책: 3만원 이상 무료배송
  시스템: 물류
  카테고리: 배송
  등록 방식: 수동 입력

  이 정책은 다음 영향도 분석 시 자동으로 매칭됩니다.
```

### 정책 사전 파일 (policies.yaml)

대량의 정책을 한번에 등록하거나, 체계적으로 관리하기 위한 파일 기반 입력 방식이다.

**파일 위치**: `.impact/projects/<project-id>/policies.yaml`

```yaml
# Kurly Impact Checker - 수동 정책 사전
# 코드 주석에 정책 정보가 없는 시스템의 비즈니스 규칙을 여기에 등록하세요.

policies:
  # 배송 관련 정책
  - name: "무료배송 기준"
    description: "주문 금액 3만원 이상 시 무료배송 적용"
    category: "배송"
    system: "물류"
    relatedPaths:
      - "src/services/delivery/"
      - "src/api/delivery/"

  - name: "새벽배송 지역 제한"
    description: "새벽배송은 서울/경기 지역만 가능. 그 외 지역은 일반택배"
    category: "배송"
    system: "물류"
    relatedPaths:
      - "src/services/delivery/dawn-delivery.ts"

  # 결제 관련 정책
  - name: "쿠폰-포인트 동시 사용 제한"
    description: "하나의 주문에 쿠폰과 적립금을 동시에 사용할 수 없음"
    category: "결제"
    system: "커머스"
    relatedPaths:
      - "src/services/cart/discount/"

  # 프로모션 관련 정책
  - name: "중복 할인 금지"
    description: "동일 상품에 두 개 이상의 할인을 중복 적용할 수 없음"
    category: "프로모션"
    system: "프로모션"
    relatedPaths:
      - "src/services/promotion/"
```

**policies.yaml 로드 규칙**:
- 인덱싱 시 자동으로 policies.yaml을 읽어 정책 사전에 병합
- 코드 주석 기반 정책과 수동 입력 정책을 구분하여 저장 (source: "comment" vs "manual")
- 수동 입력 정책은 코드 주석 정책보다 우선하지 않되, 보완 역할
- policies.yaml이 없으면 무시 (에러 아님)

인덱싱 완료 후 프로젝트 프로파일의 약점을 기반으로 보완 가이드를 자동 생성한다.

- 프로젝트 프로파일의 분석 준비도가 70% 미만일 때
- 특정 Layer의 점수가 30% 미만일 때
- 정책 주석이 0건일 때

**가이드 출력 형식 (CLI)**:

       파일 위치: .impact/projects/kurly-legacy-settlement/policies.yaml

## 분석 결과 시 "신뢰도"를 PM에게 어떻게 보여줄 것인가

### 시각화 방안

#### 1. 시스템 노드에 신뢰도 배지

플로우차트의 시스템 노드에 신뢰도 등급을 배지로 표시한다.

**High Confidence 시스템 노드**:
```
┌─────────────────────────────────┐
│  [서버] 컬리 커머스     [HC 92%]│  ← 초록 배지
│                                 │
│  종합 32점  Medium               │
│  화면 3개 | 작업 8개             │
└─────────────────────────────────┘
```

**Low Confidence 시스템 노드**:
```
┌─────────────────────────────────┐
│  [서버] 레거시 정산     [LC 42%]│  ← 주황 배지 + 경고 아이콘
│                                 │
│  종합 15점  Low                  │
│  화면 2개 | 작업 3개             │
│  정보 부족 - 보완 필요           │
└─────────────────────────────────┘
```

**신뢰도 배지 색상**:

| 등급 | 배지 색상 | 배지 텍스트 |
|------|----------|------------|
| High Confidence | `#22C55E` (Green) | HC + 퍼센트 |
| Medium Confidence | `#EAB308` (Yellow) | MC + 퍼센트 |
| Low Confidence | `#F97316` (Orange) | LC + 퍼센트 |
| Very Low Confidence | `#EF4444` (Red) | VLC + 퍼센트 |

#### 2. 대시보드에 프로파일 섹션

대시보드에 "분석 신뢰도" 섹션을 추가하여 시스템별 신뢰도 현황을 한눈에 파악한다.

```
┌─────────────────────────────────────────────────────────────┐
│  분석 신뢰도 현황                                            │
│                                                             │
│  프로모션 시스템  ████████████████████ 92%  High  [상세]     │
│  커머스 시스템    ████████████████░░░░ 78%  Med.  [상세]     │
│  물류 시스템      ████████████░░░░░░░░ 68%  Med.  [상세]     │
│  레거시 정산      ██████░░░░░░░░░░░░░░ 42%  Low   [보완 가이드]│
│                                                             │
│  평균 신뢰도: 70% (Medium)                                   │
│  낮은 신뢰도 시스템이 1개 있습니다. 보완 가이드를 확인하세요.  │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 3. 노드 상세 패널에 Layer별 상세

노드 상세 패널에서 해당 시스템/화면의 Layer별 분석 상세를 확인할 수 있다.

### 경고 메시지 형식

신뢰도가 낮은 시스템이 포함된 분석 결과에 경고를 표시한다.

**CLI 경고 (분석 완료 시)**:
```
 주의: 일부 시스템의 분석 신뢰도가 낮습니다.

  [레거시 정산] 신뢰도 42% (Low)
    원인: 정책 주석 없음, 비표준 라우팅 구조
    영향: 이 시스템의 영향도 분석은 참고용입니다.
          실제 영향은 담당 개발자와 확인이 필요합니다.
    보완: /impact policies add 명령으로 정책을 수동 등록하세요.

  나머지 2개 시스템의 신뢰도는 Medium 이상입니다.
```

```

```

## 사용자 관점 검토

이 결정이 PM(사용자)에게 미치는 영향:

1. **긍정적 영향**
   - 주석 없는 시스템도 최소한의 영향도 분석이 가능해짐
   - 분석 결과의 신뢰도를 투명하게 확인 가능
   - 보완 방법이 명확하여 점진적으로 분석 품질 향상 가능
   - "이 결과를 믿어도 되는가?"에 대한 답을 제공

2. **주의 사항**
   - 신뢰도 점수가 오히려 PM을 혼란스럽게 할 수 있음 (정보 과부하)
   - 신뢰도 표시가 너무 복잡하지 않도록 PM 친화적 설계 필요
   - "Low Confidence"라는 표현이 도구의 신뢰도 자체를 떨어뜨리지 않도록 주의
   - 보완 가이드가 PM이 수행 가능한 수준이어야 함 (코드 수정은 PM이 할 수 없음)

3. **보완 방법의 PM 접근성**
   - `/impact policies add` 명령어: PM이 채팅으로 정책을 알려주는 것과 유사 (자연스러움)
   - `policies.yaml` 편집: PM에게는 다소 기술적. 하지만 YAML은 비교적 읽기 쉬운 형식
   - 코드 주석 추가: PM이 직접 할 수 없으므로, 개발팀에게 요청하는 가이드 제공
