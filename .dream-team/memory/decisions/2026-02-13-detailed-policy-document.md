## 1. 문제 정의

### 현재 상태

현재 정책서 역생성 설계(DECISION-003 섹션 7)는 다음 수준의 정보만 제공한다:

```
| 정책명           | 설명                        | 관련 코드          | 신뢰도 |
| 멤버스 무료배송   | 멤버스 가입자에게 무료배송 적용 | shipping.ts:42    | 80%   |
```

이 수준은 기획자가 실질적으로 활용하기에 **지나치게 추상적**이다.

### 기획자가 겪는 문제

1. **"어떤 조건에서 어떤 결과가 나오는지" 모름**: "무료배송"이라는 단어만으로는 정확한 조건분기를 파악할 수 없음
2. **"변경하면 어디가 깨지는지" 모름**: 관련 코드 경로만으로는 사이드이펙트를 예측할 수 없음
3. **"하드코딩인지 설정값인지" 모름**: 금액 기준값이 코드에 직접 박혀 있는지, 외부에서 변경 가능한지 알 수 없음
4. **"이 정책이 다른 정책과 어떻게 연결되는지" 모름**: 배송비 정책이 프로모션 정책과 어떤 관계인지 파악 불가

### 기획자가 필요로 하는 수준

```
정책: 멤버스 무료배송

[조건 -> 로직 -> 결과]
  IF 사용자.멤버스등급 == "멤버스" AND 주문금액 >= 0원
    THEN 배송비 = 0원 (무료배송)
  ELSE IF 사용자.멤버스등급 != "멤버스" AND 주문금액 >= 40,000원
    THEN 배송비 = 0원 (금액 기반 무료배송)
  ELSE
    THEN 배송비 = 3,000원 (기본 배송비)

[관련 변수]
  - isMember: boolean (멤버스 가입 여부)
  - orderAmount: number (주문 금액)
  - shippingFee: number (계산된 배송비)
  - FREE_SHIPPING_THRESHOLD: 40,000 (무료배송 기준 금액 - 하드코딩)

[기획자 확인 사항]
  (!) 무료배송 기준 금액이 40,000원으로 하드코딩되어 있음
     -> 이 금액을 변경하려면 코드 수정 필요
  (!) 멤버스 등급 체크가 boolean(가입/미가입)만 구분
     -> 멤버스 등급별(실버/골드/VIP) 차등 배송비는 현재 미지원
```

## 2. PM 관점: 기획자가 정책서에서 확인해야 할 항목 목록

### 2.1 비즈니스 로직 상세 (조건 -> 로직 -> 결과)

기획자가 정책을 이해하려면 **코드의 조건분기를 자연어로 변환한 로직 흐름**이 필수적이다.

**필수 포함 항목**:
- IF/ELSE IF/ELSE 조건 분기를 자연어 또는 의사코드(pseudocode)로 표현
- 각 분기별 최종 결과값 (예: 배송비 = 0원, 배송비 = 3,000원)
- 조건에 사용되는 비교 연산 (>=, ==, != 등)과 비교 대상값
- 예외 처리 로직 (try/catch, fallback 값)
- 기본값(default) 처리

**표현 형식**: 기획자가 코드를 읽지 않아도 이해할 수 있는 의사코드 형태

```
[조건 -> 로직 -> 결과]
  IF {조건1}
    THEN {결과1}
  ELSE IF {조건2}
    THEN {결과2}
  ELSE
    THEN {기본 결과}

  [예외 처리]
    에러 발생 시 -> {폴백 결과}
```

### 2.2 입출력 변수 및 상수값

기획자가 "이 정책이 어떤 데이터를 사용하고 어떤 값을 만들어내는지" 파악할 수 있어야 한다.

**필수 포함 항목**:
- **입력 변수**: 함수에 전달되는 파라미터 목록 (이름, 타입, 역할 설명)
- **출력 변수**: 함수가 반환하는 값 (이름, 타입, 역할 설명)
- **사용 상수값**: 함수 내에서 사용하는 고정값 (매직넘버, 상수, enum 등)
  - 하드코딩 여부 표시 (하드코딩 / 설정 파일 참조 / 환경 변수 / DB 조회)
  - 현재 설정된 값
- **내부 변수**: 로직 흐름에서 중간에 생성/변경되는 핵심 변수

**표현 형식**:

```
[입력]
  - orderAmount: number  -- 주문 금액 (원)
  - user: User           -- 사용자 정보 객체

[출력]
  - shippingFee: number  -- 계산된 배송비 (원)

[사용 상수]
  - FREE_SHIPPING_THRESHOLD: 40,000  -- 무료배송 기준 금액 [하드코딩]
  - DEFAULT_SHIPPING_FEE: 3,000      -- 기본 배송비 [하드코딩]
  - ISLAND_ADDITIONAL_FEE: 3,000     -- 도서산간 추가비 [하드코딩]
```

### 2.3 변경 영향 범위

기획자가 "이 정책을 변경하면 시스템 어디에 영향이 가는지" 미리 파악할 수 있어야 한다.

**필수 포함 항목**:
- **이 정책을 호출하는 함수/화면 목록**: 이 함수를 직접 호출하는 상위 함수들
- **이 정책이 호출하는 함수 목록**: 이 함수가 내부적으로 사용하는 하위 함수들
- **관련 API 엔드포인트**: 이 정책이 사용되는 API 경로
- **영향 받는 화면 목록**: 이 정책의 결과가 표시되는 프론트엔드 화면
- **영향도 요약**: 변경 시 예상되는 파급 범위를 한 줄로 요약

**표현 형식**:

```
[이 정책을 사용하는 곳]
  -> calculateOrderTotal()     : 주문 총액 계산 시 배송비 합산
  -> OrderSummaryPage          : 주문 요약 화면에서 배송비 표시
  -> CartPage                  : 장바구니에서 예상 배송비 표시

[이 정책이 사용하는 것]
  <- getUserMembershipLevel()  : 멤버스 등급 조회
  <- getRegionInfo()           : 배송 지역 정보 조회

[관련 API]
  POST /api/v1/orders/calculate  : 주문 금액 계산
  GET  /api/v1/shipping/estimate : 배송비 예상 조회

[변경 시 영향 요약]
  이 정책 변경 시 주문 결제 금액, 장바구니 표시 금액, 배송비 안내 문구에 영향
```

### 2.4 현재 제약사항 및 한계

기획자가 "현재 구현이 지원하지 않는 것, 변경하기 어려운 것"을 인식해야 한다.

**필수 포함 항목**:
- **하드코딩된 값 목록**: 코드 수정 없이는 변경 불가능한 값들
- **미지원 케이스**: 현재 코드가 처리하지 않는 경우 (예: 등급별 차등, 시간대별 차등)
- **예외 처리 부재**: catch 블록이 없거나 불충분한 경우
- **기술적 부채**: deprecated 패턴, TODO 주석, 임시 처리 등
- **확장 제한**: 현재 구조로는 특정 기능 추가가 어려운 경우

**표현 형식**:

```
[현재 제약사항]
  (!) 무료배송 기준 금액(40,000원)이 하드코딩
     -> 변경 시 코드 수정 + 배포 필요
     -> 권장: 설정 파일 또는 관리자 페이지에서 변경 가능하도록 개선

  (!) 멤버스 등급이 boolean(가입/미가입)으로만 구분됨
     -> 등급별(실버/골드/VIP) 차등 배송비 미지원
     -> 추가 개발 필요 시 User 인터페이스 + 이 함수 모두 수정

  (?) TODO 주석 발견: "// TODO: 새벽배송 추가비용 분리"
     -> 새벽배송 로직이 이 함수에 혼재되어 있을 수 있음
```

### 2.5 데이터 출처

기획자가 "이 정책에서 사용하는 데이터가 어디서 오는지" 알아야 한다.

**필수 포함 항목**:
- **DB 조회 데이터**: 테이블명, 컬럼명, 조회 조건
- **API 호출 데이터**: 외부 API에서 받아오는 데이터
- **사용자 입력 데이터**: 프론트엔드에서 사용자가 직접 입력하는 값
- **세션/상태 데이터**: 로그인 정보, 장바구니 상태 등
- **설정 데이터**: 환경 변수, 설정 파일에서 읽어오는 값

**표현 형식**:

```
[데이터 출처]
  - 사용자.멤버스여부  : DB users 테이블 is_member 컬럼
  - 주문금액          : 장바구니 상태에서 실시간 계산
  - 배송지역          : 사용자 입력 (배송지 주소에서 추출)
  - 무료배송기준금액   : 코드 내 상수 (하드코딩)
```

### 2.6 정책 간 의존 관계

기획자가 "이 정책이 다른 정책과 어떻게 얽혀 있는지" 전체 그림을 파악해야 한다.

**필수 포함 항목**:
- **선행 정책**: 이 정책이 실행되기 전에 먼저 적용되어야 하는 정책
- **후행 정책**: 이 정책의 결과를 참조하는 다른 정책
- **충돌 가능 정책**: 이 정책과 동시에 적용될 때 충돌할 수 있는 정책
- **의존 관계 다이어그램**: 정책 간 관계를 시각적으로 표현

**표현 형식**:

```
[정책 의존 관계]
  [선행] 멤버스 등급 조회 정책
     -> 이 정책 실행 전에 사용자의 멤버스 등급이 결정되어야 함

  [후행] 최종 결제 금액 계산 정책
     -> 이 정책의 배송비 결과를 합산하여 최종 금액 산출

  [충돌 가능] 프로모션 무료배송 쿠폰
     -> 멤버스 무료배송과 쿠폰 무료배송이 동시에 적용될 때 중복 할인 여부 확인 필요

[의존 다이어그램]
  멤버스 등급 조회 -> [멤버스 무료배송] -> 최종 결제 금액 계산
                                       -> 주문 요약 표시
  프로모션 쿠폰 적용 ----(충돌?)----> [멤버스 무료배송]
```

### 2.7 기획자 확인 체크리스트

기획자가 정책서를 검토한 후 반드시 확인해야 할 항목들의 자동 생성 체크리스트.

```
[기획자 확인 체크리스트]
  [ ] 조건분기 로직이 현재 기획 의도와 일치하는가?
  [ ] 하드코딩된 값(40,000원, 3,000원)이 현재 비즈니스 요구와 맞는가?
  [ ] 미지원 케이스(등급별 차등)가 향후 기획에 필요한가?
  [ ] 정책 간 충돌 가능성(프로모션 쿠폰)에 대한 규칙이 정의되어 있는가?
  [ ] 새벽배송 추가비용 로직이 별도 관리되고 있는가?
  [ ] 변경 시 영향 범위(주문 결제, 장바구니, 배송비 안내)를 모두 고려했는가?
```

## 3. TPO 관점: 기술 설계

### 3.1 보강 주석(InferredPolicy) 스키마 확장

현재 `InferredPolicy` 인터페이스는 `name`, `description`, `confidence`, `category`, `inferred_from` 5개 필드만 가지고 있다. 상세 정책서를 생성하기 위해 대폭 확장한다.

#### 확장된 InferredPolicy 인터페이스

```typescript
interface InferredPolicy {
  // === 기존 필드 (유지) ===
  name: string;                    // 정책명
  description: string;             // 정책 설명 (1~2문장 요약)
  confidence: number;              // 0.0~1.0
  category: string;                // 카테고리 (배송, 결제, 프로모션 등)
  inferred_from: string;           // 추론 근거 (코드 패턴 설명)

  // === 신규 필드 (상세 정책서용) ===

  // 조건-로직-결과 흐름
  conditions: PolicyCondition[];   // IF/ELSE 조건 분기 목록
  defaultResult: string;           // ELSE (기본) 결과 설명
  exceptionHandling: string | null; // 예외 처리 로직 설명 (없으면 null)

  // 입출력 변수
  inputVariables: PolicyVariable[];   // 입력 파라미터
  outputVariables: PolicyVariable[];  // 출력(반환) 값
  constants: PolicyConstant[];        // 사용 상수값
  internalVariables: PolicyVariable[]; // 내부 핵심 변수

  // 제약사항 및 한계
  constraints: PolicyConstraint[];    // 현재 제약사항 목록

  // 데이터 출처
  dataSources: PolicyDataSource[];    // 데이터 출처 목록

  // 정책 간 관계
  dependencies: PolicyDependency[];   // 정책 간 의존 관계

  // 변경 영향 범위
  impactScope: PolicyImpactScope;     // 변경 시 영향 범위

  // 기획자 확인 사항 (LLM이 자동 도출)
  reviewItems: PolicyReviewItem[];    // 기획자 확인 체크리스트
}

// 조건 분기 항목
interface PolicyCondition {
  order: number;                   // 분기 순서 (1, 2, 3...)
  type: 'if' | 'else_if' | 'else'; // 분기 유형
  condition: string;               // 조건 설명 (자연어)
  conditionCode: string;           // 원본 조건 코드 (예: "isMember && orderAmount >= 0")
  result: string;                  // 결과 설명 (자연어)
  resultValue: string;             // 결과값 (예: "shippingFee = 0")
}

// 변수 정보
interface PolicyVariable {
  name: string;                    // 변수명
  type: string;                    // 타입 (boolean, number, string 등)
  description: string;             // 역할 설명 (자연어)
}

// 상수값 정보
interface PolicyConstant {
  name: string;                    // 상수명 또는 변수명
  value: string;                   // 현재 설정된 값
  type: string;                    // 타입
  description: string;             // 역할 설명
  source: 'hardcoded' | 'config_file' | 'env_variable' | 'db_query' | 'api_call';
  // 값의 출처 (하드코딩 / 설정 파일 / 환경 변수 / DB 조회 / API 호출)
  codeLocation: string;            // 값이 정의된 코드 위치 (예: "shipping.ts:15")
}

// 제약사항
interface PolicyConstraint {
  severity: 'warning' | 'info' | 'critical'; // 심각도
  type: 'hardcoded_value' | 'unsupported_case' | 'no_exception_handling'
        | 'tech_debt' | 'extension_limit';   // 제약 유형
  description: string;             // 제약사항 설명 (자연어)
  recommendation: string;          // 개선 권장사항
  relatedCode: string;             // 관련 코드 위치
}

// 데이터 출처
interface PolicyDataSource {
  variableName: string;            // 사용하는 변수명
  sourceType: 'db' | 'api' | 'user_input' | 'session' | 'config' | 'constant';
  sourceDetail: string;            // 상세 출처 (테이블명, API 경로 등)
  description: string;             // 설명
}

// 정책 간 의존 관계
interface PolicyDependency {
  relationType: 'prerequisite' | 'dependent' | 'conflict' | 'related';
  // prerequisite: 이 정책 실행 전에 필요한 선행 정책
  // dependent: 이 정책 결과를 참조하는 후행 정책
  // conflict: 충돌 가능한 정책
  // related: 관련 있지만 직접 의존은 아닌 정책
  policyName: string;              // 관련 정책명
  description: string;             // 관계 설명
  functionName: string | null;     // 관련 함수명 (있으면)
  filePath: string | null;         // 관련 파일 경로 (있으면)
}

// 변경 영향 범위
interface PolicyImpactScope {
  callers: PolicyReference[];      // 이 정책을 호출하는 상위 함수/화면
  callees: PolicyReference[];      // 이 정책이 호출하는 하위 함수
  relatedApis: string[];           // 관련 API 엔드포인트
  affectedScreens: string[];       // 영향 받는 화면 목록
  impactSummary: string;           // 변경 시 영향 요약 (1문장)
}

interface PolicyReference {
  name: string;                    // 함수/화면명
  filePath: string;                // 파일 경로
  description: string;             // 호출/참조 이유
}

// 기획자 확인 항목
interface PolicyReviewItem {
  priority: 'high' | 'medium' | 'low'; // 우선순위
  category: 'logic_check' | 'value_check' | 'scope_check' | 'conflict_check' | 'missing_case';
  question: string;                // 확인 질문 (자연어)
  context: string;                 // 왜 확인이 필요한지 설명
  relatedConstraint: string | null; // 관련 제약사항 (있으면)
}
```

#### FunctionAnnotation 확장

기존 `FunctionAnnotation`의 `policies` 필드 타입을 확장된 `InferredPolicy`로 교체한다. 기존 데이터와의 호환성을 위해, 신규 필드들은 모두 optional로 처리하여 점진적 마이그레이션을 지원한다.

```typescript
interface FunctionAnnotation {
  // ... 기존 필드 유지 ...
  policies: InferredPolicy[];       // 확장된 InferredPolicy 사용
  // 신규 필드
  policyFlowSummary: string | null; // 전체 정책 흐름 요약 (함수 내 모든 정책의 관계 요약)
}
```

### 3.2 LLM 분석 프롬프트 전략 (깊이 있는 분석)

현재 보강 주석 생성 프롬프트는 "함수 요약 + 정책명/설명 추출" 수준이다. 상세 정책서를 위해 프롬프트를 **3단계 분석 전략**으로 강화한다.

#### 1단계: 함수 레벨 상세 분석 (기존 프롬프트 확장)

```
<system>
당신은 코드 분석 전문가이자 비즈니스 로직 문서화 전문가입니다.
주어진 함수를 분석하여 기획자(비개발자)가 이해할 수 있는 수준의
상세한 비즈니스 로직 명세를 작성합니다.
</system>

<instruction>
다음 함수를 분석하여 아래 항목을 모두 추출하세요:

1. **조건 분기 분석**: 모든 IF/ELSE IF/ELSE/switch 분기를 자연어로 기술
   - 각 분기의 조건(자연어) + 조건 코드(원본)
   - 각 분기의 결과(자연어) + 결과값(원본)
   - 기본 결과(default/else)
   - 예외 처리(try/catch)의 fallback 결과

2. **변수 분석**: 함수의 입출력과 핵심 내부 변수를 추출
   - 입력 파라미터: 이름, 타입, 역할
   - 반환값: 이름, 타입, 역할
   - 함수 내에서 사용하는 상수값: 이름, 값, 타입, 출처(하드코딩/설정파일/DB/API)
   - 비교에 사용되는 매직넘버: 값, 의미, 출처

3. **제약사항 분석**: 현재 구현의 한계와 주의사항
   - 하드코딩된 비즈니스 값 (변경 시 코드 수정 필요)
   - 지원하지 않는 케이스 (코드에 분기가 없는 경우)
   - TODO/FIXME/HACK 주석
   - 에러 처리가 없거나 불충분한 경우
   - deprecated 패턴 사용

4. **데이터 출처 분석**: 각 입력 데이터의 궁극적 출처
   - DB 테이블/컬럼
   - API 호출
   - 사용자 입력
   - 세션/상태값
   - 설정값

5. **기획자 확인 사항 도출**: 기획자가 반드시 검토해야 할 질문
   - 하드코딩 값이 현재 비즈니스 요구와 맞는지
   - 미지원 케이스가 향후 필요한지
   - 조건 로직이 기획 의도와 일치하는지
   - 정책 변경 시 영향 범위가 수용 가능한지

응답은 반드시 아래 JSON 스키마에 맞추어 구조화된 형태로 반환하세요.
</instruction>

<source>
{소스코드}
</source>

<context>
{프로젝트 컨텍스트 - 기존 보강 주석, 관련 함수 시그니처 등}
</context>
```

#### 2단계: 정책 간 관계 분석 (신규)

함수 호출 관계 그래프(인덱스의 dependencies.json)를 활용하여 정책 간 의존성을 추출한다.

```
<system>
당신은 시스템 아키텍처 분석 전문가입니다.
함수 호출 관계와 각 함수의 보강 주석을 분석하여,
비즈니스 정책 간의 의존 관계를 도출합니다.
</system>

<instruction>
아래 정보를 바탕으로 정책 간 의존 관계를 분석하세요:

1. 함수 호출 그래프에서 정책 관련 함수 간의 호출 관계 식별
2. 각 정책의 입출력을 비교하여 데이터 의존성 파악
3. 동일한 데이터를 참조하는 정책 간 충돌 가능성 분석
4. 선행/후행 관계, 충돌 관계, 연관 관계로 분류

결과를 PolicyDependency[] 형태로 반환하세요.
</instruction>

<call_graph>
{함수 호출 관계 (인덱스의 dependencies.json에서 추출)}
</call_graph>

<policies>
{이 시스템의 모든 InferredPolicy 목록 (이름, 함수, 파일)}
</policies>
```

#### 3단계: 변경 영향 범위 분석 (신규)

인덱스의 컴포넌트 의존 관계 그래프와 라우트 정보를 활용하여 각 정책의 변경 영향 범위를 도출한다.

```
<system>
당신은 소프트웨어 영향 분석 전문가입니다.
코드 인덱스의 의존 관계 그래프를 분석하여,
특정 함수/정책 변경 시 영향 받는 범위를 도출합니다.
</system>

<instruction>
다음 정책 함수의 변경 영향 범위를 분석하세요:

1. 이 함수를 직접 호출하는 상위 함수(callers) 식별
2. 이 함수가 호출하는 하위 함수(callees) 식별
3. callers를 추적하여 관련 API 엔드포인트 식별
4. callers를 추적하여 관련 화면(라우트) 식별
5. 변경 시 예상 영향을 1문장으로 요약

결과를 PolicyImpactScope 형태로 반환하세요.
</instruction>

<target_function>
{분석 대상 함수 정보}
</target_function>

<dependency_graph>
{인덱스의 컴포넌트 의존 관계}
</dependency_graph>

<routes>
{인덱스의 화면/라우트 정보}
</routes>

<apis>
{인덱스의 API 엔드포인트 정보}
</apis>
```

#### 프롬프트 전략 통합 흐름

```
[기존] Step 3.8: 보강 주석 생성 (함수 요약 + 정책명/설명)
        |
        v
[확장] Step 3.8-A: 함수 레벨 상세 분석 (1단계 프롬프트)
        |  - 조건분기, 변수, 상수, 제약사항, 데이터 출처 추출
        |  - 기획자 확인 사항 자동 도출
        |
[신규] Step 3.8-B: 정책 간 관계 분석 (2단계 프롬프트)
        |  - 시스템 내 모든 정책의 의존/충돌 관계 도출
        |  - 함수 호출 그래프 + 보강 주석 기반
        |
[신규] Step 3.8-C: 변경 영향 범위 분석 (3단계 프롬프트)
        |  - 인덱스 의존 그래프 기반 callers/callees 추적
        |  - 관련 API, 화면 목록 도출
        v
[기존] Step 3.9: YAML 파일 저장 + meta.json 업데이트
```

**비용 영향**: 1단계는 기존 프롬프트 확장이므로 추가 비용 미미(토큰 수 약 1.5배 증가). 2단계와 3단계는 시스템 단위 1회씩 호출이므로 파일 수에 비례하지 않음. 전체 비용 약 20~30% 증가 예상.

### 3.3 정책 간 관계 추출 로직

정책 간 관계는 두 가지 경로로 추출한다:

#### 경로 1: 정적 분석 기반 (코드 인덱스 활용)

```
[입력] 인덱스의 dependencies.json (함수 호출 그래프)
       + 보강 주석의 policies 필드 (정책이 속한 함수 정보)

[처리]
1. 정책 P1이 속한 함수 F1이 호출하는 함수 목록 추출
2. 호출된 함수들 중 다른 정책 P2가 속한 함수 F2가 있는지 확인
3. F1 -> F2 호출 관계이면:
   - P1은 P2의 결과를 참조 (P2는 P1의 prerequisite)
   - P2는 P1에 의존 (P1은 P2의 dependent)
4. F1과 F2가 동일한 변수를 수정하면:
   - P1과 P2는 conflict 관계

[출력] PolicyDependency[]
```

#### 경로 2: LLM 추론 기반 (2단계 프롬프트)

정적 분석으로 파악하기 어려운 의미적 관계(예: 같은 비즈니스 도메인 내 암묵적 의존)를 LLM이 추론한다.

## 4. 상세 정책서 템플릿 (전체 예시)

하나의 정책에 대한 완전한 상세 정책서 예시이다. 기획자가 바로 이해할 수 있는 수준을 목표로 한다.

```markdown
### 정책 1: 멤버스 무료배송 [배송] [신뢰도: 85%]

**요약**: 컬리 멤버스 가입자에게 주문 금액에 관계없이 무료배송을 적용하고,
비멤버스 사용자에게는 주문 금액 40,000원 이상 시 무료배송을 적용한다.
그 외에는 기본 배송비 3,000원을 부과한다.

**관련 코드**: src/services/shipping/calculateFee.ts:42 - calculateShippingFee()

#### 조건 -> 로직 -> 결과

  IF 사용자가 멤버스 가입자이고 주문금액이 0원 이상
    THEN 배송비 = 0원 (멤버스 무료배송)
  ELSE IF 사용자가 멤버스 비가입자이고 주문금액이 40,000원 이상
    THEN 배송비 = 0원 (금액 기반 무료배송)
  ELSE IF 배송 지역이 제주 또는 도서산간
    THEN 배송비 = 3,000원 + 3,000원 = 6,000원 (기본배송비 + 도서산간 추가비)
  ELSE
    THEN 배송비 = 3,000원 (기본 배송비)

  [예외 처리]
    사용자 정보 조회 실패 시 -> 배송비 = 3,000원 (기본값 적용)
```

## 6. 기획자 확인 사항 자동 생성 로직

LLM이 "기획자 확인 사항(reviewItems)"을 자동으로 도출하는 규칙을 정의한다.

### 자동 생성 규칙

| # | 트리거 조건 | 생성되는 확인 사항 | 우선순위 |
|---|------------|-------------------|---------|
| 1 | PolicyConstant의 source가 'hardcoded' | "{상수명}({값})이 코드에 하드코딩되어 있음. 현재 비즈니스 요구와 일치하는가?" | high |
| 2 | PolicyConstraint의 type이 'unsupported_case' | "미지원 케이스: {설명}. 향후 기획에 이 기능이 필요한가?" | high |
| 3 | PolicyDependency의 relationType이 'conflict' | "충돌 가능 정책: {정책명}. 두 정책이 동시에 적용될 때의 규칙이 정의되어 있는가?" | medium |
| 4 | PolicyConstraint의 type이 'no_exception_handling' | "예외 처리 부재: {설명}. 에러 발생 시 사용자 경험에 문제가 없는가?" | medium |
| 5 | PolicyConstraint의 type이 'tech_debt' | "기술적 부채: {설명}. 향후 기능 확장에 영향을 줄 수 있음" | low |
| 6 | PolicyImpactScope의 affectedScreens가 3개 이상 | "변경 영향 범위가 넓음(화면 {N}개). 모든 영향 화면을 검토했는가?" | medium |
| 7 | PolicyCondition이 5개 이상 | "조건 분기가 복잡함({N}개 분기). 모든 케이스가 기획 의도와 일치하는가?" | medium |
| 8 | confidence가 0.7 미만인 InferredPolicy | "이 정책의 분석 신뢰도가 {N}%로 낮음. 개발팀과 검증이 필요함" | high |
| 9 | PolicyDataSource에 'db'가 포함 | "DB 데이터({테이블명})를 사용함. 데이터 변경 시 이 정책에도 영향" | low |
| 10 | PolicyDependency의 prerequisite가 2개 이상 | "선행 정책이 {N}개. 선행 정책 변경 시 이 정책도 영향 받음" | medium |

### 확인 사항 포맷

각 확인 사항은 다음 포맷으로 생성된다:

```
- [ ] [{우선순위}] {확인 질문}
  - 배경: {왜 확인이 필요한지 1~2문장}
  - 관련 코드: {파일:라인}
  - 트리거: {어떤 분석 결과에서 이 확인 사항이 도출되었는지}
```

### LLM 추가 도출

위 규칙 외에도 LLM이 코드 분석 과정에서 발견한 비정상적 패턴, 주석에 기재된 주의사항, 복잡한 비즈니스 로직 등에서 추가로 확인 사항을 도출할 수 있다. 이 경우 `category: 'logic_check'`으로 표시하고, LLM이 판단한 우선순위를 적용한다.

## 7. MVP 범위 영향

### 기존 결정 (DECISION-003)

정책서 역생성은 Phase 2로 결정되었다.

### 변경 제안

**변경 없음**: 정책서 역생성은 여전히 Phase 2로 유지한다.

단, 이번 상세 설계의 핵심 요소인 **InferredPolicy 확장 스키마**와 **1단계 상세 분석 프롬프트**는 **MVP의 보강 주석 생성 단계에 포함**하는 것을 권장한다.

**근거**:
1. InferredPolicy에 conditions, variables, constraints 등을 **보강 주석 생성 시점에 이미 추출**하면, Phase 2에서 정책서 생성 시 추가 LLM 호출 없이 수집만 하면 된다.
2. 기획자가 보강 주석 조회 시(`/impact annotations view`) 이미 상세 정보를 볼 수 있어 MVP 가치가 높아진다.
3. 1단계 프롬프트는 기존 프롬프트의 확장이므로 추가 구현 비용이 크지 않다.

### MVP에 추가 포함 항목

| # | 항목 | 근거 |
|---|------|------|
| 1 | InferredPolicy 확장 스키마 | 보강 주석 데이터 모델의 일부로 포함 |
| 2 | 1단계 상세 분석 프롬프트 | 기존 프롬프트 확장으로 추가 비용 미미 |
| 3 | CLI에서 상세 정책 조회 | `/impact annotations view`에서 조건/변수/제약사항 표시 |

### Phase 2에 유지하는 항목

| # | 항목 | 근거 |
|---|------|------|
| 1 | 2단계 정책 간 관계 분석 | 보강 주석 축적 후 의미 있음 |
| 2 | 3단계 변경 영향 범위 분석 | 인덱스 + 보강 주석 모두 필요 |
| 3 | 정책서 MD 파일 자동 생성 | 상세 데이터 축적 후 생성 |
| 4 | 전체 통합 정책서 | 시스템별 정책서 생성 후 통합 |
| 5 | 웹 UI 정책서 뷰 | Phase 2 웹 UI 기능 |

## 8. 리스크

| 리스크 | 심각도 | 대응 방안 |
|--------|--------|----------|
| **LLM 분석 정확도** | 높음 | 조건분기 추출 정확도는 confidence 점수로 표현. 0.5 미만은 "확인 필요" 마커 부착 |
| **프롬프트 토큰 증가** | 중간 | 1단계 프롬프트는 기존 대비 1.5배 수준으로 비용 증가 제한적. 2~3단계는 시스템 단위 1회 |
| **복잡한 코드의 조건분기 추출 한계** | 중간 | 중첩 조건문, 동적 조건(변수에 따라 달라지는 조건)은 정확도 낮음. 이 경우 confidence를 낮게 설정하고 기획자에게 검증 요청 |
| **정책 간 관계 오추론** | 중간 | 정적 분석(호출 그래프) 결과를 우선하고, LLM 추론은 보조 수단으로 활용. 관계별 confidence 표시 |
| **정책서 분량 과다** | 낮음 | 시스템 내 정책이 50건 이상이면 카테고리별 요약만 기본 표시, 상세는 접기/펼치기 또는 개별 파일로 분리 |
| **기존 보강 주석 스키마와의 호환성** | 낮음 | 신규 필드를 모두 optional로 처리하여 기존 데이터 무결성 유지. 마이그레이션 스크립트 불필요 |

---

> **결론**: 상세 정책서 역생성은 기획자가 실제로 활용할 수 있는 수준의 정보(조건/로직/결과, 변수/상수, 제약사항, 의존관계, 기획자 확인 체크리스트)를 제공하는 것이 핵심이다. InferredPolicy 스키마 확장과 1단계 상세 분석 프롬프트는 MVP 보강 주석 생성에 포함하여 조기에 데이터를 축적하고, 정책서 문서 생성과 정책 간 관계 분석은 Phase 2에서 구현한다.
